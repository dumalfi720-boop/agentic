🎯 Layer 1 — Smart Planning

How to turn a vague objective into an executable plan with dependencies, priorities and cost estimates.

What is it:

Use an LLM to decompose a high-level objective into atomic, executable subtasks, validating completeness and avoiding overlap between tasks.

Why learn:

Without structured decomposition, the orchestrator attempts to execute impossible objectives all at once, resulting in failures and expensive rework.

Key concepts:

Atomic subtask · Completeness criterion · Avoid overlap · LLM as decomposer · Plan validation

What is it:

Representation of tasks as a Directed Acyclic Graph, identifying which tasks block others and which can be executed in parallel for maximum efficiency.

Why learn:

DAG is the framework that enables safe parallelism — running tasks in the wrong order causes cascading failures and incorrect output.

Key concepts:

DAG · Blocking node · Parallel task · Topological sort · Transitive dependency · Invalid cycle

What is it:

How to extract executable dimensions from the objective in natural language: What to do? For whom? With what success criteria? What is the expected delivery format?

Why learn:

Ambiguous goals generate ambiguous plans. Structured parsing forces clarity before any execution — saves tokens and avoids rework.

Key concepts:

Structured extraction · Success criteria · Delivery format · Implicit stakeholder · Intentional vs. intentional ambiguity accidental

What is it:

Hierarchical structure where high-level tasks are decomposed into subtasks, and subtasks into atomic actions — forming a tree navigable by the orchestrator.

Why learn:

The tree allows for granular progress tracking — knowing exactly where the system is in the run, even across hours-long runs.

Key concepts:

Root node · Atomic leaf · Abstraction level · Progress per level · Depth estimation · Recursion limit

What is it:

When multiple tasks are unlocked, which one to do first? Criteria: criticality for the critical path, estimated duration, number of downstream dependents, estimated cost.

Why learn:

The order of execution affects the total time and total cost. Performing critical tasks first reduces cascading wait time.

Key concepts:

Critical path · Criticality · FIFO vs. priority queue · Cost of delay · Duration estimation · Priority score

What is it:

Complete walkthrough: "Prepare due diligence of company

Why learn:

Seeing the completed planner in action anchors all of the previous concepts into a real, representative use case of what you will build.

Key concepts:

Due diligence · 5 task parallelism · 3 waves of execution · Estimated cost $2.40 · Estimated time 12 min · Complete Python code

🔀 Layer 2 — Multi-Agent Routing

Given a pool of specialized agents, dynamically select the best agent for each task — by capacity, cost, and availability.

What is it:

Given a pool of specialized agents (financial, legal, technical, writing), which one should perform each task? Wrong routing generates poor quality and high costs.

Why learn:

Routing is the orchestrator's most impactful decision — it affects quality, cost, and latency of the entire execution. Intelligent routing can reduce cost by 60%.

Key concepts:

Agent pool · Specialization · Selection criteria · Fallback · Misrouting cost · Benchmark by task type

What is it:

Fixed table mapping task type to responsible agent. Simple, predictable, easy to debug — but difficult to maintain as the number of agents and task types grows.

Why learn:

Understanding static routing first is critical — it's the baseline against which you compare dynamic routing and measure the actual gain.

Key concepts:

Routing table · Manual maintenance · Combinatorial explosion · When to use · When to migrate to dynamic · Code example

What is it:

A router agent uses LLM to read the task description and select the most suitable agent from the pool, with a fallback to a generalist agent when no specialist responds.

Why learn:

The LLM router scales naturally: adding a new agent to the pool only requires updating the capabilities description, without changing routing tables.

Key concepts:

LLM classifier · Agent capabilities description · Confidence score · Fallback chain · Inexpensive model for classification · Routing latency

What is it:

When there are multiple instances of the same type of agent, distribute the load by availability, current cost, or round-robin to avoid bottlenecks and timeouts.

Why learn:

Without load balancing, one instance is overloaded while others are idle — increasing latency and risk of timeout at the worst moment.

Key concepts:

Round-robin · Least connections · Health-aware routing · Agent pool · Concurrency limit · Queue depth per agent

What is it:

Sort tasks by complexity and route to the agent with the cheapest model capable of executing with acceptable quality — simple tasks do not need GPT-4o.

Why learn:

Using GPT-4o for everything can cost 30x more than necessary. Cost routing is the most direct lever for cost reduction in production.

Key concepts:

Complexity classifier · Model tier · Complexity cost · Quality floor · Cost/quality trade-off · Decision table

What is it:

Complete implementation of the AgentRouter class with agent pool, route(task) → agent method, logging of routing decisions and success metrics.

Why learn:

Seeing the concrete implementation solidifies the concepts and provides a reusable template for the final trail project.

Key concepts:

AgentPool · route() method · Decision log · Routing accuracy · A/B strategy testing · Router extensibility

======== MODULE 5.3 ========
5.3 ~50 min

🗄️ Layer 3 — State Control

Redis for running state, Postgres for persistence, session management and failure recovery.

What is it:

When multiple agents run in parallel on different processes or machines, the shared state cannot be stored in local memory — it needs distributed storage.

Why learn:

State in local memory is the #1 cause of failure in real multi-agent systems — works in dev, breaks in production when there is more than one worker.

Key concepts:

Shared state · Race condition · Single source of truth · Eventual consistency · Locking · Atomicity of operations

What is it:

Redis structures for orchestrator: hashes for the state of each task, sorted sets for priority queue, streams for execution events, pub/sub for notifications between agents.

Why learn:

Redis is the industry-standard choice for fast execution state — sub-millisecond latency and perfect data structures for orchestration.

Key concepts:

HSET/HGET · ZADD/ZRANGE · XADD/XREAD · State TTL · Keyspace for session · Atomicity with MULTI/EXEC

What is it:

Postgres schema for persistent data: historical execution tables, immutable audit log, orchestrator settings — what goes into Redis vs. Postgres and why.

Why learn:

Redis is volatile — data expires or is lost on restart. Postgres ensures that the execution history, essential for auditing and debugging, is permanent.

Key concepts:

Execution scheme · Audit log append-only · Redis vs. Postgres decision matrix · Sync between Redis and Postgres · State backup

What is it:

Each orchestrator run has a unique session_id. Every state, log and result is associated with it. Session TTL ensures automatic cleanup of abandoned sessions.

Why learn:

Without session management, concurrent executions interfere with each other and data from old sessions contaminates new executions — a bug that is difficult to reproduce in production.

Key concepts:

UUID as session_id · Redis namespace per session · Automatic TTL · Orphan session · Cleanup job · Intersession isolation

What is it:

The orchestrator injects only the relevant context for each agent — not the entire history of 1000 messages. Summarization and context selection techniques based on relevance.

Why learn:

Full context window = exponential cost and high latency. Actively managing context can reduce cost per agent by 70% without loss of quality.

Key concepts:

Context budget · History summarization · Relevance selection · Embedding similarity · Minimum viable context · Sliding window

What is it:

If the orchestrator process dies in the middle of a run, how can you resume exactly where you left off using the saved state in Redis/Postgres — without restarting from scratch.

Why learn:

Long runs (hours) in production always face failures. Starting over from scratch can cost $50+ in tokens. Checkpoint recovery is mandatory in production.

Key concepts:

Checkpoint · Task state (PENDING/RUNNING/DONE/FAILED) · Resume logic · Task idempotency · Restart detection · Recovery cost

======== MODULE 5.4 ========
5.4 ~50 min

👁️ Layer 4 — Supervision and Replanning

Health check, SLA monitoring, stall detection, intelligent retry and escalation to human with clear criteria.

What is it:

Each agent emits periodic heartbeat. The supervisor monitors: if heartbeat stops, the agent has crashed. Distinction between "working normally" vs. "locked" vs. "cracked".

Why learn:

Without health monitoring, a stuck agent can block all execution indefinitely while the orchestrator waits for a result that will never come.

Key concepts:

Heartbeat interval · Last seen timestamp · Timeout threshold · Dead vs. slow · Zombie process · Forced termination

What is it:

Each task type has a time SLA (analysis = 30s, report = 120s). If it exceeds, it triggers an alert and potentially replanning or automatic escalation.

Why learn:

SLA monitoring turns intuition into data. Instead of "it seems slow", you have "task X is within 2.3x of the SLA" and you can act proactively.

Key concepts:

SLA by task type · Warning threshold · Breach threshold · SLA breaches/hour · Cascade impact · Contract with downstream

What is it:

Heuristics to detect when an agent is stuck in a loop: N tokens generated without changing state, N consecutive identical tool calls, N iterations without measurable progress.

Why learn:

A looping agent can spend $100+ in tokens before being manually detected. Automatic stall detection is the difference between $0.50 and $500 in cost.

Key concepts:

Stall threshold · Repetition detection · Progress metric · Loop fingerprint · Controlled interruption · Repetitive behavior log

What is it:

Smart retry does not blindly repeat: retry with additional context about the previous error, with a different model, with an alternative approach, or with a different decomposition of the task.

Why learn:

Naive retry (trying the same thing 3x) fails exactly the same way 3x. Smart retry increases success rate from 40% to 85%+ on transient failures.

Key concepts:

Error context injection · Model fallback · Alternative approach · Exponential backoff · Max retries · Retry budget

What is it:

When N retry attempts fail, trigger complete replanning of the subtask: generate new approach using the current state and failure history as context for the planner.

Why learn:

Replanning is what differentiates a robust orchestrator from a retry script. The ability to replan is the essence of controlled autonomy.

Key concepts:

Replanning trigger · Failure context · Alternative decomposition · Replanning budget · Maximum replanning depth · Escalation after replanning

What is it:

Defining clear triggers for human escalation (cost > $X, irreversible action, trust < Y%), implementing asynchronous notification via Slack/email, approval/rejection interface.

Why learn:

Human-in-the-loop is the ultimate safety net of any agentic system. A well-defined pipeline ensures that humans are activated at the right time — not too early, not too late.

Key concepts:

Escalation triggers · Async notification · Approval interface · Approval timeout · Default behavior without response · Audit trail of human decisions

📋 Layer 5a — Governance: Logs and Permissions

Structured logs, immutable audit trail, RBAC for agents and compliance — what regulated companies demand.

What is it:

Each agent action generates a structured log with: agent_id, task_id, session_id, action_type, input, output, tokens_used, cost, timestamp, duration_ms.

Why learn:

Unstructured logs are useless for debugging at scale. Structured logs enable automatic queries, alerts and dashboards without fragile regex.

Key concepts:

JSON structured log · Fixed schema · Required vs. mandatory fields optional · Log level · Correlation by session_id · Log retention

What is it:

Immutable sequence of all orchestrator actions: who (agent/human) decided what, when, with what input and with what result — enterprise requirement in regulated sectors.

Why learn:

Auditing is not optional in fintech, healthcare or legal. The audit trail is the document that proves that the system acted within the authorized limits.

Key concepts:

Append-only log · Hash chain · Immutability · Timestamp with timezone · Actor identifier · Event sourcing pattern

What is it:

Each agent has a role, each role has specific permissions for tools it can use and environments it can access — the same human RBAC model applied to agents.

Why learn:

Without RBAC, any agent can use any tool — an analysis agent could delete production data. RBAC is the principle of least privilege applied to agents.

Key concepts:

Role · Permission · Tool allowlist · Environment restriction · Privilege escalation check · Dynamic role assignment

What is it:

Bank reading tool: any agent. Writing Tool: Approved agents only. Deployment tool: only with explicit human approval. Risk hierarchy by tool.

Why learn:

Permission granularity per tool is what allows maximum autonomy where it is safe and requires approval where there is real risk — without blocking everything or releasing everything.

Key concepts:

Tool risk level · Read vs. write vs. execute · Approval gate · Tool interceptor · Permission middleware · Blast radius of each tool

What is it:

Ingest logs from multiple agents into Elasticsearch or Loki, enabling cross-agent queries, automatic alerts by log pattern, and configurable retention by data type.

Why learn:

Agent-distributed logs are impossible to analyze manually. Centralization is what makes debugging and auditing viable on a scale of dozens of agents.

Key concepts:

Log aggregation · Elasticsearch index · Loki label · Grafana query · Log retention policy · Log storage cost

What is it:

How to prepare an audit report showing all agent actions in a period — format required by regulated companies in fintech, healthcare and legal.

Why learn:

Compliance is not an additional feature — it is the minimum requirement for the agentic system to be approved in enterprise environments. Preparing this from the beginning avoids expensive rewrites.

Key concepts:

Audit report · Mandatory retention period · Export to regulator · Anonymization of sensitive data · LGPD/GDPR with agents · Digital signature of logs

💰 Layer 5b — Governance: Cost and Audit

Budget by execution, real-time alerts, cost allocation by department and agent ROI.

What is it:

Real cases of agents in a loop that generated costs of $500+ in less than an hour, how this happens and why cost controls are mandatory from the first deployment.

Why learn:

Without cost monitoring, a loop bug in production can result in a $10,000+ invoice in hours. This ruined startups. It's not an exaggeration — it's documented.

Key concepts:

Cost loop · Cost per token · Unexpected invoice · Kill switch · Emergency budget cap · Cost postmortem

What is it:

Each job has a configurable maximum budget (e.g. $5.00). The orchestrator monitors accumulated costs in real time and stops execution when it reaches the limit, delivering partial results.

Why learn:

Budgeting by execution is the simplest and most effective control. With 5 lines of code you can prevent 99% of accidental overspends in production.

Key concepts:

Budget per job · Real-time cost tracking · Soft vs. hard limit · Graceful stop · Partial result delivery · Budget warning up to 80%

What is it:

Multilayer alert system: per execution (above X), per hour (above Y), per user/tenant (above Z monthly), with Slack and email integration.

Why learn:

Alerts are the difference between discovering the problem in 5 minutes vs. at the end of the month on the invoice. Multilayer ensures that no anomalous pattern goes unnoticed.

Key concepts:

Alert threshold · Slack webhook · Email notification · Alert fatigue · Smart alerting (baselines) · Alert escalation

What is it:

Each execution has an associated cost_center (department, project, customer). Monthly report automated by cost_center for chargeback and internal accountability.

Why learn:

Without cost allocation, the AI ​​cost remains in a generic account with no person responsible. Cost allocation creates accountability and incentives for efficient use.

Key concepts:

Cost center tag · Chargeback · Showback · Monthly report · Department vs. project vs. customer · Infrastructure overhead

What is it:

Dashboard showing total cost, average cost by type of task, time trend, and comparison with equivalent human cost to calculate agents' real ROI.

Why learn:

ROI is the final argument to justify investment in agents for stakeholders. No cost vs. cost data value, it is difficult to maintain the budget to expand the system.

Key concepts:

Cost per result · Equivalent human cost · Calculated ROI · Break-even point · Efficiency trend · Presentation for C-level

What is it:

Techniques to reduce cost while maintaining quality: heterogeneous routing by complexity, caching similar results with embeddings, batching multiple LLM calls into one.

Why learn:

The combination of heterogeneous routing + caching can reduce total cost by 70-90% without noticeable quality loss — transforming system ROI.

Key concepts:

Semantic caching · Prompt batching · Heterogeneous routing · Cache hit rate · Minimum acceptable quality · Pre/post optimization measurement

📊 Layer 6a — Observability: Metrics

SLIs, SLOs, Prometheus + Grafana, smart alerts and executive dashboards for multi-agent systems.

What is it:

The minimum set of technical metrics: success_rate per task type, P50/P95/P99 latency, tokens_per_task, cost_per_task, error_rate, queue_depth per agent.

Why learn:

Without technical metrics, you operate “in the dark.” With them, you detect degradation before it affects the final result and identify bottlenecks with surgical precision.

Key concepts:

Success rate · P50/P95/P99 · Error budget · Queue depth · Throughput · Metrics per agent vs. by task type

What is it:

Metrics that business stakeholders understand: objective_achieved_rate (% of final objectives met), total_time_per_workflow, cost_per_result, NPS of system users.

Why learn:

Technical metrics are for engineers. Business metrics are for managers and directors — and they are what determine the project’s future budget.

Key concepts:

Goal completion rate · Time to completion · Cost per outcome · What is the "result" of your system · Leading vs. lagging

What is it:

SLI (Service Level Indicator): analysis success rate. SLO (Service Level Objective): this SLI must be >= 95% 99.9% of the time. Error budget = what is left when the SLO is met.

Why learn:

SLOs turn "the system looks OK" into "the system is X% within the agreed target." It is the language that aligns engineering and business around quality.

Key concepts:

SLI definition · SLO target · Error budget · Burn rate · SLO breach · SLO negotiation with stakeholders

What is it:

Configure prometheus_client in Python, expose /metrics endpoint, create counters and histograms for agent metrics, import ready-to-use Grafana dashboard.

Why learn:

Prometheus + Grafana is the standard observability stack in 80% of tech companies. Knowing how to instrument Python code is a skill in high demand and low supply.

Key concepts:

prometheus_client · Counter · Histogram · Gauge · PromQL query · Grafana panel · /metrics scraping

What is it:

Alert when error rate exceeds historical baseline (not on every single error), using simple anomaly detection — avoiding alert fatigue while capturing real problems.

Why learn:

Poorly calibrated alerts create alert fatigue — staff ignore all alerts because 95% are false positives. Smart alerts ensure every alert deserves attention.

Key concepts:

Historical baseline · Rolling average · Standard deviation · False positive rate · SLO burn rate alert · PagerDuty integration

What is it:

Executive dashboard with: jobs running now, success rate in the last 24 hours, total cost today, most used agents, top errors of the week — designed to be read in 30 seconds.

Why learn:

Executive dashboards are what make leaders trust the system. A clear dashboard increases the likelihood of expanding the project budget and scope.

Key concepts:

KPI panel · Traffic light status · Trend sparkline · Daily summary · Top N errors · Drill-down for technical details

🔍 Layer 6b — Observability: Tracking

Distributed tracing with OpenTelemetry, decision tracking and debugging of complex executions.

What is it:

Each execution has a unique trace_id propagated across all participating agents, allowing you to reconstruct the complete end-to-end flow of decisions and actions.

Why learn:

Without trace_id, a failure in a multi-agent system can take hours to diagnose — you have logs from 5 agents without knowing how they are related temporally.

Key concepts:

trace_id · Context propagation · W3C TraceContext · Agent correlation · End-to-end latency · Bottleneck identified by trace

What is it:

Each LLM call, each tool call, each routing decision becomes a span with parent_span_id, forming a tree that represents the hierarchy of execution calls.

Why learn:

The span tree is the map of every execution. With it, you immediately see which span consumed 80% of the time — and intervene in the right place.

Key concepts:

Span · Parent span · Span duration · Span attributes · Waterfall visualization · Critical path no trace

What is it:

Instrument the orchestrator in Python with opentelemetry-sdk, using decorators and context managers to create spans, sending to Jaeger or Grafana Tempo for visualization.

Why learn:

OpenTelemetry is the open standard that works with any backend (Jaeger, Tempo, Zipkin, Datadog). Instrument once, change backend without rewriting code.

Key concepts:

opentelemetry-api · Tracer · @tracer.start_as_current_span · OTLP exporter · Jaeger UI · Span attributes for LLM calls

What is it:

In addition to what happened, record why: what was LLM's reasoning for each routing decision, each tool choice, each quality assessment — intention traceability.

Why learn:

Decision tracking is what allows you to iteratively improve the system. Without understanding the reason for each choice, optimization is blind trial and error.

Key concepts:

Decision log · Reasoning trace · LLM rationale · Span attribute for decision · Explainability · Reasoning audit

What is it:

Given a failed execution, use the trace to identify exactly where and why: waterfall analysis, identification of the problematic span, root cause diagnosis.

Why learn:

Debugging a multi-agent system without a trace is like debugging code without a stack trace. With trace, what would take hours takes minutes — an essential skill in production.

Key concepts:

Error span · Root cause analysis · Waterfall analysis · Span that failed vs. span affected · Play in staging · Fix verified via trace

What is it:

When a latency alert triggers in Prometheus, how to automatically correlate with the specific trace that caused the problem — direct navigation from metric to trace.

Why learn:

Full correlation closes the observability loop: alert → metric → trace → log → root cause. Without correlation, each layer exists in independent silos.

Key concepts:

Exemplary (Prometheus) · TraceID in log · Grafana Tempo integration · Metric click-through to trace · MELT (Metrics Events Logs Traces)

⚡ Cost-Optimization: Heterogeneous Routing

Sort calls by complexity and route to the cheapest model capable — reducing cost by up to 90% without loss of quality.

What is it:

Using GPT-4o or Claude Opus for all LLM calls — including simple sorts, formatting, and trivial extractions — results in 10-50x more cost than necessary.

Why learn:

This is the most common error in first generation agentic systems. Understanding the problem is the first step towards a solution that reduces costs by the same order of magnitude.

Key concepts:

Cost by Model · 2026 Pricing Table · Simple vs. Simple Tasks complex · Overkill analysis · Estimated wasted cost

What is it:

Sort each LLM call by complexity and route to the cheapest model with acceptable quality: GPT-4o-mini for simple, GPT-4o for medium, o1 for deep reasoning.

Why learn:

Heterogeneous routing is the most powerful cost reduction lever. Implemented correctly, it reduces total cost by 70-90% while maintaining 95%+ of the original quality.

Key concepts:

Model tier (cheap/medium/expensive) · Complexity threshold · Quality floor · Routing policy · Fallback to higher tier · A/B quality test

What is it:

A very cheap model (Haiku, GPT-4o-mini) classifies the complexity of each task before routing. The cost of the classifier is < 1% of the total execution cost.

Why learn:

The classifier is the key to the entire heterogeneous routing system. Calibrating your accuracy is what determines how much cost you save vs. how much quality you lose.

Key concepts:

Complexity features · Few-shot classifier · Classifier accuracy · Classifier cost vs. savings generated · Calibration with real examples

What is it:

Plan once with expensive model (1 call), execute N times with cheap model (N calls). Reduces total cost by up to 90% in workflows with many execution tasks.

Why learn:

Plan-and-Execute is the most impactful architectural pattern for cost reduction. A well-structured planner can transform the cost of a workflow from $10 to $1.

Key concepts:

Planner (expensive model) Executor (cheap model) Plan as structured prompt Number of executions Planning cost vs. total cost

What is it:

When two executions ask semantically similar questions to LLM, return cached result instead of making another expensive call. Cache with embeddings detects similarity in addition to exact match.

Why learn:

In systems with multiple users asking similar questions, semantic caching can reduce LLM calls by 40-60%, with a proportional impact on cost.

Key concepts:

Embedding similarity · Cosine distance threshold · Cache hit rate · Cache invalidation · TTL per query type · GPTCache / LangChain cache

What is it:

Same workflow executed with and without optimization: table comparing total cost, latency and quality score (evaluation by LLM or human). Where to accept the quality/cost trade-off.

Why learn:

Benchmark data is what turns optimization arguments into rational decisions. Without measuring, you don't know if the optimization was worth it or where to stop.

Key concepts:

Baseline without optimization · Post-optimization cost · Quality score · Pareto frontier cost/quality · Acceptance point · Optimization ROI
