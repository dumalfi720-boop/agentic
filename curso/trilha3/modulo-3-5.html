🏛️ MCP Architecture Review

MCP (Model Context Protocol) is a protocolclient-serverstandardized for communication between AI agents and tool/data sources. An agent (MCP client) discovers tools available on an MCP server and calls them via JSON-RPC.

🌐MCP full flow

1
Connection:Agent connects to the server via stdio (local) or HTTP+SSE (remote)
2
Discovery:Agent calls tools/list — receives list of tools with name, description and schema
3
Selection:LLM chooses which tool to call based on descriptions and context
4
Execution:Agent calls tools/call — server executes and returns result

💡Why MCP is important

Without MCP, each framework has its own way of defining tools — LangChain, CrewAI, Llama Index are all incompatible. With MCP, a tool server works with any compatible client: LangGraph, CrewAI, Claude Desktop, Cursor.

MCP is theAI Tools USB-C— a pattern that connects everything.

🛠️ Creating complete MCP server with FastMCP

FastMCP is the Python library that makes creating MCP servers trivial. You decorate functions with@mcp.tool()and the server automatically exposes tools/list and tools/call with schema generated from type hints.

🔨Full MCP Server

# server.py
from fastmcp import FastMCP
from pydantic import BaseModel
from typing import Optional
import httpx

# Initialize the server
mcp = FastMCP(
    name="empresa-tools",
    description="Ferramentas internas da Empresa XYZ para agentes de IA"
)

#Tool 1: Customer search
@mcp.tool()
async def buscar_cliente(
    customer_id: str,
    include_history: bool = False
) -> dict:
    """
    Busca informações de um cliente pelo ID.

    Args:
        customer_id: ID único do cliente (formato: CUS_xxxxx)
        include_history: Se True, inclui histórico de compras

    Returns:
        Dict com nome, email, plano e histórico (opcional)
    """
    # Real logic: database query
    cliente = await db.get_customer(customer_id)
    result = {
        "id": cliente.id,
        "nome": cliente.nome,
        "email": cliente.email,
        "plano": cliente.plano,
        "created_at": cliente.created_at.isoformat()
    }
    if include_history:
        result["compras"] = await db.get_purchases(customer_id)
    return result

# Tool 2: sending email
@mcp.tool()
async def enviar_email(
    destinatario: str,
    assunto: str,
    corpo: str,
    template: Optional[str] = None
) -> dict:
    """
    Envia email para um destinatário usando templates corporativos.

    Returns:
        Dict com message_id e status de entrega
    """
    msg_id = await email_service.send(
        to=destinatario,
        subject=assunto,
        body=corpo,
        template=template
    )
    return {"message_id": msg_id, "status": "enviado"}

# Resource: access to static data
@mcp.resource("config://empresa/politicas")
async def get_politicas() -> str:
    """Retorna as políticas de atendimento da empresa em Markdown."""
    return open("politicas.md").read()

# Start the server
if __name__ == "__main__":
    mcp.run(transport="stdio")       # location
    # mcp.run(transport="sse", port=8080) # HTTP

💡Docstrings generate the LLM descriptions

The docstrings of your functions become the descriptions of the tools that LLM sees. A clear and specific docstring makes LLM choose the right tool at the right time. Treat the docstrings as prompt engineering for your MCP server.

🔒 Authentication and security on MCP servers

MCP servers accessible via HTTP need to authenticate clients. Bearer token in the Authorization header is the simplest pattern for internal use. OAuth 2.0 for multi-user scenarios and granular scopes.

🔐Bearer authentication with FastAPI

# insurance_server.py
from fastmcp import FastMCP
from fastapi import FastAPI, HTTPException, Security
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials
import os, hashlib, hmac

app = FastAPI()
security = HTTPBearer()
mcp = FastMCP(name="empresa-tools-seguro")

# Valid tokens (in production: database)
VALID_TOKENS = {
    os.environ["AGENTE_TOKEN_1"]: "agente-producao",
    os.environ["AGENTE_TOKEN_2"]: "agente-staging",
}

def validar_token(credentials: HTTPAuthorizationCredentials) -> str:
    """Valida o Bearer token e retorna o nome do cliente."""
    token = credentials.credentials
    cliente = VALID_TOKENS.get(token)
    if not cliente:
        raise HTTPException(status_code=401, detail="Token inválido")
    return cliente

# Authentication middleware
@app.middleware("http")
async def auth_middleware(request, call_next):
    if request.url.path.startswith("/mcp"):
        auth = request.headers.get("Authorization", "")
        if not auth.startswith("Bearer "):
            return JSONResponse({"error": "Unauthorized"}, status_code=401)
        token = auth.removeprefix("Bearer ")
        if token not in VALID_TOKENS:
            return JSONResponse({"error": "Invalid token"}, status_code=401)
        # Audit log
        logger.info(f"Tool call from {VALID_TOKENS[token]}")
    return await call_next(request)

# Simple rate limiting
from slowapi import Limiter
from slowapi.util import get_remote_address
limiter = Limiter(key_func=get_remote_address)

@app.get("/mcp/tools/list")
@limiter.limit("60/minute")  # 60 calls per minute
async def list_tools(request: Request):
    return await mcp.handle_list_tools()

# Client configuring authentication
# In LangGraph/CrewAI, pass the header in the MCP server configuration:
# MCPClient(url="https://...", headers={"Authorization": "Bearer TOKEN"})

⚠️Security checklist for MCP in production

  • ✗Never use hardcoded tokens in code — use environment variables
  • ✗Never expose the server without HTTPS in production
  • ✓Audit all calls with timestamp, client and parameters
  • ✓Rate limit per customer to prevent abuse
  • ✓Validate and sanitize all inputs before executing

🔭 Dynamic tool discovery

The MCP protocol allows agents to discover toolsdynamically at runtime— without having to hardcode the tools into the agent code. Add a tool to the server and all agents will automatically see it on the next connection.

🔍Complete Discovery Example

# What the server returns in tools/list
{
  "tools": [
    {
      "name": "buscar_cliente",
      "description": "Busca informações de um cliente pelo ID...",
      "inputSchema": {
        "type": "object",
        "properties": {
          "customer_id": {"type": "string", "description": "ID único (CUS_xxxxx)"},
          "include_history": {"type": "boolean", "default": false}
        },
        "required": ["customer_id"]
      }
    },
    {
      "name": "enviar_email",
      "description": "Envia email para destinatário usando templates...",
      "inputSchema": { ... }
    }
  ]
}

# How LangGraph consumes the MCP server
from langchain_mcp_adapters.client import MultiServerMCPClient

async def criar_agente_com_mcp():
    async with MultiServerMCPClient({
        "empresa": {
            "url": "https://mcp.empresa.com/sse",
            "transport": "sse",
            "headers": {"Authorization": f"Bearer {TOKEN}"}
        },
        "github": {
            "command": "npx",
            "args": ["-y", "@modelcontextprotocol/server-github"],
            "transport": "stdio"
        }
    }) as client:
        # Tools automatically discovered from both servers
        tools = client.get_tools()  # complete list combined

        llm = ChatOpenAI(model="gpt-4o")
        agente = llm.bind_tools(tools)

        # Agent can use tools from both servers
        resposta = await agente.ainvoke("Liste os PRs abertos do repo X")

🚀 MCP in production: deployment and operation

A local MCP server only works in development. For agents in production, the server needs to be an HTTP service with high availability. Railway, Fly.io and Docker are the most used options.

🐳Dockerfile + Railway deploy

# Dockerfile
FROM python:3.11-slim
WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

# Health check endpoint
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s \
  CMD curl -f http://localhost:8080/health || exit 1

EXPOSE 8080
CMD ["python", "servidor.py", "--transport", "sse", "--port", "8080"]

🌐 Ready-made server ecosystem

There is a growing ecosystem of over 200 ready-to-use MCP servers. Before building from scratch, check to see if it already exists. Most common integrations are already available as a plug-and-play server.

Official Anthropic Servers

  • •filesystem— access to local file system
  • •github— repositories, PRs, issues
  • •postgres— secure SQL queries
  • •brave-search— web search with Brave API
  • •slack— messages and channels
  • •puppeteer— browser automation

Community Servers

  • •notion— workspace and databases
  • •linear— project management
  • •stripe— payments and subscriptions
  • •jira— tickets and sprints
  • •aws-kb—AWS Knowledge Base
  • •playwright— automated E2E testing
