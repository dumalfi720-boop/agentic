🗺️ What is a state graph

LangGraph represents the flow of an agent as adirected graph, where each node is a Python function that takes the current state and returns an updated version. Unlike linear pipelines, the graph can have branches, loops and decision points — just like the logic of a human being solving a complex problem.

🧩State graph anatomy

Three essential components make up any LangGraph: theState(TypedDict with the shared data), theUs(functions that process the state) and theEdges(connections that define the flow).

# 1. State: the heart of the graph
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, START, END
import operator

class AgentState(TypedDict):
    mensagem: str
    historico: Annotated[list, operator.add]  # acumula
    iteracoes: int
    aprovado: bool

# 2. Nodes: Functions that transform state
def processar(state: AgentState) -> AgentState:
    return {"iteracoes": state["iteracoes"] + 1}

# 3. Graph: connects everything
grafo = StateGraph(AgentState)
grafo.add_node("processar", processar)
grafo.add_edge(START, "processar")
grafo.add_edge("processar", END)

Immutable state by convention

  • ✓Return a new dict with updated fields
  • ✓Use Annotated[list, operator.add] to accumulate lists
  • ✓Fields not returned retain previous value
  • ✓TypedDict guarantees type safety throughout execution

Common Pitfalls

  • ✗Don't mutate the state directly (state["x"] = ...)
  • ✗Don't forget to return fields you want to update
  • ✗Avoid very large state — impacts serialization
  • ✗Don't use non-serializable types in state

💡Practical Tip

Design the state first, before any nodes. Think: what data needs to flow through the system? Which ones need to be accumulated? The state's TypedDict is the data contract for the entire graph.

🔗 Nodes and edges: building blocks

Nodes are pure Python functions — they receive the state, do something (call LLM, run tool, make decision) and return a partial update of the state. Edges are the connections:fixed(they always go to the same node) orconditionals(they choose the next node based on the current state).

⚙️Fixed vs. Fixed Edges conditionals

# FIXED edge: always goes to the same next node
grafo.add_edge("analisar", "formatar")  # sempre

# Edge CONDITIONAL: function decides the next node
def decidir_rota(state: AgentState) -> str:
    if state["aprovado"]:
        return "finalizar"
    elif state["iteracoes"] >= 3:
        return "escalar"
    else:
        return "revisar"

grafo.add_conditional_edges(
    "avaliar",          # source node
    decidir_rota,       # router function
    {                   # mapping string → node
        "finalizar": "finalizar",
        "escalar": "escalar",
        "revisar": "revisar"
    }
)

# Multiple possible destinations from the same node
grafo.add_node("avaliar", avaliar_resultado)
grafo.add_node("finalizar", gerar_output)
grafo.add_node("escalar", notificar_humano)
grafo.add_node("revisar", melhorar_resposta)

📊Why conditional edges are powerful

  • •Data-driven branching:the flow changes based on actual state, not hardcoded logic in the node
  • •Separation of concerns:the node does the analysis, the edge does the routing — clear responsibilities
  • •Testability:the router function can be unit tested without running the entire graph
  • •View:LangGraph can generate visual graph diagram including all possible paths

⚙️ StateGraph in practice: complete code

See a complete analytics agent built from the ground up. The patterndefine state → create nodes → connect edges → compile → executeis always the same, regardless of the complexity of the system.

🏗️Complete Analysis Agent

from langgraph.graph import StateGraph, START, END
from langchain_openai import ChatOpenAI
from typing import TypedDict

# 1. Sets the state
class AnaliseState(TypedDict):
    pergunta: str
    rascunho: str
    critica: str
    resposta_final: str
    tentativas: int

llm = ChatOpenAI(model="gpt-4o-mini")

# 2. Defines the nodes
def gerar_rascunho(state: AnaliseState):
    resposta = llm.invoke(f"Analise: {state['pergunta']}")
    return {"rascunho": resposta.content, "tentativas": state.get("tentativas", 0) + 1}

def criticar(state: AnaliseState):
    prompt = f"Critique esta análise: {state['rascunho']}"
    critica = llm.invoke(prompt)
    return {"critica": critica.content}

def refinar(state: AnaliseState):
    prompt = f"Melhore com base na crítica: {state['rascunho']}\nCrítica: {state['critica']}"
    melhorado = llm.invoke(prompt)
    return {"rascunho": melhorado.content}

def finalizar(state: AnaliseState):
    return {"resposta_final": state["rascunho"]}

# 3. Routing logic
def checar_qualidade(state: AnaliseState) -> str:
    if "APROVADO" in state["critica"] or state["tentativas"] >= 3:
        return "finalizar"
    return "refinar"

# 4. Assemble the graph
grafo = StateGraph(AnaliseState)
grafo.add_node("gerar", gerar_rascunho)
grafo.add_node("criticar", criticar)
grafo.add_node("refinar", refinar)
grafo.add_node("finalizar", finalizar)

grafo.add_edge(START, "gerar")
grafo.add_edge("gerar", "criticar")
grafo.add_conditional_edges("criticar", checar_qualidade, {
    "finalizar": "finalizar",
    "refinar": "refinar"
})
grafo.add_edge("refinar", "criticar")  #loop!
grafo.add_edge("finalizar", END)

# 5. Compiles and runs
app = grafo.compile()
resultado = app.invoke({"pergunta": "Qual o impacto do LangGraph em produção?"})

💡Preview the graph before running

# Generate PNG image of the graph
from IPython.display import Image
Image(app.get_graph().draw_mermaid_png())

# Or print in text format
print(app.get_graph().draw_ascii())

🔁 Controlled cycles and loops

The ability toreturn to a previous nodeIt's what makes LangGraph fundamentally different from linear pipelines. Controlled loops allow for automatic retry, iterative reflection, and human approval before continuing.

A

Auto reflection loop

Agent generates → criticizes → improves → repeats until quality or limit

# Counter in state prevents infinite loop
def checar_loop(state) -> str:
    if state["qualidade"] > 0.8:
        return END
    if state["tentativas"] >= 5:
        return "escalar"  # fallback
    return "melhorar"     # back
B

Human-in-the-loop with interrupt

Pause the graph, wait for human approval, resume from the same point

# Compile with breakpoint
app = grafo.compile(
    interrupt_before=["acao_critica"]
)

# Execute until interrupt
estado = app.invoke(input_inicial, config={"thread_id": "1"})

# Human approves and resumes
app.update_state(config, {"aprovado": True})
resultado = app.invoke(None, config=config)

⚠️Warning: infinite loops

Every loop must have a guaranteed exit criterion. Use an iteration counter on the state and set a maximum limit. An agent stuck in an infinite loop consumes tokens nonstop — and the bill arrives at the end of the month.

💾 Checkpoints and persistence

LangGraph saves state after each node runs using acheckpointer. This allows you to resume interrupted workflows, audit each step of execution and implement human-in-the-loop without losing progress.

💽MemorySaver and PostgresSaver

# Development: memory (does not persist between reboots)
from langgraph.checkpoint.memory import MemorySaver
checkpointer = MemorySaver()
app = grafo.compile(checkpointer=checkpointer)

# Production: PostgreSQL (persists between restarts)
from langgraph.checkpoint.postgres import PostgresSaver
with PostgresSaver.from_conn_string(DATABASE_URL) as checkpointer:
    app = grafo.compile(checkpointer=checkpointer)

# thread_id identifies the session — ALWAYS use one
config = {"configurable": {"thread_id": "usuario_123_sessao_456"}}

# Each invocation with the same thread_id resumes from the previous point
resultado1 = app.invoke({"mensagem": "olá"}, config=config)
resultado2 = app.invoke({"mensagem": "continue"}, config=config)

# Inspect the saved state
estado_atual = app.get_state(config)
print(estado_atual.values)   # current data
print(estado_atual.next)     # next node to run

# Complete history of all checkpoints
for snapshot in app.get_state_history(config):
    print(snapshot.config, snapshot.values)

MemorySaver

  • • Ideal for development and testing
  • • State in RAM — fast
  • • You lose everything when you restart the process
  • • No database dependency

PostgresSaver/SQLiteSaver

  • • Production with real resilience
  • • Survives restarts
  • • Allows historical auditing
  • • Concurrent multi-thread_id support

📡 Streaming results

Withgraph.stream(), you receive real-time events with each node you run. This transforms the UX: the user sees the systemprocessinginstead of waiting for a response that takes 30 seconds.

📡Streaming Modes

# stream_mode="values": full state after each node
for estado in app.stream(input, stream_mode="values"):
    print(f"Estado: {estado}")

# stream_mode="updates": just the diff (more efficient)
for chunk in app.stream(input, stream_mode="updates"):
    no, atualizacao = list(chunk.items())[0]
    print(f"Nó '{no}' atualizou: {atualizacao}")

# stream_mode="debug": everything, including internal metadata
for evento in app.stream(input, stream_mode="debug"):
    print(evento)

# Streaming LLM tokens within nodes
async for evento in app.astream_events(input, version="v2"):
    if evento["event"] == "on_chat_model_stream":
        token = evento["data"]["chunk"].content
        print(token, end="", flush=True)

# Integration with FastAPI SSE for real-time UI
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app_api = FastAPI()

@app_api.get("/executar")
async def executar_agente(pergunta: str):
    async def gerar():
        async for chunk in app.astream({"pergunta": pergunta}):
            yield f"data: {str(chunk)}\n\n"
    return StreamingResponse(gerar(), media_type="text/event-stream")

💡Use astream_events for rich UI

The methodastream_events(version="v2")emits granular events: node start, node end, LLM tokens, tool calls, and more. With these events, you can build a UI that shows exactly what the agent is doing at any given moment.
