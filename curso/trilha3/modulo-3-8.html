👁️ Why observability is different from logging

Logging captures isolated events.Observabilityconnects these events into a coherent view of the system — traces that show the complete flow, spans that measure each step, metrics that reveal patterns, and evaluation that judges the quality of the output.

🔭The 4 pillars of observability for agents

Traces

Complete flow of an execution: from input to final output, with all intermediate nodes and tool calls. Answer: "What exactly happened in this run?"

Spans

Duration and data of each individual stage. Where was the time spent? Which node is slow? Which tool is failing the most? Answer: "Where is the bottleneck?"

Metrics

Aggregations over time: error rate, average cost per run, P95 latency. Responds: "Is the system healthy? Is it getting worse?"

Assessment

Output quality: did the agent respond correctly? Was the answer relevant? Did you use the right information? Answer: "Is the system doing the job well?"

⚠️The cost of not having observability

  • •You discover problems by the user complaining — not by alerting
  • •4 hour debug that could be 10 minutes with a trace
  • •Cost of tokens growing without anyone knowing why
  • •Quality regression after prompt change undetected for weeks

🔬 Advanced LangSmith

LangSmith integrates with LangGraph in 3 configuration lines. Once configured, it automatically captures complete traces of each run — without additional code — and exposes an interface for debugging, analyzing and comparing runs.

⚙️Advanced Setup and Usage

# Minimum configuration (3 environment variables)
import os
os.environ["LANGCHAIN_TRACING_V2"] = "true"
os.environ["LANGCHAIN_ENDPOINT"] = "https://api.smith.langchain.com"
os.environ["LANGCHAIN_API_KEY"] = "ls__..."

# Optionally define the project
os.environ["LANGCHAIN_PROJECT"] = "meu-agente-producao"

# After that, EVERY LangGraph execution is automatically tracked
resultado = app.invoke(input, config=config)
# → Trace visible on smith.langchain.com immediately

# Tags to filter runs on the dashboard
resultado = app.invoke(input, config={
    **config,
    "tags": ["producao", "usuario-premium", "analise-mercado"],
    "metadata": {"user_id": "u123", "version": "2.1.0"}
})

# Creating datasets for regression testing
from langsmith import Client
client = Client()

# Create dataset from previous successful runs
dataset = client.create_dataset("golden-set-analise-mercado")
runs = client.list_runs(
    project_name="meu-agente-producao",
    filter='and(eq(feedback_key, "qualidade"), gte(feedback_score, 4))'
)
for run in runs:
    client.create_example(
        inputs=run.inputs,
        outputs=run.outputs,
        dataset_id=dataset.id
    )

# Batch evaluation on the dataset
from langsmith.evaluation import evaluate, LangChainStringEvaluator

evaluator = LangChainStringEvaluator("criteria", config={
    "criteria": "helpfulness"
})
results = evaluate(
    lambda inputs: app.invoke(inputs),
    data=dataset.name,
    evaluators=[evaluator],
    experiment_prefix="v2.1-vs-v2.0"
)

🔍What you see at LangSmith

  • •Complete trace with each graph node, inputs and outputs
  • •Time spent on each step (identify bottlenecks)
  • •Token cost per run and per node
  • •Filter by status (error, success), tags and metadata
  • •Side-by-side comparison between two runs
  • •User feedback noted directly in the run

🔆 Arize Phoenix: open-source observability

Arize Phoenix is ​​the open-source alternative to LangSmith — it runs locally, uses OpenTelemetry as a standard, visualizes embeddings in 3D and allows quality assessment. Ideal for those who cannot use SaaS due to compliance or cost.

🔆Setup and use of Phoenix

# Installation
# pip install arize-phoenix openinference-instrumentation-langchain

import phoenix as px
from openinference.instrumentation.langchain import LangChainInstrumentor
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider

# 1. Start Phoenix locally
session = px.launch_app()
# → UI available at http://localhost:6006

# 2. Configure OpenTelemetry to export to Phoenix
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.trace.export import SimpleSpanProcessor

tracer_provider = TracerProvider()
tracer_provider.add_span_processor(
    SimpleSpanProcessor(
        OTLPSpanExporter(endpoint=session.url + "/v1/traces")
    )
)
trace.set_tracer_provider(tracer_provider)

# 3. Automatically instrument LangChain/LangGraph
LangChainInstrumentor().instrument()

#4. From here on, every run appears in Phoenix!
resultado = app.invoke(input)
# → Trace visible at http://localhost:6006

# Exporting to CSV for offline analysis
df = px.Client().get_spans_dataframe()
df.to_csv("traces_export.csv", index=False)

# Quality assessment in Phoenix
from phoenix.evals import (
    HallucinationEvaluator, QAEvaluator, run_evals
)
from phoenix.evals.models import OpenAIModel

eval_model = OpenAIModel(model="gpt-4o-mini")
hallucination_evaluator = HallucinationEvaluator(eval_model)
qa_evaluator = QAEvaluator(eval_model)

results = run_evals(
    dataframe=df,
    evaluators=[hallucination_evaluator, qa_evaluator],
    provide_explanation=True
)

💰 Business metrics vs. techniques

Technical stakeholders want latency and error rate. Business stakeholders want ROI and user satisfaction. You need to monitorboth— and know how to translate one to the other when you need to justify investment.

Technical Metrics

  • •
    Latency P50/P95/P99

    P95 < 30s is acceptable. P99 reveals extreme cases.

  • •
    Error rate by type

    API timeout, context too long, guardrail triggered

  • •
    Tokens per run

    Input + output + cache hits. Direct cost.

  • •
    We slower

    Where time is spent within the graph

Business Metrics

  • •
    Objective Completion Rate

    % of runs that reach the user's goal

  • •
    Cost per task completed

    R$ in tokens / (successful runs)

  • •
    Time saved vs. manual

    Human hours replaced by execution

  • •
    agentic system NPS

    Do users recommend using the agent?

📊Implementing custom metrics

import time
from dataclasses import dataclass
from langsmith import Client

client = Client()

# Tracking business metrics via feedback
async def executar_com_metricas(input_data: dict, user_id: str):
    start_time = time.time()

    # Run the agent
    resultado = await app.ainvoke(input_data, config={
        "configurable": {"thread_id": f"{user_id}_{int(start_time)}"},
        "metadata": {"user_id": user_id}
    })

    latencia = time.time() - start_time
    objetivo_atingido = verificar_objetivo(resultado, input_data)

    # Send metrics to LangSmith as feedback
    run_id = resultado.get("run_id")  # trace ID
    if run_id:
        client.create_feedback(
            run_id=run_id,
            key="objetivo_atingido",
            score=1.0 if objetivo_atingido else 0.0
        )
        client.create_feedback(
            run_id=run_id,
            key="latencia_segundos",
            value=latencia
        )
        client.create_feedback(
            run_id=run_id,
            key="custo_usd",
            value=calcular_custo(resultado)
        )

    return resultado

# Aggregated metrics query
runs = client.list_runs(
    project_name="producao",
    start_time=datetime.now() - timedelta(days=7)
)
taxa_sucesso = sum(r.feedback_stats.get("objetivo_atingido", {}).get("avg", 0)
                   for r in runs) / len(list(runs))

🚨 Alerts and dashboards in production

Proactive alerts allow you to fix issues before users are affected. The golden rule:alert on symptoms, not causes— high error rate is a symptom, latency of a specific tool is the cause.

🚨Threshold alert system

import asyncio
from datetime import datetime, timedelta

# Alert Thresholds
ALERTAS = {
    "taxa_erro_pct": {"warning": 5.0, "critical": 15.0},
    "latencia_p95_seg": {"warning": 30.0, "critical": 60.0},
    "custo_hora_usd": {"warning": 10.0, "critical": 25.0},
    "qualidade_score": {"warning": 3.5, "critical": 2.5},  # minimum
}

async def monitorar_metricas():
    """Loop de monitoramento que roda a cada 5 minutos."""
    while True:
        metricas = await coletar_metricas_recentes(janela_minutos=5)
        alertas_disparados = []

        for metrica, valor in metricas.items():
            threshold = ALERTAS.get(metrica)
            if not threshold:
                continue

            if valor >= threshold.get("critical", float("inf")):
                alertas_disparados.append({
                    "metrica": metrica,
                    "valor": valor,
                    "nivel": "CRITICAL",
                    "threshold": threshold["critical"]
                })
            elif valor >= threshold.get("warning", float("inf")):
                alertas_disparados.append({
                    "metrica": metrica,
                    "valor": valor,
                    "nivel": "WARNING",
                    "threshold": threshold["warning"]
                })

        if alertas_disparados:
            await enviar_alerta(alertas_disparados)

        await asyncio.sleep(300)  # 5 minutes

async def enviar_alerta(alertas: list):
    """Envia alerta para Slack e/ou PagerDuty."""
    mensagem = f"🚨 Alerta em produção: {datetime.now()}\n"
    for alerta in alertas:
        emoji = "🔴" if alerta["nivel"] == "CRITICAL" else "🟡"
        mensagem += f"{emoji} {alerta['metrica']}: {alerta['valor']:.2f} "
        mensagem += f"(threshold: {alerta['threshold']})\n"

    #Slack
    await slack_client.post(channel="#alertas-prod", text=mensagem)

    # PagerDuty for critics
    if any(a["nivel"] == "CRITICAL" for a in alertas):
        await pagerduty.trigger(summary=mensagem, severity="critical")

💡Hysteresis: avoid flood alert

Configure hysteresis to avoid alert spam: only alert when the threshold is exceeded for 3 consecutive windows, and only resolve when it falls below for 2 consecutive windows. A momentary spike shouldn't wake the team up at 3am.

⚖️ Continuous assessment with LLM-as-judge

Manual quality assessment does not scale to production. WithLLM-as-judge, a second LLM automatically evaluates the quality of each answer — scoring relevance, completeness and correctness — and you monitor trends over time.

⚖️LLM-as-judge in production

from langchain_openai import ChatOpenAI
from pydantic import BaseModel
import random

# Evaluation scheme
class AvaliacaoQualidade(BaseModel):
    relevancia: int        #1-5: Answer meets the objective?
    completude: int        #1-5: Does it cover all aspects?
    corretude: int         #1-5: Correct information?
    tom: int               #1-5: Appropriate tone for the context?
    score_geral: float     # weighted average
    justificativa: str
    pontos_melhoria: list[str]

# Judge LLM (separate, cheaper model)
judge_llm = ChatOpenAI(model="gpt-4o-mini").with_structured_output(
    AvaliacaoQualidade
)

RUBRICA_AVALIACAO = """
Você é um avaliador de qualidade de assistentes de IA.
Avalie a resposta do assistente com a seguinte rubrica (1-5 cada):

- Relevância (1-5): A resposta atende ao objetivo específico da pergunta?
  1 = completamente irrelevante
  5 = perfeitamente alinhada ao objetivo

- Completude (1-5): Todos os aspectos importantes foram cobertos?
  1 = resposta superficial e incompleta
  5 = todos os aspectos abordados com profundidade

- Corretude (1-5): As informações fornecidas são corretas e verificáveis?
  1 = informações incorretas
  5 = todas as afirmações são corretas e verificáveis

- Tom (1-5): O tom é apropriado para o contexto e perfil do usuário?
  1 = tom completamente inadequado
  5 = tom perfeito para o contexto

Contexto da avaliação:
- Pergunta do usuário: {pergunta}
- Resposta do agente: {resposta}
- Contexto (se disponível): {contexto}
"""

async def avaliar_qualidade(
    pergunta: str,
    resposta: str,
    contexto: str = ""
) -> AvaliacaoQualidade:
    """Avalia a qualidade de uma resposta usando LLM-as-judge."""
    prompt = RUBRICA_AVALIACAO.format(
        pergunta=pergunta,
        resposta=resposta,
        contexto=contexto
    )
    avaliacao = await judge_llm.ainvoke(prompt)
    # Calculate weighted score
    avaliacao.score_geral = (
        avaliacao.relevancia * 0.35 +
        avaliacao.completude * 0.25 +
        avaliacao.corretude * 0.30 +
        avaliacao.tom * 0.10
    )
    return avaliacao

# Continuous evaluation pipeline
async def avaliar_amostra_producao(taxa_amostragem: float = 0.1):
    """
    Avalia 10% das execuções de produção automaticamente.
    Roda como background job a cada hora.
    """
    runs_recentes = await buscar_runs_sem_avaliacao(limite=100)

    for run in runs_recentes:
        # Random sampling for cost control
        if random.random() > taxa_amostragem:
            continue

        avaliacao = await avaliar_qualidade(
            pergunta=run["input"],
            resposta=run["output"]
        )

        # Register with LangSmith for trending
        client.create_feedback(
            run_id=run["id"],
            key="llm_judge_score",
            score=avaliacao.score_geral / 5.0  # normalize 0-1
        )

        # Alert if quality below threshold
        if avaliacao.score_geral < 3.0:
            await notificar_qualidade_baixa(run, avaliacao)
            logger.warning(
                f"Qualidade baixa detectada: run_id={run['id']}, "
                f"score={avaliacao.score_geral:.2f}, "
                f"melhorias={avaliacao.pontos_melhoria}"
            )

📈Using quality trending

  • •After each prompt change → compare average score before and after
  • •Score falling for 3 consecutive days → automatic alert for review
  • •Segment by type of question → identify where the agent has the most difficulty
  • •Average score below 3.5/5 per week → trigger for quality sprint
