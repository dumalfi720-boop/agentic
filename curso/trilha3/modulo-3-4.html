🏗️ SDK architecture: Agent, Runner and Context

The OpenAI Agents SDK defines three primitives:agent(execution unit with instructions, tools and handoffs),Runner(executor that manages the call loop) andContext(state shared between agents of the same execution).

⚙️Basic structure of the SDK

from agents import Agent, Runner, function_tool
from pydantic import BaseModel

# Context: shared state between agents
class SupportContext(BaseModel):
    user_id: str
    account_tier: str
    conversation_history: list[str] = []
    issue_type: str = ""

# Tools as decorated functions
@function_tool
def buscar_pedido(order_id: str) -> str:
    """Busca informações de um pedido pelo ID."""
    # Real logic here
    return f"Pedido {order_id}: Em trânsito, entrega prevista 2 dias"

@function_tool
def processar_reembolso(order_id: str, motivo: str) -> str:
    """Processa solicitação de reembolso."""
    return f"Reembolso para {order_id} aprovado. Prazo: 5 dias úteis"

# Agent with instructions, tools and typed context
agente_billing = Agent[SupportContext](
    name="Billing Specialist",
    instructions="""Você é especialista em faturamento e pedidos.
    Sempre verifique o histórico antes de tomar ações.
    Para reembolsos acima de R$500, escale para o supervisor.""",
    tools=[buscar_pedido, processar_reembolso],
    model="gpt-4o"
)

# Runner runs the agent
async def main():
    context = SupportContext(user_id="usr_123", account_tier="premium")
    resultado = await Runner.run(
        agente_billing,
        input="Quero reembolso do pedido #4521",
        context=context
    )
    print(resultado.final_output)

💡Typed context is key

The SDK Context is typed with Pydantic — each tool and each agent accesses the same shared object, with automatic validation. This eliminates "I passed the wrong data between agents" bugs that are common in untyped systems.

🤝 Handoffs: intelligent passing of control

Handoff is when an agent decides that another agent is better suited to continue — it hands over control with all context intact. It's like a smart telephone transfer: the specialist receives the case without needing a new explanation.

🔀Support system with handoffs

from agents import Agent, handoff

# Specialist agents
agente_tecnico = Agent[SupportContext](
    name="Technical Support",
    instructions="""Especialista em problemas técnicos: bugs, configuração,
    integrações. Resolva issues técnicas. Para problemas de cobrança,
    faça handoff para Billing.""",
    tools=[verificar_logs, resetar_configuracao]
)

agente_billing = Agent[SupportContext](
    name="Billing Support",
    instructions="""Especialista em faturamento, reembolsos e upgrades.
    Para problemas técnicos, faça handoff para Technical.""",
    tools=[buscar_pedido, processar_reembolso]
)

agente_supervisor = Agent[SupportContext](
    name="Supervisor",
    instructions="""Lida com casos escalados e clientes insatisfeitos.
    Tem autoridade para oferecer compensações especiais.""",
    tools=[aplicar_credito, criar_ticket_vip]
)

# Triage agent with all handoffs configured
agente_triage = Agent[SupportContext](
    name="Triage Agent",
    instructions="""Você é a entrada do suporte. Classifique o problema
    e faça handoff para o especialista correto:
    - Técnico: bugs, erros, configuração
    - Billing: pagamento, reembolso, plano
    - Supervisor: cliente muito insatisfeito, casos complexos""",
    handoffs=[
        agente_tecnico,
        agente_billing,
        agente_supervisor
    ]
)

# Execution: triage decides who to transfer to
resultado = await Runner.run(
    agente_triage,
    input="Meu pagamento foi cobrado duas vezes este mês",
    context=SupportContext(user_id="u456", account_tier="basic")
)
# Triage analyzes → automatic handoff to Billing
# Billing resolves with full context access

📋How handoffs work internally

  • 1The originating agent decides to handoff — this is a special tool call in the SDK
  • 2Runner transfers the execution to the new agent, passing the complete history
  • 3The new agent has access to the full context — without re-explanation by the user
  • 4The flow continues until the agent finishes or makes another handoff

🛡️ Guardrails: security without overhead

Guardrails are validations that runin parallelwith the agent — without adding latency. They check whether the user input is safe and whether the agent's output meets quality criteria before being returned.

🛡️Input and Output guardrails

from agents import (
    Agent, InputGuardrail, OutputGuardrail,
    GuardrailFunctionOutput, RunContextWrapper
)
from pydantic import BaseModel

# Schema for guardrail decision
class TopicCheck(BaseModel):
    is_on_topic: bool
    reason: str

# Classifying agent (used by guardrail)
topic_classifier = Agent(
    name="Topic Classifier",
    instructions="Verifique se a mensagem é sobre suporte ao produto.",
    output_type=TopicCheck,
    model="gpt-4o-mini"  # Inexpensive template for quick screening
)

# Input guardrail: checks if the input is valid
@input_guardrail
async def verificar_topico(
    ctx: RunContextWrapper, agent: Agent, input: str
) -> GuardrailFunctionOutput:
    resultado = await Runner.run(topic_classifier, input, context=ctx.context)
    check = resultado.final_output
    return GuardrailFunctionOutput(
        output_info=check,
        tripwire_triggered=not check.is_on_topic  # True = block
    )

# Output guardrail: checks response quality
@output_guardrail
async def verificar_qualidade(
    ctx: RunContextWrapper, agent: Agent, output: str
) -> GuardrailFunctionOutput:
    # Checks whether the response contains sensitive information
    has_pii = any(term in output.lower() for term in ["cpf", "senha", "cartão"])
    return GuardrailFunctionOutput(
        output_info={"has_pii": has_pii},
        tripwire_triggered=has_pii
    )

# Agent with guardrails configured
agente_seguro = Agent[SupportContext](
    name="Secure Support Agent",
    instructions="Ajude clientes com problemas do produto.",
    input_guardrails=[verificar_topico],
    output_guardrails=[verificar_qualidade],
    tools=[buscar_pedido]
)

# If guardrail fires: InputGuardrailTripwireTriggered exception
try:
    resultado = await Runner.run(agente_seguro, "Quero saber o CPF do usuário X")
except Exception as e:
    print(f"Bloqueado pelo guardrail: {e}")

🔍 Integrated tracing: debugging without configuration

The OpenAI SDK automatically records each execution as a complete trace — which tools were called, which handoffs occurred, how much it cost, how long it took. Without a line of observability code.

🔍Automatic and custom tracing

from agents import Runner, trace, custom_span
import asyncio

# Automatic tracing — no configuration!
resultado = await Runner.run(agente_triage, "Preciso de ajuda")
# Complete trace automatically available on the OpenAI portal

# Grouping multiple executions into a single trace
with trace("pipeline_suporte_completo"):
    triagem = await Runner.run(agente_triage, "Meu acesso está bloqueado")
    follow_up = await Runner.run(
        agente_tecnico,
        "Desbloqueie o acesso do usuário",
        context=triagem.context
    )

# Custom spans for specific operations
async def processar_com_trace(user_input: str):
    with custom_span("validacao_input"):
        input_limpo = sanitizar(user_input)

    with custom_span("execucao_agente"):
        resultado = await Runner.run(agente_billing, input_limpo)

    with custom_span("persistencia"):
        salvar_historico(resultado)

    return resultado

# Accessing trace data programmatically
resultado = await Runner.run(agente_billing, "Cancele minha assinatura")
print(resultado.raw_responses)    # all API responses
print(resultado.new_messages())   # messages generated in this run
print(resultado.last_agent.name)  # last agent to run

🐝 Swarm: swarm of specialists

The patternSwarmit is a set of specialist agents coexisting, with a screening agent as the gateway. Each specialist receives only the cases that belong to their domain — via automatic handoff.

🐝Full Support Swarm

# 5 experts + 1 triage = Swarm complete
especialista_onboarding = Agent(
    name="Onboarding Specialist",
    instructions="Guia novos usuários. Ajuda com setup inicial e primeiros passos.",
    tools=[criar_tour_interativo, enviar_email_boas_vindas]
)

especialista_seguranca = Agent(
    name="Security Specialist",
    instructions="Lida com contas comprometidas, 2FA e problemas de acesso.",
    tools=[bloquear_conta, resetar_2fa, gerar_link_recuperacao]
)

especialista_integracao = Agent(
    name="Integration Specialist",
    instructions="Suporte para APIs, webhooks e integrações de terceiros.",
    tools=[verificar_webhook, testar_api, consultar_docs_api]
)

especialista_billing = Agent(
    name="Billing Specialist",
    instructions="Faturamento, upgrades, cancelamentos e reembolsos.",
    tools=[buscar_pedido, processar_reembolso, alterar_plano]
)

especialista_tecnico = Agent(
    name="Technical Specialist",
    instructions="Bugs, performance e questões técnicas complexas.",
    tools=[verificar_logs, criar_ticket, escalar_engenharia]
)

# Triage knows all the experts
triage = Agent(
    name="Triage",
    instructions="""Classifique e transfira para o especialista correto.
    Seja eficiente: uma pergunta de classificação no máximo.""",
    handoffs=[
        especialista_onboarding,
        especialista_seguranca,
        especialista_integracao,
        especialista_billing,
        especialista_tecnico
    ]
)

# The Swarm in action
resultado = await Runner.run(
    triage,
    "Meu webhook não está sendo chamado",
    context=SupportContext(user_id="u789", account_tier="pro")
)
# Triage → handoff to Integration Specialist → resolve

🎯 SDK vs. SDK Frameworks: when to use each one

The OpenAI SDK is optimized for simplicity with OpenAI. For more complex cases — fine state control, multi-provider, complex cycles — LangGraph is more suitable. There is no universal right choice: there is a right choice for the context.

Use OpenAI SDK when...

  • ✓New project, OpenAI stack, without vendor lock-in concern
  • ✓Direct handoffs between experts (support, triage)
  • ✓Safety guardrails without additional infrastructure
  • ✓Small team, delivery speed is a priority
  • ✓Enough native tracing for your needs

Use LangGraph when...

  • ✓Fine state control across multiple steps
  • ✓Multi-provider (Anthropic + OpenAI + Gemini)
  • ✓Complex retry and reflection loops
  • ✓Human-in-the-loop with granular checkpointing
  • ✓Critical system in production with full audit

💡The smart migration strategy

Start with the SDK for rapid prototyping. If the system grows in complexity — more state control, multi-provider, complex loops — migrate to LangGraph. The tools (MCP servers) stay the same, only the orchestrator changes.
