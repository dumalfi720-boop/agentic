🕸️ Deep LangGraph

Master the StateGraph, nodes, edges, controlled cycles and checkpoints. The framework that gives full control over your agent's flow.

What is it:

A state graph is a representation where each node is a Python function that takes a shared state and returns an updated version of that state. The execution flow follows the edges that connect the nodes.

Why learn:

Without understanding the mental model of the graph, you cannot design complex flows with branches and loops. It's the foundation of everything in LangGraph.

Key concepts:

State as TypedDict · Nodes as pure functions · Edges as transitions · START and END as entry/exit points · State immutability

What is it:

Nodes are Python functions that process state. Edges are connections between nodes: fixed (always follow the same path) or conditional (choose the next node based on the current state).

Why learn:

The combination of fixed and conditional edges is what allows you to create flows with decisions, branches and returns — impossible to do with simple linear chains.

Key concepts:

add_node() · fixed add_edge() · add_conditional_edges() with router function · Return string to choose next node · Multiple possible destinations

What is it:

Practical construction of a StateGraph: defining the state's TypedDict, creating nodes as functions, connecting with edges and compiling the graph for execution with graph.compile().

Why learn:

Theory without practice doesn't work. You need to know the exact syntax, construction order, and how to debug common graph compilation errors.

Key concepts:

TypedDict as state schema · StateGraph(State) · graph.set_entry_point() · graph.compile() · graph.invoke() vs graph.stream()

What is it:

Unlike linear chains, LangGraph allows the graph to return to a previous node based on a condition — creating retry, reflection, or iteration loops until a criterion is met.

Why learn:

Controlled loops are essential for agents who need to retry after an error, review their own output, or wait for human approval before continuing.

Key concepts:

Conditional edge pointing to previous node · State iteration counter · Loop exit criteria · Infinite loop prevention · Human-in-the-loop with interrupt()

What is it:

The checkpoint system saves the state of the graph after each node. With MemorySaver (in memory) or PostgresSaver (persistent), the graph can be resumed from the exact point where it left off.

Why learn:

Without checkpoints, a failure in the middle of a long stream restarts everything from scratch. With checkpoints, you have resiliency, auditing, and the ability to pause/resume workflows.

Key concepts:

MemorySaver for development · PostgresSaver for production · thread_id as session identifier · graph.get_state() for inspect · Full execution history

What is it:

graph.stream() emits events with each node that runs, allowing the UI to show real-time progress — which node is running, what the current state is, what tokens the LLM generated.

Why learn:

Slow agents without feedback cause user abandonment. Streaming transforms the experience: the user sees the system “thinking” instead of waiting for a blank screen.

Key concepts:

graph.stream() with for loop · stream_mode="values" vs "updates" vs "debug" · LLM token streaming · FastAPI SSE integration · Filtering events by node

👥 CrewAI in Practice

Assemble teams of agents with defined roles, tasks and flows. The framework that thinks about teams, not graphs.

What is it:

A Crew is a team of AI agents with a collective mission. Each agent has a defined role, and they collaborate to complete tasks that they would not be able to perform alone.

Why learn:

Thinking about teams of agents instead of isolated agents solves complex problems more naturally — just like how human teams organize themselves.

Key concepts:

Crew as a team abstraction · Agents as members · Tasks as assignments · Process as a way of working · Shared collective mission

What is it:

Each Agent in CrewAI is defined by three attributes: role (its title/function), goal (what it wants to achieve) and backstory (its life context and specialty). These persona prompts guide LLM behavior.

Why learn:

Well-defined personas generate more consistent and specialized outputs. A "Senior Data Analyst" thinks differently than a "Content Writer" — even using the same LLM underneath.

Key concepts:

Agent(role=..., goal=..., backstory=...) · tools=[...] to equip with tools · llm= to define model · verbose=True for debug · allow_delegation= to allow delegation

What is it:

Task is a unit of work with description (what to do), expected_output (what the expected result is) and agent (who executes it). Flows define how tasks are chained and share context.

Why learn:

Well-defined tasks with clear expected_output are the secret to consistent outputs. Without this, the agent "improvises" the response format on each run.

Key concepts:

Task(description=..., expected_output=..., agent=...) · context=[task_previous] to pass result · output_pydantic= to structure output · async_execution= for parallelism

What is it:

In Process.hierarchical, a Manager Agent coordinates the others — decides who executes each task, reviews outputs and can redirect work. It is an agent that manages other agents.

Why learn:

For complex workflows with dynamic decisions about who does what, hierarchy eliminates the rigidity of fixed sequence and allows for runtime adaptation.

Key concepts:

manager_agent= in Crew · Process.hierarchical · Automatic delegation of tasks · Manager as LLM-powered coordinator · Approval of outputs before continuing

What is it:

Process.sequential executes tasks one after the other (the output of one becomes the input of the next). With async_execution=True on independent tasks, they run in parallel — reducing overall latency.

Why learn:

Tasks that don't depend on each other waste time if they are sequential. Parallelism can reduce latency from minutes to seconds in analysis workflows.

Key concepts:

Process.sequential (default) · async_execution=True per task · Implicit dependencies via context= · Parallel tasks without dependency · Aggregation of results at the end

What is it:

Complete project with 3 agents: Researcher (collects market data with web tools), Analyst (interprets trends and metrics) and Writer (transforms analysis into executive report).

Why learn:

Seeing the complete pattern working is the best way to understand how the components fit together — from the Agent to the final output structured in Pydantic.

Key concepts:

Three agents with different tools · Three tasks in sequence · Context passed automatically · Final output in Pydantic · crew.kickoff() as entry point

💬AutoGen/AG2

Agents that talk to each other, criticize each other and execute code. The framework for research, code generation and structured debates.

What is it:

AutoGen models interaction between agents as a multi-turn conversation. Instead of a state graph, the flow emerges from the exchange of messages — each agent responds to what the other said.

Why learn:

Problems that benefit from multiple perspectives (code review, research, debate) fit naturally into AutoGen's conversational model.

Key concepts:

Conversation as control flow · Messages as state · Agents as participants · Termination criteria · Conversation history as accumulated context

What is it:

AssistantAgent uses an LLM to respond and generate code. UserProxyAgent can execute code locally and simulate the human user. Together, they form the fundamental pair of AutoGen.

Why learn:

This pair allows for an automated code generation and execution loop — the agent writes Python, the proxy executes it, the error becomes feedback, the agent corrects it. All without human intervention.

Key concepts:

AssistantAgent(name=..., llm_config=...) · UserProxyAgent(code_execution_config=...) · human_input_mode="NEVER" for full autonomy · is_termination_msg for stopping criteria

What is it:

GroupChat is an abstraction where N agents participate in a conversation mediated by a GroupChatManager, which decides who speaks next based on rules or another LLM.

Why learn:

When you need more than two agents collaborating (e.g. researcher + analyst + critic), GroupChat is AutoGen's native mechanism to coordinate this type of interaction.

Key concepts:

GroupChat(agents=[...], messages=[], max_round=...) · GroupChatManager as moderator · speaker_selection_method="auto" vs "round_robin" · History shared between agents

What is it:

Pattern where one agent generates a response and a second agent (the critic) evaluates, points out problems and suggests improvements. The first one reviews and the cycle continues until sufficient quality.

Why learn:

Reflection dramatically improves output quality without needing a human in the feedback loop. It's the equivalent of having an automatic reviewer for each answer.

Key concepts:

Writer Agent + Critic Agent · Clear approval criteria · Defined maximum iterations · Structured critique prompt · Final output approved by the critic

What is it:

AutoGen allows the UserProxyAgent to run Python code generated by the AssistantAgent in a controlled environment — whether locally, in Docker, or in a remote sandbox.

Why learn:

Automated code execution is AutoGen's superpower. It allows data analysis, graph generation, scraping, calculations and any task that Python code solves — without a human.

Key concepts:

code_execution_config={"work_dir": "..."} · Docker for isolation · last_n_messages for limiting context · Execution timeout · Capturing stdout and stderr as feedback

What is it:

AutoGen shines at: exploratory research, code generation and debugging, debates between perspectives, and workflows where the flow emerging from the conversation is more natural than pre-defined graphs.

Why learn:

Choosing the wrong framework for the right problem is one of the most costly mistakes. Understanding where AutoGen outperforms LangGraph saves weeks of refactoring.

Key concepts:

AutoGen ideal for: code gen + iterative execution · multi-perspective search · emergent workflows · LangGraph ideal for: precise flow control · complex state · critical production

🤖 OpenAI Agents SDK

Native handoffs, guardrails and tracing. The official OpenAI SDK for simple, reliable multi-agent systems.

What is it:

The OpenAI SDK defines Agent as a unit with instructions (prompt system), tools (available tools) and handoffs (agents to whom it can pass control). Runner is the enforcer.

Why learn:

Understanding the SDK architecture allows you to use it without fighting it. Its opinionated design favors certain patterns — knowing them speeds up development.

Key concepts:

Agent(name=..., instructions=..., tools=..., handoffs=...) · Runner.run() as main executor · Context object as shared state · Result as RunResult with history

What is it:

Handoff is when an agent decides that another agent is better suited to continue — he "passes the baton" with all the context. It's like a smart phone transfer between experts.

Why learn:

Handoffs enable specialization: a triage agent routes to the right specialist without the user having to re-explain the problem — context is automatically preserved.

Key concepts:

handoffs=[specialist_agent] in definition · Automatic context transfer · Triage agent as main input · Default: triage → billing / technical / general · Handoff as special tool call

What is it:

Guardrails are validations that run in parallel with the agent — checking whether the user input is safe and whether the agent's output meets quality criteria before being returned.

Why learn:

Native guardrails eliminate the need for custom validation wrappers. They are executed in parallel, without adding latency to the main flow.

Key concepts:

InputGuardrail and OutputGuardrail · @input_guardrail decorator · GuardrailFunctionOutput with tripwire · Parallel execution without overhead · Integration with OpenAI content policy

What is it:

The SDK automatically records each execution — which tools were called, which handoffs occurred, which guardrails triggered. Without additional configuration, you have a complete trace.

Why learn:

Debugging multi-agent systems without tracing is guesswork. With native tracing, you know exactly where a workflow failed and what decisions the agent made.

Key concepts:

Automatic trace by default · Export to OpenTelemetry · Integration with LangSmith and Arize · span by tool call · RunResult.trace for programmatic access

What is it:

Swarm is a pattern where multiple expert agents coexist and activate each other via handoff — a triage agent distributes work to the most appropriate expert at each time.

Why learn:

For customer service, technical support or corporate assistant systems, the Swarm pattern is more scalable than a monolithic agent trying to do everything.

Key concepts:

Triage agent as gateway · N expert agents · Bidirectional handoffs · Shared context via Context · Horizontal scalability by adding agents

What is it:

OpenAI SDK is ideal for simple OpenAI projects and direct handoffs. LangGraph for fine state control and complex cycles. The choice depends on: complexity, vendor lock-in and control requirements.

Why learn:

Starting with the SDK and migrating to LangGraph when necessary is a valid path. Understanding the trade-offs prevents premature over-engineering.

Key concepts:

SDK: simplicity, OpenAI-first, native handoffs · LangGraph: multi-provider, complex cycles, detailed state · CrewAI: team abstraction, less code · AutoGen: conversational, code gen

🔌 MCP Advanced

Create complete MCP servers, add authentication, deploy to production and explore the ecosystem of ready-made servers.

What is it:

MCP (Model Context Protocol) is a client-server protocol where the agent (client) discovers and calls tools exposed by external servers via JSON-RPC over stdio or HTTP.

Why learn:

MCP decouples tools from frameworks — an MCP server works with LangGraph, CrewAI, and any compatible client. It is the ecosystem's interoperability standard.

Key concepts:

JSON-RPC as base protocol · tools/list for discovery · tools/call for execution · resources/ for data · prompts/ for templates · Transports: stdio vs SSE

What is it:

FastMCP is a Python library that simplifies creating MCP servers — you decorate functions with @mcp.tool() and the server automatically exposes tool/list and tool/call.

Why learn:

Creating your own MCP server allows you to expose any company's internal API to agents — without having to change the code of the frameworks that use them.

Key concepts:

FastMCP() as instance · @mcp.tool() decorator · Type hints generate JSON schema automatically · @mcp.resource() for data · mcp.run() to start · Pydantic for validation

What is it:

MCP servers over HTTP need to authenticate clients. Bearer token in the Authorization header is the simplest pattern. OAuth 2.0 for enterprise scenarios with multiple users.

Why learn:

An unauthenticated MCP server exposes its internal tools to any client. In production, any externally accessible server needs authentication.

Key concepts:

Bearer token via Authorization header · Validation middleware · Rate limiting per client · IP allowlist for internal servers · Call auditing · Mandatory HTTPS in production

What is it:

The MCP client calls tools/list on startup to discover all available tools — name, description, and parameter schema. Then call tools/call to execute.

Why learn:

Dynamic discovery allows the agent to adapt to new servers without reconfiguration. You can add tools on the server and the agent uses them automatically.

Key concepts:

tools/list returns list of ToolDefinition · JSON Schema for parameters · Clear descriptions guide LLM · Tools versioning · List caching for performance

What is it:

Deploy an MCP server as an HTTP service on cloud platforms: Railway (zero config), Fly.io (more control) or Docker container on any cloud. The server is accessible via URL.

Why learn:

A local MCP server only works in development. For agents in production to serve real users, the server needs to be available 24/7 with high availability.

Key concepts:

Dockerfile for MCP server · Railway deploy in 5 min · Environment variables for secrets · Health check endpoint · Structured logging · Automatic restart on failures

What is it:

There's a growing ecosystem of ready-made MCP servers: filesystem, GitHub, PostgreSQL, Brave Search, Slack, Notion, and dozens more — install and connect in minutes.

Why learn:

Before building, check if it already exists. The MCP ecosystem is growing quickly — many common integrations are already available as ready-to-use servers.

Key concepts:

mcp.so as marketplace · Official Anthropic servers · Community servers · Installation via npx or pip · Configuration in claude_desktop_config.json · Composition of multiple servers

⚖️ Framework Comparison

Complete Decision Table: LangGraph vs. CrewAI vs. AutoGen vs. OpenAI SDK. Definitive guide to choosing the right framework.

What is it:

The 6 criteria that define the choice: flow control (precision vs. abstraction), learning curve, support for cycles/loops, native observability, community size and enterprise support.

Why learn:

Without clear criteria, you choose frameworks based on hype or familiarity. Objective criteria ensure that the choice serves the problem, not personal taste.

Key concepts:

Flow control · Learning curve · Loop support · Observability · Community and maturity · Enterprise support · Vendor lock-in

What is it:

LangGraph offers precise flow control via a directed graph — ideal for complex workflows with branches and states. CrewAI offers team abstraction — ideal for collaboration between specialized agents.

Why learn:

They are the most compared pair. Understanding the fundamental distinction (flow vs. team) avoids using the right tool in the wrong context.

Key concepts:

LangGraph: high curve, full control, native loops · CrewAI: low curve, rich abstraction, less flow control · LangGraph shines in: critical production · CrewAI shines in: rapid prototyping

What is it:

CrewAI organizes agents into teams with fixed roles and defined tasks. AutoGen organizes agents into dynamic conversations. The difference is structure vs. emergency.

Why learn:

Both use multiple agents, but in radically different ways. Confusing the two leads to strange code that fights against the framework design.

Key concepts:

CrewAI: defined tasks, clear expected output, structured process · AutoGen: emergent conversation, more flexible, better for code gen · Choose for: predictability vs. flexibility

What is it:

AutoGen uses conversation as a control mechanism — agents communicate via messages. OpenAI SDK uses handoffs — one agent explicitly transfers control to another. They are opposing philosophies of coordination.

Why learn:

For customer service and support, SDK handoffs are more predictable. For research and code gen, AutoGen talk is more powerful. Context defines the choice.

Key concepts:

AutoGen: multi-provider, exec code, search · SDK: OpenAI-first, direct handoffs, native guardrails · AutoGen: more setup · SDK: opinionated but simpler for simple cases

What is it:

Table with practical scenarios and the recommended framework: complex flow with branches (LangGraph), team of agents with roles (CrewAI), search and code gen (AutoGen), simple OpenAI project (SDK).

Why learn:

Having a quick reference table eliminates choice paralysis. Instead of endlessly comparing, you consult the guide and decide in minutes.

Key concepts:

By usage scenario · By technical requirement · By team experience level · By infrastructure budget · By observability need · By available development time

What is it:

Strategies for migrating from one framework to another without rewriting everything: identify the logical core of the agent, isolate tools such as MCP servers, migrate incrementally by module.

Why learn:

It's common to start with CrewAI and migrate to LangGraph as the system grows. Knowing how to migrate without big bang rewrites saves weeks of project risk.

Key concepts:

Isolate tools via MCP · Extract business logic · Migrate node to node · Feature flags for gradual rollout · A/B tests between versions · Maintain backwards compatibility during transition

🔗 LangGraph + CrewAI together

The best of both worlds: LangGraph as a flow orchestrator and CrewAI as a specialized team execution node.

What is it:

LangGraph excels at precise flow control (when to do what) but verbose for multi-agent collaboration. CrewAI excels at team abstraction but weak at flow control. Combined, they cover both.

Why learn:

Real systems in production often need precise flow in some steps and teamwork in others. Knowing how to combine is the mature agentic engineering skill.

Key concepts:

LangGraph as macro orchestrator · CrewAI as specialized node · Clear separation of responsibilities · State flowing between the two · Checkpoint before and after Crew

What is it:

LangGraph defines the macro structure of the system — which steps exist, in what order, with what exit conditions. CrewAI runs inside specific nodes without LangGraph knowing the internal details of the Crew.

Why learn:

Separating macro orchestration (LangGraph) from specialized execution (CrewAI) creates more maintainable systems — you can replace the Crew without changing the graph, and vice versa.

Key concepts:

LangGraph node instantiates and executes a Crew · Crew receives input from the state · Crew returns result that goes back to the state · LangGraph decides next step based on the result · Clear interface between the two

What is it:

A LangGraph node that, when executed, instantiates a complete Crew, calls crew.kickoff() with the state inputs, and returns the result back to the graph state.

Why learn:

It is the concrete implementation pattern. Knowing exactly how to encapsulate Crew in a node function is the core technical skill of this integration.

Key concepts:

def crew_node(state: State): crew = create_crew(state) · result = crew.kickoff() · return {"crew_result": result} · Crew created fresh at each execution or reused · Timeout on the node to limit execution

What is it:

The LangGraph (TypedDict) state is converted to the Crew input format, and the Crew output is serialized back to the state. The border is the node that makes this translation.

Why learn:

The interface between frameworks is where bugs happen. Clearly defining the input/output contract between LangGraph and CrewAI prevents type issues and missing data.

Key concepts:

Pydantic to validate the interface contract · Explicit serialization of complex types · Logs at the boundary for debugging · Integration node unit testing · Partial vs. partial state complete status in Crew

What is it:

Three proven patterns: (1) Single Crew Node, (2) Multiple Crews on Parallel Nodes, (3) Crew that uses LangGraph sub-graph internally. Each pattern solves a different type of problem.

Why learn:

Named patterns facilitate team communication and avoid reinventing solutions. A "Crew-on-Node Pattern" is self-explanatory to any engineer who knows the stuff.

Key concepts:

Pattern 1: Sequential single node · Pattern 2: Parallel Crews in Send() API · Pattern 3: Crew with internal graph · Choice by complexity and parallelism · Documentation of the pattern in the code

What is it:

Due diligence system where LangGraph orchestrates the steps (data collection → analysis → validation → report) and a CrewAI executes the analysis step with a specialized team (financial, legal, market).

Why learn:

Seeing a complete end-to-end project with both frameworks consolidates all previous concepts into a coherent system — from the initial TypedDict to the final report.

Key concepts:

4 nodes in LangGraph (collection, analysis via Crew, validation, reporting) · Crew of 3 analysts · Checkpoint before and after · Progress streaming · Final output structured in Pydantic

📊 Observability

LangSmith, Arize Phoenix, and metrics that matter. Monitor agents in production with alerts, dashboards and continuous evaluation.

What is it:

Observability for agents goes beyond logging — it involves traces (complete execution flow), spans (duration of each step), metrics (success rates, cost, latency) and response quality assessment.

Why learn:

Without observability, you are production blind. It doesn't know which node is failing, how much it costs per execution, or whether the quality of the output has degraded after a prompt change.

Key concepts:

Traces vs. Logs vs. Metrics · Span as a unit of execution · LLM as a judge for quality · Cost tracking per run · Latency P50/P95/P99 · Automatic alerts

What is it:

LangSmith is LangChain's observability platform — automatically captures traces of LangGraph executions, shows inputs/outputs for each node, token costs and allows you to filter runs with errors.

Why learn:

LangSmith integrates with LangGraph in 3 configuration lines. It is the most used tool for debugging and monitoring LangChain/LangGraph-based systems.

Key concepts:

LANGCHAIN_TRACING_V2=true · LANGCHAIN_API_KEY= · Automatic tracing without additional code · Filters by status, model, data · A/B run comparison · Datasets for regression testing

What is it:

Arize Phoenix is ​​an open-source observability platform for LLMs — it runs locally, captures traces via OpenTelemetry, visualizes embeddings and allows you to evaluate response quality.

Why learn:

For those who cannot use SaaS due to compliance or cost, Phoenix offers similar capabilities to self-hosted LangSmith. It is the open-source observability ecosystem standard for LLMs.

Key concepts:

pip install arize-phoenix · px.launch_app() for local UI · OpenInference as trace format · Automatic instrumentation via openinference-instrumentation · Export to Arize cloud optional

What is it:

Technical metrics: latency, tokens, errors. Business metrics: success rate per user objective, average cost per successful execution, agentic system NPS, ROI vs. manual process.

Why learn:

Business stakeholders don't care about P95 latency. They want to know if the system solves the problem and how much it costs. Business metrics translate technical performance into value.

Key concepts:

Objective completion rate · Cost per task completed · Average execution time · Error rate by type · End user satisfaction · ROI vs. alternative automation

What is it:

Configure automatic alerts when metrics exceed thresholds: error rate above 5%, cost per hour above budget, P95 latency above 30s. Dashboards for real-time vision.

Why learn:

Discovering problems from the customer is the worst case scenario. Proactive alerts allow you to detect and correct degradations before they affect users — the difference between an SLA met and an incident.

Key concepts:

Threshold by metric with hysteresis · Notification via Slack/PagerDuty · Dashboard by agent trail · Deployment notes for correlation · Alert response runbooks

What is it:

LLM-as-judge: Use another LLM to automatically evaluate the quality of agent responses in production — scoring relevance, completeness, tone, and correctness on each run.

Why learn:

Manual quality assessment does not scale. With LLM-as-judge, you get automatic feedback on the quality of each response — detecting degradation before it causes real problems.

Key concepts:

LLM Evaluator separate from main agent · Structured evaluation rubric · Score 1-5 per dimension · Random sampling in production · Quality trending over time · Re-training trigger when score drops
