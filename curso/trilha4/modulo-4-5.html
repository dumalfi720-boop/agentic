⚠️ The state problem in multi-agent systems

Shared state in multi-agent systems is like a Google Docs document being edited by 10 people at the same time — no version control, no locks, no coordination. The inevitable result is conflicts, overwrites, and corrupted data that appear hours later in production.

⚠️ Real scenario: 3 agents, 1 shared state

# Three agents running in parallel

Agent A (t=0ms): reads status = "in_progress"

Agent B (t=1ms): reads status = "in_progress"

Agent A (t=50ms): writes status = "done", result = "analysis A"

Agent B (t=51ms): writes status = "done", result = "analysis B"

# Problem: Agent B overwrote A's result

# "Analysis A" was silently lost

# The system believes it has finished successfully

Race Condition

Two agents read the same old state, process it, and write incompatible versions. The result depends on who wrote last.

DirtyRead

Agent B reads data that Agent A is in the middle of writing. The data is in a transient state and is inconsistent.

Lost Update

Agent B overwrites Agent A's update without knowing it existed. Lost processing hours.

🔴 Shared state with Redis

Redis as a blackboard is the de facto standard for shared state in high-performance multi-agent systems. All agents read and write to the same Redis, and Redlock guarantees mutual exclusion for critical operations.

🔴 Redis Patterns for Agents

Hash for structured state

# Each agent has its own namespace

redis.hset("run:123:agentA",

"status", "done",

"result", json.dumps(result)

)

Pub/Sub for notifications

# Agent publishes when finished

redis.publish(

"run:123:events",

'{"agent": "A", "status": "done"}'

)

Redlock for mutual exclusion

# Lock before writing critical state

with redis_lock("run:123:state"):

state = get_state()

state.update(new_data)

set_state(state)

TTL for temporary data

# Status expires automatically

redis.set("run:123:temp",

value, ex=3600 # 1 hour

)

💡 Redlock: correct distributed lock

Redlock acquires lock on 3 Redis instances in parallel. If 2 out of 3 respond successfully, the lock is valid. If the process dies, the lock automatically expires (TTL). Use the libraryredis-pywith Redlock for correct implementation.

🔒 Agent-isolated state

The simplest alternative to shared state: each agent maintains its own isolated state, executes completely independently, and delivers the result to the coordinator upon completion. The coordinator combines all partial states in the fan-in phase.

✓ When to use isolated state

  • ✓Agents do not need to read the results of others during execution
  • ✓Tasks are actually independent of each other
  • ✓You prefer simplicity over extreme performance
  • ✓Small team with no expertise in distributed systems

✗ When NOT to use isolated state

  • ✗Agent B needs Agent A's result before continuing
  • ✗Agents need to collaborate in real time to adjust strategy
  • ✗State is too big to pass via result (memory)

Python — Isolated state pattern with merge

async def run_parallel_with_isolated_state(agents, task):
    # Each agent receives copy of initial state (not reference)
    results = await asyncio.gather(*[
        agent.run(task, initial_state.copy())
        for agent in agents
    ], return_exceptions=True)

    # Coordinator combines partial states
    final_state = merge_results([
        r for r in results
        if not isinstance(r, Exception)
    ])
    return final_state

def merge_results(results: list) -> dict:
    # Merge strategy: combines results without conflict
    merged = {}
    for result in results:
        merged.update(result)  # or more sophisticated logic
    return merged

⚡ State conflicts and resolution strategies

Even with good practices, state conflicts happen. The point isn't to avoid them completely — it's to have an explicit resolution strategy before they happen. Discovering the strategy in production when the user has already lost data is too late.

1

Last-Write-Wins (LWW)

The most recent write (by timestamp) wins. Simple to implement, but may lose valid data. Acceptable when data is cumulative and partial loss is tolerable.

Pros: simple, no blocking Cons: May lose data
2

Three-Way Merge

Compare the two conflicting versions with the common ancestor. Merges non-conflicting changes automatically, flags true conflicts for manual resolution. Same algorithm as Git.

Pros: Preserves changes from both Cons: may still have unresolvable conflict
3

Human-in-the-Loop for critical conflicts

For high-value data (end results, irreversible decisions), conflict → escalation to human. The human chooses which version is correct or how to combine. Slow but sure.

Pros: Zero risk of losing critical data Cons: Human latency, doesn't scale

📚 State versioning with Event Sourcing

Instead of overwriting the current state, add each change to an immutable log. You keep the entire history of the state, can roll back to any point in the past, and have a complete audit trail — free of charge. It is the most used pattern in high reliability systems.

📚 Event Sourcing for Agent Status

# Instead of:

state["status"] = "done"# overwrite, lose history

# Use event sourcing:

events.append({"

"timestamp": now(),

"agent_id": "executor_A",

"event_type": "status_changed",

"date": {"from": "running","to": "done"},

"prev_hash": events[-1]["hash"]# integrity

})

# Current state = playback of all events

current_state = replay_events(events)

Rollback

Only replay events up to the desired timestamp. You return to any past state.

Temporal debugging

"What exactly happened between 2pm and 2:30pm?" The event log responds with millisecond precision.

Automatic audit

The event log is the audit trail. Each change has an agent, timestamp and integrity hash.

💡 Snapshot + Log for performance

Replaying 10,000 events every time you need the current state is slow. The solution:periodic snapshot(serialized state at one point) + event log since the snapshot. Replay only recent events, not all from the beginning.

TOPIC 6
6

🏭 Status in production

The state issues that appear in production are rarely the ones you anticipated in the design. These are cumulative problems: a state that grows indefinitely, backups that never test recovery, and failures that leave the system in an inconsistent state without warning.

Common production problems

State without TTL

Status of old runs never expires. In 3 months, Redis is full and operations are slow.

Backup not tested

Backup exists, but no one has tested restore. At the time of the failure, you discover that the backup has been corrupt for weeks.

Recovery without idempotence

After restarting, tasks are performed again from the beginning — paying twice for the same work.

Production Solutions

TTL strategy

Active run status: 24h. Completed run status: 7 days. Archived status: 90 days → cold storage.

Backup + restore weekly test

Schedule automatic restore in a staging environment every week. If it fails, alert before needing it in production.

Idempotence in tasks

Each task checks whether it has already been executed (by task_id) before running. Recovery = safe replay.

📊 RPO and RTO for agentic systems

Define before going to production:

RPO (Recovery Point Objective)

How much agent work can I lose? If 30 min is acceptable, take snapshots every 30 min.

RTO (Recovery Time Objective)

How soon do I need to be back? If 5 min, the restore needs to be automated and tested.
