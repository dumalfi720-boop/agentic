🤝 What is a handoff

A handoff isn't just "agent A finishes and agent B starts". It is a formal transfer of control and context, with consistency checking before transfer and confirmation of receipt by the destination agent. Formality is what ensures that nothing is lost in the transition.

🤝 The formal handoff protocol

1 Agent A finishes work and checks that the state is consistent
2 Agent A prepares the handoff packet (result + relevant context + metadata)
3 Orchestrator records the handoff in the audit log
4 Agent B receives and acknowledges
5 Agent A releases the resources and ends its execution
! If Agent B does not confirm in X seconds → automatic rollback

The handoff packet

task_id: "T3"

from_agent: "executor_A"

to_agent: "reviewer_B"

result: {result_data}

context_summary: "summary of what was done"

handoff_reason: "task_complete"

timestamp: 1748345678

Informal handoff (the problem)

Without formal protocol, informal handoffs cause:

  • ✗Context lost in transition
  • ✗Target agent starts without confirmation
  • ✗No audit log of the transfer
  • ✗No rollback when target fails

🎛️ Routing criteria

Routing defines which agent receives which task. The wrong decision wastes tokens (wrong agent for domain) or time (overloaded agent). Four criteria cover most real-world cases — use the simplest one that solves your problem.

1

Content Routing

The task type determines the target agent. Legal analysis → legal agent. Code → engineering agent. Writing → content agent.

if "contract" in task: → legal_agent

elif "def " in task: → code_agent

else: → general_agent

2

Load Routing

The agent with the fewest tasks in the queue receives the next one. Classic load balancing applied to agents. Ideal when agents are equivalent to each other.

agent = min(agents,

key=lambda a: a.queue_size)

3

Skill Routing

The agent with the highest historical success_rate for this task type is selected. Self-learning routing — improves with use.

agent = max(capable_agents,

key=lambda a: a.success_rate(

task_type=task.type))

4

Cost Routing

For restricted budget, the cheapest agent compatible with the task requirements is selected. Balances cost and quality.

agent = min(capable_agents,

key=lambda a: a.cost_per_token)

When deterministic rules are not enough to classify the task correctly, a lightweight LLM acts as a classifier: it analyzes the input and decides which agent has the best combination of skills to deal with it.

↩️ Handoff rollback

What happens when the destination agent fails after receiving the handoff? Without a defined rollback, the task remains in limbo — neither on the source agent nor on the destination — and the system silently crashes. Formal rollback is what guarantees that no task is lost.

↩️ Two-phase commit for handoffs

Phase 1: Prepare

  • → Agent A prepares handoff packet and records it in the log as "pending"
  • → Agent B is notified and reserves resources to receive
  • → Agent B confirms “ready” or responds “not ready”

Phase 2: Commit (or Abort)

  • → If B confirmed: handoff executed, log updated to "committed"
  • → If B did not confirm in timeout: handoff aborted, log as "rolled_back"
  • → Rollback: task returns to agent A (or goes to fallback agent)

⚠️ The phantom handoff

Without two-phase commit, there is a risk of "phantom handoff": Agent A delivered (from his point of view), but Agent B never received it (crash before confirming). Neither of them knows that the task is lost. The system continues to work, but without completing that specific task — discovered by the end user hours later.

TOPIC 5

📦 Efficient Context passing

Giving too much context is the most common mistake in handoffs. If Agent A processed 50 documents and passes the entire history to Agent B, you pay for context twice and may exceed the context window. Minimum required context is an engineering skill.

📦 Context compression strategies

1

Semantic summarization

Use an LLM to summarize Agent A's work into 3-5 bullet points before moving on to B. From 10,000 tokens to 200 tokens with 90% of the relevant information preserved.

2

Extraction of relevant entities

Identify and pass only the entities that Agent B will use: names, numbers, URLs, decisions. The rest is irrelevant to the next task.

3

Hierarchical context

Pass two levels: summary (always) + full_context (available via reference if the agent needs it). The agent starts with the summary and requests details on demand.

🗺️ The 4 handoff patterns

Four patterns cover virtually all handoff cases in real systems. Always choose the simplest option that solves your case — simplicity is reliability.

➡️

Sequential (A→B→C)

Each agent completes and passes to the next in a linear sequence. The simplest and most predictable.

+ Easy to audit and debug
- Slower (no parallelism)
🔀

Conditional (if X→B, if Y→C)

The destination agent is chosen based on a condition evaluated by the orchestrator or by agent A itself.

+ Flexible for different types of tasks
- Condition logic can grow indefinitely
⚡

Parallel (A→B and C simultaneously)

Agent A passes the result to multiple agents simultaneously. Everyone processes in parallel and the results are combined in fan-in.

+ Very fast for independent tasks
- Cost multiplied, merge logic required
🧠

Dynamic (A decides at runtime)

Agent A decides at runtime which agent to handoff to, based on the results of its own processing. Maximum flexibility.

+ Adapts to execution result
- Harder to predict and test

💡 Pattern choice rule

Always start with Sequential. Change to Conditional when the target depends on clear criteria. Use Parallel when latency matters more than cost. Reserve Dynamic only for systems that genuinely cannot be deterministic — they are the most difficult to operate in production.
