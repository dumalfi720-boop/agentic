======== MODULE 4.1 ========
4.1 ~45 min

🏗️ Multi-Agent Architectures

Hierarchical, peer-to-peer, pipelined, parallel — each pattern has its use case. Learn how to choose the right architecture for each problem.

What is it:

The technical justification for dividing a complex problem among multiple specialized agents rather than one generalist agent.

Why learn:

Understanding when multi-agent pays off (versus when it unnecessarily complicates) is the most important architectural decision you will make.

Key concepts:

Domain specialization · Parallel execution of independent tasks · Resilience through redundancy · Separation of responsibilities

What is it:

A central orchestrator receives the objective, plans, and delegates subtasks to specialized sub-agents. All communication goes through the orchestrator.

Why learn:

It's the most common and easiest to audit pattern — clear governance, centralized logs, cost control in one place.

Key concepts:

Orchestrator as single source of truth · Bottleneck in the orchestrator · Single point of failure · Controlled fan-out

What is it:

Agents communicate directly with each other via a standardized protocol, without a mandatory central orchestrator. Each agent can initiate communication with another.

Why learn:

Eliminates the hierarchical bottleneck and increases resilience — if one agent goes down, the others continue operating. Basis of the A2A protocol.

Key concepts:

Agent discovery · Standardized communication protocol · Distributed coordination complexity · Difficult to audit

What is it:

Agent A processes → delivers to Agent B → which processes → delivers to Agent C. Linear flow where each step transforms the previous data.

Why learn:

Ideal for predictable sequential transformations. Easy to debug (you know exactly where the failure occurred). Adopted by agentic ETL and document processing.

Key concepts:

Sequential transformation · Immutable data between steps · Cascading failure · Checkpoint between steps · Ideal for document processing

What is it:

Multiple agents run simultaneously on different parts of the problem. The results are combined (fan-in) by the coordinator at the end.

Why learn:

Dramatically reduces latency. Analyze 10 documents in 1 min (parallel) vs. 10 min (sequential). Critical for high performance systems.

Key concepts:

Fan-out pattern · Fan-in and result combination · asyncio.gather() · 10x higher cost · Partial result timeout

What is it:

Decision framework to choose between hierarchical, P2P, pipeline and parallel based on measurable criteria for your problem.

Why learn:

Choosing the wrong one is costly. Hierarchical where it should be parallel = unnecessary latency. Parallel where it should be pipeline = unnecessary cost.

Key concepts:

Criteria: latency · cost · governance · fault tolerance · coordination complexity · size of the team that will maintain

======== MODULE 4.2 ========
4.2 ~50 min

🎯 The Orchestrator

The orchestrator is not just "the agent who calls others". It's infrastructure. Understand why it is the most critical component of the system.

What is it:

System responsible for coordinating agents — not just calling functions, but managing state, handling failures, keeping the objective in focus throughout the entire execution.

Why learn:

Confusing orchestrator with "agent that calls other agents" leads to fragile systems. The orchestrator is an infrastructure component, not a business logic component.

Key concepts:

Persistent state · Failure handling · Goal maintenance · Cost control · Centralized logs · Difference from agent to orchestrator

What is it:

As the database is infrastructure (you don't think about it while it works, you panic when it fails), the orchestrator is infrastructure for multi-agent systems.

Why learn:

Infrastructure requires investment in reliability, monitoring and recovery that application code does not require. This mindset changes how you design it.

Key concepts:

High availability · Automatic recovery · Proactive monitoring · Latency SLA · Scalability · Operating cost

What is it:

Complete list of responsibilities: plans task decomposition, routes to correct agents, monitors status, detects failures, replans, controls cost, maintains logs.

Why learn:

Every responsibility you don't implement is a silent point of failure that will show up in production at the worst possible time.

Key concepts:

Planning · Dynamic routing · State monitoring · Deviation detection · Replanning · Budget control · Audit trail

What is it:

The fundamental architectural decisions when designing an orchestrator: monolithic vs. distributed, synchronous vs. asynchronous, stateful vs. stateless.

Why learn:

Each decision has direct trade-offs in operating cost, debugging complexity, and ability to scale. Choosing without criteria is gambling.

Key concepts:

Monolithic: simple to operate, does not scale horizontally · Distributed: scales, but complex · Stateful: easy recovery · Stateless: easy scale

What is it:

The point at which an orchestrator gains enough features (logs + permissions + cost + coordination + UI) to become a standalone enterprise product.

Why learn:

Understanding this threshold helps you decide: build in-house or buy an orchestration product (LangGraph Cloud, CrewAI Enterprise, etc.).

Key concepts:

Features that cross the threshold: RBAC · audit trail · cost dashboard · multi-tenant · guaranteed SLA · enterprise support

What is it:

Orchestration products landscape in 2026: LangGraph, CrewAI Enterprise, Azure AI Studio, OpenAI Enterprise Controls — all have an orchestration layer.

Why learn:

Knowing what exists avoids reinventing the wheel. But few products expose the orchestrator as a transparent product — knowing the difference saves months of work.

Key concepts:

LangGraph: stateful workflows · CrewAI Enterprise: RBAC + audit · Azure AI Studio: Microsoft integration · OpenAI Controls: budget + permissions

4.3 ~45 min

📋 BMad in Practice

Breakdown-Manage-Delegate: the architectural method for decomposing complex objectives into tasks executable by specialized agents.

What is it:

BMad (Breakdown-Manage-Delegate) is a reasoning method for decomposing complex objectives into agent-executable tasks, with explicit dependencies and priorities.

Why learn:

Without a decomposition method, agents are given vague tasks and fail or produce inconsistent results. BMad makes decomposition repeatable and auditable.

Key concepts:

Systematic decomposition · Dependencies between tasks · Success criteria per task · Traceability from objective to result

What is it:

Receives the objective → identifies subproblems → creates a list of tasks with dependencies → prioritizes by criticality. Output: TaskList with dependency graph.

Why learn:

A bad decomposition propagates error throughout all subsequent phases. Ill-defined tasks guarantee ill-defined results.

Key concepts:

Task granularity · Dependency graph (DAG) · MECE (Mutually Exclusive, Collectively Exhaustive) criterion · Impact priority

What is it:

Organizes tasks in an execution tree, defines which are parallel (independent) vs. sequential (dependent), estimates the cost of tokens for each.

Why learn:

The organization defines latency and total cost. Executing in parallel what could be parallel reduces time from 10x to 1x. Sequencing what needs to be sequential avoids inconsistent data.

Key concepts:

Tree vs. execution graph · Identification of critical paths · Cost estimation per task · Total execution budget

What is it:

Assigns each task to the most appropriate agent (by skill, cost, availability), provides the necessary context and defines the success criteria per task.

Why learn:

Delegating the wrong task to the wrong agent is the most expensive waste: you pay for the processing AND the useless result.

Key concepts:

Agent-task matching · Minimum context injection · Acceptance criteria per task · Timeout per agent · Result callback

What is it:

Practical implementation: TaskTree class, Task with dependencies field, executor that respects the order of the graph, progress monitor with callbacks.

Why learn:

Seeing the code transforms the method from theory to actual implementation. You'll recognize the pattern in LangGraph, CrewAI, and other tools.

Key concepts:

TaskTree as DAG · Topological sort for execution · Status tracking: pending → running → done/failed · Progress callbacks

What is it:

"Analyze competition and generate reports" → Breakdown into 8 tasks → Manage with 3 parallel + 5 sequential → Delegate for researcher, analyst and writer.

Why learn:

Concrete examples establish the method. You will see exactly how the 3 phases connect in a case that any company could use tomorrow.

Key concepts:

Real decomposition of business problem · Identification of parallelism · Selection of agents by specialty · Estimated cost vs. human cost

======== MODULE 4.4 ========
4.4 ~50 min

👥 Specialized Agents

Planner, Executor, Reviewer, Auditor — each role has defined responsibilities and interfaces. Learn how to define contracts between agents for robust systems.

What is it:

The logic behind using narrowly scoped agents: less context needed, lower token cost, lower chance of hallucination, easier to assess quality.

Why learn:

A generalist do-it-all agent is expensive and inaccurate. Specialization is the difference between $50 and $5 cost for the same result.

Key concepts:

Minimal context window · Specialized prompt by domain · Quality assessment by specific criteria · Smaller model = lower cost

What is it:

Agent that receives complex objective and produces structured plan (list of tasks with dependencies, necessary agents, success criteria). Use a larger and more expensive model.

Why learn:

A bad plan contaminates the entire execution. Investing in a better planning model saves 10x in wasted execution tokens.

Key concepts:

Structured output (JSON) · Identification of required agents · Cost estimation · Verifiable success criteria

What is it:

Receives a specific and well-defined task, executes it using its tools (APIs, database, code), returns structured results. You can use a smaller and cheaper model.

Why learn:

It is the agent that executes the most and consumes the most tokens. Optimizing the executor (template, prompt, tools) is where you control the biggest cost of the system.

Key concepts:

Strict input schema · Tool selection · Output schema with trust flag · Max_iterations to avoid infinite loop · Cost per execution

What is it:

Receives results from the executor, evaluates quality against criteria defined by the planner, approves (passes on) or rejects with structured feedback for the executor to try again.

Why learn:

Without a reviewer, you deliver results of varying quality to the end user. The reviewer is the automatic quality control of the system.

Key concepts:

Assessment rubric · Structured feedback · Approval with score · Max_revision_cycles · Escalation to human when necessary

What is it:

Records all actions of all agents, checks compliance with policies, generates immutable audit trail, alerts when anomalies or violations are detected.

Why learn:

In an enterprise environment, without an audit trail you cannot explain what happened, you cannot pass an audit and you do not comply with regulations such as SOC2, LGPD, HIPAA.

Key concepts:

Append-only log · Integrity hash · Real-time alerts · Compliance reporting · Regulatory data retention

What is it:

Formal definition of interfaces between agents: input schema (what it accepts), output schema (what it returns), time SLA, maximum allowed cost, failure behavior.

Why learn:

Explicit contracts allow you to exchange one agent for another without breaking the system. Without contracts, you have tight coupling disguised as flexibility.

Key concepts:

Interface-first design · Pydantic schemas · SLA as time contract · Budget per agent · Schema version compatibility

======== MODULE 4.5 ========
4.5 ~50 min

🗃️ State Management

When 3 agents read and write the same data simultaneously, what happens? Learn how to manage state in conflict-free multi-agent systems.

What is it:

The central challenge of multi-agent systems: when multiple agents need to read and write to the same shared state, race conditions, inconsistencies and corrupted data arise.

Why learn:

Corrupt state in production is the most difficult bug to debug and the most dangerous. Understanding the problem before choosing the solution avoids days of pain.

Key concepts:

Race condition · Dirty read · Lost update · Stale data · CAP theorem applied to agents

What is it:

Use Redis as a shared blackboard where all agents read and write. Distributed locks (Redlock) avoid simultaneous write conflicts.

Why learn:

Redis is the industry standard solution for high-performance distributed state. Redlock resolves conflicts without the complexity of distributed databases.

Key concepts:

Redis as shared memory · Redlock for mutual exclusion · TTL for temporary data · Pub/Sub for notifications · Pipeline for atomic operations

What is it:

Each agent maintains its own isolated state. When finished, it delivers results to the coordinator who combines all partial states into the final state.

Why learn:

Simpler, less conflicts, easier to debug. For tasks where agents do not need to share state during execution, it is the obvious choice.

Key concepts:

Immutable local state · Result as immutable output · Merge function in coordinator · No lock required · Simpler to test

What is it:

When two agents update the same field simultaneously, you need a strategy: last-write-wins (simple, loses data), merge (combines, complex), or manual (human decides).

Why learn:

The conflict resolution strategy defines the data integrity of your system. Choosing last-write-wins can make you lose hours of processing results.

Key concepts:

Last-write-wins (LWW) · Three-way merge · Operational transformation · Conflict-free Replicated Data Types (CRDTs) · Human-in-the-loop for critical conflicts

What is it:

Instead of overwriting state, add each change to an immutable log. You can view the entire state history, roll back to any point, and audit changes.

Why learn:

When the system produces unexpected results at 3 am, a versioned state log is the difference between debug in 10 minutes and debug in 10 hours.

Key concepts:

Event sourcing · Append-only log · State reconstruction · Temporal queries · Snapshot + log for performance

What is it:

Real production problems: state grows indefinitely without TTL, backups fail silently, recovery after failure loses hours of agent work.

Why learn:

Poorly managed state in production is slow (overgrown), expensive (storage) or inconsistent (recovery failure). These problems appear months after launch.

Key concepts:

TTL strategy · Old log compression · Incremental backup · Recovery point objective (RPO) · Recovery time objective (RTO)

🔀 Handoffs and Routing

The formal transfer of control between agents. Learn routing criteria, handoff rollback and how to pass context without exploding the context window.

What is it:

Formal transfer of control AND context from one agent to another, with assurance that state is consistent before transfer and confirmation of receipt.

Why learn:

Informal handoff (agent A ends, agent B begins) is the main source of context and data loss in multi-agent systems. Formalizing ensures continuity.

Key concepts:

Handoff packet (transferred data) · Confirmation of receipt · Consistent state before transfer · Handoff log · Rollback if destination does not confirm

What is it:

Four routing strategies: by content (task type), by load (least busy agent), by skill (best suited agent for the domain), by cost (cheapest agent).

Why learn:

Bad routing is wasted efficiency. Sending legal analysis to a code agent, or a simple task for the more expensive model, are avoidable errors with good routing.

Key concepts:

Content-based routing · Load balancing · Skill-based routing · Cost-aware routing · Combination of criteria with weights

What is it:

A lightweight LLM (classifier) ​​analyzes the input and decides which agent should receive the task. Alternative: deterministic rules based on keywords or data type.

Why learn:

Routing with LLM is flexible but adds latency and cost. Deterministic rules are fast but strict. Knowing when to use each approach is an architectural skill.

Key concepts:

Classifier prompt · Few-shot examples for classification · Confidence threshold · Fallback to human when confidence low · Routing latency

What is it:

What happens when the destination agent fails after receiving the handoff? Rollback protocol: control returns to the originating agent or is transferred to a third fallback agent.

Why learn:

Without a defined rollback, a handoff failure leaves the task in an undefined state — neither at the source agent nor at the destination. This silently crashes the system.

Key concepts:

Two-phase commit for handoff · Confirmation timeout · Automatic rollback · Pending handoff log · Partial state cleanup

What is it:

Techniques for passing relevant context from one agent to another without including everything (which would explode the context window and the cost): summarization, entity extraction, semantic compression.

Why learn:

Giving too much context is expensive. Going too little leads to mistakes. Mastering this technique is what separates efficient multi-agent systems from expensive ones.

Key concepts:

Context summarization · Entity extraction · Relevance filtering · Hierarchical context (full + summary) · Context versioning

What is it:

Four patterns: sequential (A→B→C), conditional (if X→B, if Y→C), parallel (A→B and C simultaneously), dynamic (A decides at runtime which agent to go to).

Why learn:

Each pattern solves a different class of problem. Knowing them all allows you to choose the simplest one that solves your case — simplicity is reliability.

Key concepts:

Sequential: predictable and auditable · Conditional: flexible with risk of complex logic · Parallel: fast but costly · Dynamic: maximum flexibility with maximum complexity

4.7 ~45 min

⚡ Parallel Execution

Fan-out, fan-in, asyncio and timeouts. Learn how to run multiple agents simultaneously, combine results, and manage the cost of parallelism.

What is it:

Run multiple agents simultaneously when tasks are independent of each other. 10 documents in parallel = 1 min. 10 documents in series = 10 min.

Why learn:

Latency defines user experience and competitiveness. Systems that perform independent tasks in series are 10x slower than they need to be.

Key concepts:

Task independence (precondition) · Amdahl's Law speedup · Critical path identification · When parallelism compensates for overhead

What is it:

An input is distributed to multiple agents that execute simultaneously, each with its own subtask. The orchestrator waits for everyone to complete.

Why learn:

Fan-out is the pattern that transforms sequential analysis into simultaneous analysis. It is the foundation of any system that needs to scale to large volumes.

Key concepts:

Input partitioning · Uniform load distribution · Maximum concurrency control · API rate limiting · Semaphore to limit parallelism

What is it:

Multiple results from parallel agents are combined into a single result. Strategies: merge (concatenates), vote (majority), summarize (summarizes), aggregate (calculates).

Why learn:

The quality of the fan-in determines the quality of the final result. Naively combining results produces inconsistent results despite quality parallel analysis.

Key concepts:

Merge strategy: concatenation, voting, summarization · Conflict of results · Weight by agent confidence · Partial result when an agent fails

What is it:

Python implementation: asyncio.gather() to run multiple coroutine agents in parallel, TaskGroup for better control, exception handling in asynchronous context.

Why learn:

asyncio is the Python standard for I/O parallelism (which is exactly what LLM calls are). Without asyncio, you are leaving performance on the table.

Key concepts:

asyncio.gather() with return_exceptions · asyncio.TaskGroup (Python 3.11+) · Semaphore to limit concurrency · Asynchronous exception handling

What is it:

What happens when one of 10 parallel agents takes 5x as long? asyncio.wait_for() for timeout, canceling slow tasks, returning partial results without the agent that delayed.

Why learn:

Without timeout, a slow agent blocks the entire system. With too aggressive a timeout, you lose valid results. Calibrating timeout is a systems engineering skill.

Key concepts:

asyncio.wait_for() · asyncio.wait(return_when=FIRST_EXCEPTION) · Partial result strategy · P95 latency as a basis for timeout · Retry after timeout

What is it:

The math of parallelism: 10 agents in parallel cost approximately 10x more in tokens, but deliver in 1/10 the time. How to optimize: batching, smaller model for simple subtasks.

Why learn:

Parallelism without cost control is like hiring 10 senior consultants when 8 juniors and 2 seniors do the same work for less. The architecture defines the operational cost.

Key concepts:

Cost per parallel task · Batching to reduce overhead · Tier model (cheap for simple analysis, expensive for synthesis) · Maximum budget per parallel execution

4.8 ~50 min

🔧 Error Handling

Agent-specific failure types, retry with backoff, circuit breaker and when to escalate to human. Resilience is not optional.

What is it:

Classification of agent-specific failures: LLM returns invalid format, tool call fails, infinite reasoning loop, context window overflow, cost exceeds budget, agent "hangs".

Why learn:

Each type of failure requires a different response. Treating infinite loop equal to invalid format produces systems that "mask" errors instead of resolving them.

Key concepts:

Invalid format → retry with instruction · Loop → max_iterations · Context overflow → summarize · Budget → checkpoint + escalation · Tool failure → fallback tool

What is it:

For transient failures (API timeout, rate limit, error 500), wait exponentially before trying again: 1s, 2s, 4s, 8s... with jitter to avoid thundering herd.

Why learn:

Rate limits are the most common cause of failure in systems with many parallel agents. Retry without backoff worsens the rate limit. With backoff, you resolve it automatically.

Key concepts:

Exponential backoff: 2^n seconds · Random jitter · max_retries (3-5) · Retryable error vs. non-retryable tenacity library Python

What is it:

When agent A fails beyond max_retries: try agent B with a different approach (another model, another prompt, another tool), or return partial result with uncertainty flag.

Why learn:

Systems without fallback become systems that fail completely. With fallback, you degrade gracefully — deliver inferior results but deliver. That's the difference between 99.9% and 95% uptime.

Key concepts:

Fallback chain: A → B → C → partial result · Fallback model (more expensive, more capable) · Partial result with confidence score · Graceful degradation

What is it:

Pattern that "opens the circuit" when a service fails N times in sequence, temporarily stopping calling it. Tests periodically if recovered, closes circuit when OK.

Why learn:

Without a circuit breaker, you keep trying to call a dead service, wasting tokens and increasing latency. With it, you detect systemic failure and stop wasting resources.

Key concepts:

States: Closed → Open → Half-Open · Failure threshold to open · Timeout to attempt to close · pybreaker library · Circuit state monitoring

What is it:

Define criteria for scaling to human: cost above the limit, irreversible action (delete data, make payment), low confidence in the result, failure after all fallbacks.

Why learn:

Systems that never scale to human eventually make irreversible mistakes. Systems that scale too much never gain real autonomy. Calibrating the threshold is both art and science.

Key concepts:

Escalation triggers: cost · irreversibility · trust · deadline · Human approval as checkpoint · Approval timeout · Notification by channel (Slack, email)

What is it:

How to resume execution after failure without repeating everything: state checkpointing after each significant step, replay from the last stable checkpoint, skipping already completed steps.

Why learn:

If a 50-step system fails at step 48 and starts over from scratch, you lose $40 in tokens every time it fails. Checkpointing turns failure from catastrophe into inconvenience.

Key concepts:

Checkpoint after each step · Idempotence of steps · Replay from checkpoint · Skip of completed steps · Garbage collection of old checkpoints

🔄 Automatic Replanning

When the plan fails, the system needs to detect and create a new plan. Unlike retry — replanning changes the strategy, it doesn't repeat the same thing expecting a different result.

What is it:

Replanning is when the system detects that the current plan is not working and creates a new plan. Different from retry (trying the same thing again with the hope of a different result).

Why learn:

Without replanning, agents get stuck in loops trying the same failed plan over and over. Replanning is what transforms a fragile agent into an adaptive system.

Key concepts:

Retry: same plan, try again · Replanning: new plan, new strategy · Trigger: N failures of the same plan · Cost: new LLM call to plan

What is it:

Metrics that indicate that the plan is not progressing: N iterations without state change, cost above the threshold, success criterion not approaching, same error repeating.

Why learn:

Early drift detection saves tokens. An agent that takes 100 iterations to detect that it is lost has wasted 90 iterations of tokens.

Key concepts:

Progress metric: distance from the objective · Stagnation: N iterations without progress · Cost threshold: budget consumed · Error pattern: same error N times

What is it:

When to trigger replanning: automatic by metrics (most common), or manual by human (for critical cases). Cost: replanning requires a new LLM call to create the plan. Risk: loop replanning.

Why learn:

Automatic replanning without limits becomes an infinite planning loop. With a well-calibrated limit, you have an adaptive system without the risk of loops.

Key concepts:

Automatic trigger by metrics · Manual trigger by human flag · Cost: expensive model for replanning · max_replanning_cycles · Log of all attempted plans

What is it:

Three strategies: total replanning (new plan from scratch, discards the current one), partial replanning (adjusts only the problematic part, keeps what worked), model escalation (uses a more capable model).

Why learn:

Total replanning discards work that worked. Partial is more efficient but more complex to implement. Choosing the wrong strategy wastes tokens or doesn't solve the problem.

Key concepts:

Full replanning: simpler, loses context · Partial: preserves progress, more complex · Model escalation: claude-opus-4 in place of haiku when haiku failed

What is it:

Maximum N redesigns (typically 3) before scaling to human. Without this limit, the system can enter a replanning loop consuming tokens indefinitely.

Why learn:

Unlimited replanning systems have caused $10,000+ LLM API bills in real incidents. The limit is the last protection mechanism against runaway costs.

Key concepts:

max_replanning: 2-3 cycles · Automatic escalation to human after limit · Log of all plans attempted for debugging · Alert when limit approaches

What is it:

Two cases: search agent that does not find information after 3 searches → replanning changes the query strategy. Code agent that fails test 5x → replanning changes implementation approach.

Why learn:

Concrete examples show when replanning works and when the failure is fundamental (the goal is impossible). This distinction avoids endless replanning of unattainable goals.

Key concepts:

Failure diagnosis before replanning · Real strategy change (not minimal variation) · Learning: the new plan uses context from previous failures

📡 A2A Protocol

Agent-to-Agent: the protocol that allows agents from different systems to communicate in a standardized way. The future of interoperability between agents.

What is it:

MCP connects agent to tools (bank, API, file). A2A solves the different problem: how two agents in different systems communicate in a standardized and discoverable way.

Why learn:

Without A2A, every agent-to-agent integration is custom. With A2A, you deploy once and any compatible agent can communicate with yours.

Key concepts:

MCP: agent ↔ tool · A2A: agent ↔ agent · Problem: N×M integration without standard · Solution: standardized discovery and communication protocol

What is it:

Each agent exposes a standardized interface: Agent Card (who I am), Skill List (what I do), Task Protocol (how I receive and process tasks). Any agent can find out and call another.

Why learn:

It's the architecture that enables "agent marketplace" — you can buy a specialized agent from a vendor and it works in your system without custom integration.

Key concepts:

Agent Card: JSON with identity and capabilities · Skill registry: list of skills with input/output schema · Task protocol: how to send, receive and cancel tasks

What is it:

JSON file that describes the agent: name, version, capabilities, available skills with schemas, supported communication protocol, required authentication, limits (rate limit, cost).

Why learn:

The Agent Card is what allows automatic discovery. An orchestrator can read the Card and know exactly how to use the agent without additional documentation.

Key concepts:

JSON schema · Skills with input/output type · Authentication (API key, OAuth) · Declared rate limits · Protocol versioning · Endpoint URL

What is it:

How to send a task (task.send with payload), how to receive progress updates (task.subscribe via streaming), how to cancel a task in progress (task.cancel with cleanup).

Why learn:

The task protocol defines how systems interact. Implementing it correctly ensures your agent is a first-class citizen in the A2A ecosystem.

Key concepts:

task.send: idempotent with task_id · task.subscribe: SSE or WebSocket for streaming · task.cancel: guaranteed cleanup · task.status: alternative polling

What is it:

Complete example: Agent A reads Agent B's Agent Card, checks compatibility, sends task, monitors progress via streaming, receives results and handles errors.

Why learn:

Seeing the full A2A flow code demystifies the protocol and shows that the implementation is simpler than it seems for basic cases.

Key concepts:

httpx for async requests · SSE client for streaming · Pydantic for Agent Card validation · Error handling: task failed vs. network error

What is it:

Comparison table: direct call (simple, coupled), MCP (agent ↔ tool, Anthropic pattern), A2A (agent ↔ agent, interoperability). When to use each one and why.

Why learn:

Using A2A where you should use direct calling is over-engineering. Using direct calling where you should use A2A is a coupling that will cost you dearly when the other agent changes.

Key concepts:

Direct call: same system, zero cost · MCP: external tools, established standard · A2A: external agents, maximum interoperability · Future: A2A as agents' HTTP
