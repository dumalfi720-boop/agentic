🚀 Why parallelism in agents

Amdahl's law applied to agents: the speedup of a system is limited by the sequential component. If 90% of the tasks are parallelizable, you can get up to 10x speedup with parallelism. This turns a 10-minute process into 1 minute — without increasing quality, but eliminating waiting.

🚀 When parallelism pays off

✓

Independent tasks

The execution of each task does not depend on the results of the others. They can run simultaneously without conflict.

✓

Latency > Cost

The user waits for the result and time matters more than the additional cost of multiple agents.

✓

High volume

Too many items to process (documents, leads, analyses) where sequentially would take unacceptable time.

⚠️ When NOT to use parallelism

  • ✗Tasks dependent on each other:B needs the result of A before starting — parallelism doesn't help here
  • ✗Unjustified cost:10 agents cost 10x more — if the user can wait, save
  • ✗Tight rate limits:10 simultaneous agents can hit the API rate limit and all become slow

📤 Fan-out pattern

Fan-out is the moment when a single input is divided into multiple parallel executions. The coordinator divides the work into parts, distributes it to N agents and waits for all of them (or a subset, depending on the strategy) to complete.

📤 Partitioning strategies

1

Partitioning by item

Each agent processes a different item. 10 documents → 10 agents, each with 1 document. Simpler and more common.

2

Partitioning by aspect

Each agent analyzes a different aspect of the same item. Agent A analyzes performance, Agent B analyzes security, Agent C analyzes UX — all of the same product.

3

Batch partitioning

100 documents divided into 10 batches of 10. Each agent processes one batch. Useful when the number of items is much greater than the number of agents.

📥 Fan-in pattern

Fan-in is where the quality of the final result is defined. You can have 10 great parallel analyzes and waste it all with a bad combination strategy. Merge logic is underrated and is where most parallel system bugs hide.

Fan-in strategies

Concatenation (merge)

Unites all results into a list or document. Simple, but may generate duplicates or inconsistencies.

Vote (vote)

Each agent "votes" on an answer. The majority wins. Good for classifications where agents may disagree.

Summarize

An LLM reads all the results and generates a coherent synthesis. More expensive, better combination quality.

Aggregate

Mathematical operation on numerical results (average, sum, max). Perfect for metrics.

Treating partial result

What happens when 2 out of 10 agents fail?

→ Acceptable partial:returns the result of the 8 that worked with the "partial" flag
→ Minimum Threshold:needs at least 7 out of 10 to be valid, otherwise total failure
→ Retry of failures:rerun only the 2 that failed before doing the fan-in

🐍 asyncio in practice with agents

LLM calls are I/O-bound — Python waits for the API response. asyncio was created exactly for this: executing multiple I/O operations simultaneously without blocking. It is the native Python mechanism for agent parallelism.

Python — Fan-out and fan-in with asyncio

import asyncio
from typing import List, Any

async def analyze_document(agent, document: str) -> dict:
    """Agent analyzes a document"""
    result = await agent.run(
        f"Review this document: {document}"
    )
    return {"doc": document[:50], "analysis": result}

async def parallel_analyze(agents, documents: List[str]) -> List[dict]:
    """Fan-out: distributes, waits, returns all"""
    semaphore = asyncio.Semaphore(5)  # max 5 simultaneous

    async def bounded_analyze(agent, doc):
        async with semaphore:
            return await analyze_document(agent, doc)

    # Fan-out: all documents in parallel
    tasks = [
        bounded_analyze(agents[i % len(agents)], doc)
        for i, doc in enumerate(documents)
    ]

    # return_exceptions=True: do not cancel the others if one fails
    results = await asyncio.gather(*tasks, return_exceptions=True)

    # Fan-in: filters errors, returns valid results
    valid = [r for r in results if not isinstance(r, Exception)]
    failed = len(results) - len(valid)

    if failed > 0:
        print(f"⚠️ {failed} agents failed from {len(results)}")

    return valid

⏱️ Timeout and cancellation

In a system of 10 parallel agents, it is practically guaranteed that one will take longer than the others — due to rate limits, a more complex context or simply natural API variation. Timeout is the mechanism that prevents a slow agent from blocking the result of all others.

💡 How to calibrate timeout

Use P95 historical latency as a base: if 95% of calls end in 30s, use a timeout of 45s (1.5x). This eliminates outliers without canceling legitimate cases that take a little longer.

What to do with the partial result

→ Return results of those that ended with the "partial" flag

→ Asynchronous retry of those who timed out

→ Escalate to human if partial result is insufficient

→ Log timeout for historical P95 analysis

💰 Cost of parallelism

The financial equation for parallelism is simple: N parallel agents cost approximately N× the cost of 1 agent. You pay for each agent's time independently. The question is not whether parallel is more expensive (it is), but whether the time benefit justifies the additional cost.

💰 Calculating ROI from parallelism

Example: 100 reviews

Sequential:100 analyzes × 60s = 100min | $5.00

Parallel (10 agents):~10min | $5.00

Same cost, 10x faster

Optimization by model tier

Haiku (simple analysis): $0.25 each

Sonnet (average review): $0.50 each

Opus (complex analysis): $2.50 each

Use the right template for each subtask

Cost Optimization Strategies

  • ✓Batching:group small calls into one larger call
  • ✓Model Tier:use haiku for simple tasks and opus only for complex ones
  • ✓Caching:identical results do not need to be reprocessed
  • ✓Traffic light:competition limit prevents cost explosion
