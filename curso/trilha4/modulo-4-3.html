🎯 What is BMad

BMad is not a framework — it is a method of reasoning. It answers the most difficult orchestration question: given a complex and vague objective, how do you arrive at a set of concrete, prioritized tasks assignable to specific agents?

📊 BMad: the 3 phases

🔨

B — Breakdown

Decompose the objective into subproblems and tasks with dependencies and success criteria

🗂️

M — Manage

Organize tasks in an execution tree, define sequential vs. parallel, estimate cost

📤

D — Delegate

Assign each task to the most appropriate agent with minimal context and acceptance criteria

💡 Why BMad instead of improv?

Without a method, each new problem generates an inconsistent ad-hoc solution. With BMad, you have a repeatable process that any team member can apply. The result: auditable planning, trackable tasks, and systems that you can debug when something goes wrong.

TOPIC 2
2

🔨 Breakdown Phase: break down the objective

The most critical phase — and the most underestimated. A bad decomposition propagates error throughout the system. Ill-defined tasks guarantee ill-defined results. Breakdown transforms a vague goal into a concrete, verifiable task graph.

🔨 Step-by-step Breakdown Process

1

Formulate the objective in verifiable terms

Bad: "analyze the market". Good: "produce a report with: 5 identified competitors, feature comparison in a table, pricing analysis and positioning recommendation".

2

Identify independent subproblems

Ask, “What needs to happen for this to be complete?” Each answer is a potential subproblem. Use MECE principle: mutually exclusive, collectively exhaustive.

3

Map dependencies between tasks

Question: "Which other task does this task need before starting?" This forms the executing DAG (Directed Acyclic Graph).

4

Define success criteria per task

Each task needs a verifiable criteria: "list of 5 competitors with URL, category and pricing". Without criteria, the agent does not know when it stopped.

⚠️ Common mistakes in Breakdown

  • ✗Wrong granularity:tasks that are too big (agent doesn’t know where to start) or too small (unnecessary overhead)
  • ✗Circular dependencies:A depends on B which depends on A — the system crashes before starting
  • ✗Vague success criteria:"look carefully" is not verifiable. "Produce report with N specific fields" is.
TOPIC 3
3

🗂️ Manage Phase: organize execution

With tasks and dependencies defined, Manage organizes execution to be efficient. The central decision: what can be parallel (saves time) and what needs to be sequential (respects dependencies). The correct organization is the difference between 10 minutes and 1 hour of execution.

🗂️ What Manage produces

1. Execution tree

Transforms the DAG of dependencies into an order of execution: which tasks start first, which wait for others, which are the critical path.

2. Parallel/sequential classification

Tasks without dependencies on each other are marked as parallel. Tasks with dependencies are sequential. This defines the execution structure.

3. Total cost estimate

Adds the estimated cost of tokens for each task (based on complexity and agent that will execute). Total budget before starting execution.

📊 Identifying the critical path

The critical path is the longest sequence of dependent tasks. The total execution duration will never be less than the critical path. To speed up the system, optimize the tasks on the critical path — optimizing the others doesn't help.

Research A (2min) ─┐

Research B (2min) ─┤→ Analysis (5min) → Report (3min)

Research C (2min) ─┘

# Critical path: Analysis (5min) + Report (3min) = 8min

# Parallel searches take 2min (not 6min)

# Total: 10min (not 12min if they were all in series)

TOPIC 4

📤 Delegate phase: assign to the right agent

With the execution tree defined, Delegate assigns each task to the most appropriate agent, passes exactly the necessary context (no more, no less) and defines the acceptance criteria that the agent will use to know when it is finished.

Task → agent matching criteria

1 Domain:agent trained/promised for the type of task (research, code, writing)
2 Cost:simple task → cheap model; complex task → expensive model
3 Availability:agent with lowest current load receives the task
4 Reliability:agent success history for this type of task

The delegation package

task_description

What to do in clear and specific natural language

context (minimum required)

Result of predecessor tasks that this agent needs

acceptance_criteria

How the agent knows it is finished (expected output schema)

timeout + max_cost

Time and budget limits for this specific task

🐍 BMad in Python

The BMad implementation in Python uses directed graph concepts (DAGs) to represent dependencies between tasks and topological execution to respect the correct order. You'll recognize this exact pattern in LangGraph and other orchestration tools.

Python — BMad Data Structures

from dataclasses import dataclass, field
from enum import Enum
from typing import List, Optional, Dict, Any
import asyncio

class TaskStatus(Enum):
    PENDING  = "pending"
    RUNNING  = "running"
    DONE     = "done"
    FAILED   = "failed"

@dataclass
class Task:
    id: str
    description: str
    agent_id: str
    dependencies: List[str] = field(default_factory=list)
    acceptance_criteria: str = ""
    max_cost_usd: float = 1.0
    timeout_seconds: int = 120
    status: TaskStatus = TaskStatus.PENDING
    result: Optional[Any] = None
    cost_usd: float = 0.0

class TaskTree:
    def __init__(self):
        self.tasks: Dict[str, Task] = {}

    def add_task(self, task: Task):
        self.tasks[task.id] = task

    def get_ready_tasks(self) -> List[Task]:
        """Tasks whose dependencies have been completed"""
        ready = []
        for task in self.tasks.values():
            if task.status == TaskStatus.PENDING:
                deps_done = all(
                    self.tasks[dep].status == TaskStatus.DONE
                    for dep in task.dependencies
                )
                if deps_done:
                    ready.append(task)
        return ready

Python — BMad Executor with real-time progress

class BMadExecutor:
    def __init__(self, task_tree: TaskTree, agent_registry: dict):
        self.tree = task_tree
        self.agents = agent_registry
        self.total_cost = 0.0

    async def execute(self, budget_usd: float = 10.0):
        while not self._is_complete():
            ready = self.tree.get_ready_tasks()
            if not ready:
                break  # deadlock or complete

            # Run ready-made tasks in parallel
            tasks = [self._run_task(t, budget_usd) for t in ready]
            await asyncio.gather(*tasks, return_exceptions=True)

    async def _run_task(self, task: Task, budget: float):
        if self.total_cost >= budget:
            task.status = TaskStatus.FAILED
            return

        task.status = TaskStatus.RUNNING
        agent = self.agents[task.agent_id]
        context = self._build_context(task)

        try:
            result = await asyncio.wait_for(
                agent.run(task.description, context),
                timeout=task.timeout_seconds
            )
            task.result = result
            task.status = TaskStatus.DONE
            self.total_cost += task.cost_usd
            print(f"✓ {task.id} completed — cost: ${task.cost_usd:.3f}")
        except Exception as e:
            task.status = TaskStatus.FAILED
            print(f"✗ {task.id} failed: {e}")

📖 Real example: competition analysis

Objective: "Analyze the top 5 competitors and generate an executive report with comparative analysis and positioning recommendations." Let's apply the 3 phases of BMad to this real problem.

B

Breakdown — 8 tasks identified

T1: Identify competitors (without dependency)
T2: Collect features from Competitor A (depends on T1)
T3: Collect features Competitor B (depends on T1)
T4: Collect features Competitor C (depends on T1)
T5: Analyze everyone’s pricing (depends on T2, T3, T4)
T6: Create feature comparison table (depends on T2, T3, T4)
T7: Strategic analysis (depends on T5, T6)
T8: Generate final report (depends on T7)
M

Manage — Execution tree

Wave 1: T1 (sequential — identifies competitors)

Wave 2: T2, T3, T4 (parallel — simultaneous collection)

Wave 3: T5, T6 (parallel — simultaneous analysis)

Wave 4: T7 (sequential — strategic synthesis)

Wave 5: T8 (sequential — final report)

Estimated cost: $0.45 | Time: ~12 min

D

Delegate — Assignment by specialty

ResearcherT1 (identification), T2, T3, T4 (collection) — cheap model, lots of web search
AnalystT5 (pricing), T6 (table), T7 (strategy) — average model, structured analysis
EditorT8 (report) — medium model, professional document generation

✓ The result of the BMad applied

Instead of a generalist agent trying to do everything in 30 minutes (and probably failing in strategic analysis), you have 3 specialized agents delivering in 12 minutes, with a controlled cost and trackable results per task. When T6 produces a wrong table, you know exactly which agent was responsible and can correct just that task.
