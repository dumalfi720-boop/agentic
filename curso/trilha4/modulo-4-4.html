🎯 Why Specialize Agents

Specialization is not about division of labor for convenience — it is about economic efficiency and technical quality. A narrowly scoped agent needs less context, uses a cheaper model, produces more consistent results, and is much easier to evaluate.

Generalist Agent (the problem)

  • ✗Huge prompt with instructions for all domains
  • ✗More expensive model needed to maintain quality
  • ✗More frequent hallucination (extensive context confuses)
  • ✗Difficult to evaluate — by what criteria do you judge?
  • ✗Impossible to partially replace without redoing everything

Specialized Agents (the solution)

  • ✓Concise, domain-specific prompt
  • ✓Cheapest model — simple tasks don't need an opus
  • ✓Less hallucination — minimal context, clear instruction
  • ✓Assessment by domain-specific criteria
  • ✓Individually replaceable without affecting others

📊 The impact on cost

A generalist agent using Claude Opus 4 for each task can cost $0.50 per run. The same system with specialized agents — Opus for planning, Haiku for research, Sonnet for analysis — can cost $0.08.6x cheaper with equal or better qualityin each specialty.

TOPIC 2
2

🧭 Planning Agent

The Planner is the architect of the system — receives the complex objective and produces a structured plan executable by other agents. It is the only agent that justifies a high-cost model, because a bad plan contaminates the entire subsequent execution.

🧭 Planner Agent Features

Model: Claude Opus 4 or GPT-4o — high-quality reasoning
input: Objective in natural language + business context
Output: JSON with TaskList, dependencies, agents, criteria
Freq.: Once per objective (not per task)
Cost: High per run but only 1 call per objective
Failure: System does not start — must be validated before proceeding

Example of Planner output (JSON)

{
  "objective_summary": "Analysis of 5 competitors with executive report",
  "tasks": [
    {
      "id": "T1", "description": "Identify top 5 competitors",
      "agent": "researcher", "dependencies": [],
      "success_criteria": "JSON list with name, URL and category of 5 companies"
    }
  ],
  "estimated_cost_usd": 0.45,
  "estimated_duration_min": 12
}

⚙️ Executing Agent

The Executor is the agent that executes the most and consumes the most tokens. It receives a specific task, uses its tools to complete it, and returns a structured result. This is where system cost is incurred — and where model optimization has the greatest impact.

⚙️ The Executor loop

1 Receives task_description + context + acceptance_criteria
2 Decide which tools to use (web search, code, database, API)
3 Run the tools and observe the results
4 Evaluates whether the result satisfies the acceptance_criteria
5 If yes → returns result. If not → try again (max_iterations)
6 Returns: { result, confidence_score, cost_usd, iterations }

💡 Template selection by task

Web search:claude-haiku-3 — fast and cheap

Structured analysis:claude-sonnet-4 — good balance

Complex reasoning:claude-opus-4 — when you need it

Code generation:claude-sonnet-4 — excellent for code

⚠️ max_iterations is mandatory

An Executor with no iteration limit can go into a loop trying to satisfy an impossible criterion. Always setmax_iterations=5(or similar) — after the limit, returns partial result with failure flag.

🔍 Reviewer Agent

The Reviewer is the system's automatic quality control. Without it, you deliver results of varying quality to the end user without filtering. With it, each result undergoes careful evaluation before continuing in the pipeline.

📥
Input

Executor Result + acceptance_criteria defined by the Planner

⚖️
Process

Evaluates quality by specific criteria, assigns scores and generates structured feedback

📤
Output

approved: true/false, score: 0-10, feedback: "what's missing"

🔄 The Executor → Reviewer cycle

forattempt in range(max_revision_cycles):

result = executor.run(task, context)

review = reviewer.evaluate(result, criteria)

ifreview.approved:

returnresult# approved

# Executor receives feedback and tries again

context.add_feedback(review.feedback)

returnescalate_to_human(result)# out of stock reviews

📊 Auditor Agent

The Auditor does not participate in the execution — he observes and records it. In enterprise environments, the audit trail is not optional: it is a compliance, security and governance requirement. The Auditor is the agent who makes the system explainable.

📊 What the Auditor records

• Every action of every agentwith timestamp and cost
• Each handoffbetween agents with transferred context
• Every routing decisionwith justification
• Accumulated costin real time per agent and total
• Policy Violations(unauthorized access, cost above limit)
• Behavioral anomalies(many iterations, atypical output)

📋 Compliance requirements

  • •SOC2: audit trail of all operations
  • •LGPD/GDPR: personal data access log
  • •HIPAA: complete audit of access to health data

💡 Implementation: append-only log

The audit trail is an append-only log with integrity hash (each entry references the hash of the previous one). Impossible to change retroactively. Can export to corporate SIEM.

TOPIC 6

📜 Contracts between agents

Formal contracts between agents are what turns a set of coupled scripts into a replaceable and testable system. With explicit contracts, you can swap Executor A for Executor B without touching the Reviewer or the Planner — as long as they both respect the same contract.

📜 Anatomy of an agent contract

class ExecutorContract(BaseModel):

# Input schema (what the Executor accepts)

task_description: str

context: Dict[str, Any]

acceptance_criteria: str

max_cost_usd: float = Field(gt=0, le=5.0)

timeout_seconds: int = Field(ge=10, le=300)

# Output schema (what the Executor always returns)

classExecutorResult(BaseModel):

result: Any

confidence_score: float# 0.0 - 1.0

cost_usd: float

iterations: int

status: Literal["done", "partial", "failed"]

✓ With formal contracts

  • ✓Agent replacement without modifying others
  • ✓Unit tests by isolated agent
  • ✓Automatic output validation before use
  • ✓Automatically generated schema documentation

✗ No formal contracts

  • ✗Implicit coupling discovered only in production
  • ✗Changing one agent silently breaks another
  • ✗Impossible to test agent in isolation
  • ✗No one knows what each agent really needs

💡 Use Pydantic for contracts in Python

Pydantic automatically validates that the agent's input is valid before executing, and that the output is valid before passing on to the next one. When the output is invalid, you get a ValidationError with the specific field — not a silently wrong result discovered 10 steps later.
