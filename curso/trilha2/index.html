🧠 LLM as Brain

Model selection, context window, temperature, cost per token and how the LLM decides the agent's next action.

What is it:

Choosing the right model is an agent's first architectural decision. Claude 3.5, GPT-4o, Gemini Pro, Mistral, and Llama have very different capabilities and costs for specific use cases.

Why learn:

Using the wrong model is the most common cause of exploding costs or poor results. The right choice can reduce costs by 10x without losing quality.

Key concepts:

Reasoning ability · Response speed · Cost per token · Function calling support · Context limit

What is it:

The context window is the LLM's working memory — everything it can "see" at the same time. Understanding what goes into it (history, tools, documents, instructions) is essential to architect well.

Why learn:

Exploding the context knocks the agent down. Not using context well wastes capacity. Knowing how to estimate the token size of different types of content is a fundamental skill.

Key concepts:

Tokenization · Token estimation by content type · Context overflow · Long context vs. RAG Full context cost

What is it:

Temperature controls the randomness of LLM responses. Close to 0.0 gives deterministic and predictable answers. Closer to 1.0 gives more creative and varied answers.

Why learn:

The wrong temperature is a silent cause of difficult-to-diagnose bugs. High temperature code agent produces non-reproducible code. Creative agent with temperature 0 produces monotonous texts.

Key concepts:

Temperature vs. top-p · Determinism in agents · When to use high vs. low temperature · Seed for reproducibility

What is it:

Each call to LLM costs input tokens + output tokens. In a looping agent, these costs multiply. Knowing how to calculate the real cost per task is essential for economic viability.

Why learn:

Agents without cost control generate surprises at the end of the month. With correct planning, you know in advance how much each task will cost and define budget limits.

Key concepts:

Input vs. output pricing · Prompt cache · Cost per iteration · Budget limits · Agent ROI

What is it:

The strategy of using different models for different agent steps. Large model (GPT-4o, Claude Opus) for complex planning; small model (Haiku, GPT-4o-mini) for simple and repetitive execution.

Why learn:

Reduces costs by 80-90% in the execution stages without sacrificing quality in planning. It is the difference between an economically viable agent and one that is prohibitively expensive at scale.

Key concepts:

Frontier vs. Frontier Models small · Model hierarchy · Routing by complexity · Latency vs. cost · SLAs per stage

What is it:

The internal process by which the LLM interprets the current state, evaluates the available tools and decides: respond to the user, call a tool or request more information. It is the "brain" of the agentic loop.

Why learn:

Understanding this process allows you to write tool prompts and schematics that guide the LLM to make the correct decisions, avoiding infinite loops and unnecessary calls.

Key concepts:

System prompt as constitution · Tool schema as action menu · Stop condition · Tool choice criteria · Stop sequences

💾 Short Term Memory

Conversation buffer, sliding window, automatic summary and compression strategies when the context is full.

What is it:

The conversation buffer is the list of messages (user + assistant) passed to the LLM for each call. It is the simplest way to provide memory to the agent within a session.

Why learn:

Without buffering, each call to the LLM starts from scratch — the agent "forgets" everything. With a poorly managed buffer, the context explodes and the call fails or becomes too expensive.

Key concepts:

ConversationBufferMemory · Message format (role/content) · History size · Accumulated cost per iteration

What is it:

Technique that keeps only the N most recent messages in context, discarding the old ones. Solves the problem of infinite buffer growth.

Why learn:

It is the simplest solution to avoid context overflow. The cost per call is predictable and stable, even for long conversations.

Key concepts:

Window size · Message FIFO · Loss of historical context · ConversationBufferWindowMemory · Trade-off memory vs. cost

What is it:

When the context is about to burst, an LLM (usually small) summarizes the old history into compressed text. The summary replaces the original messages in context.

Why learn:

Preserves the essence of the conversation without breaking the context. Better than sliding window when the history contains critical information that cannot be discarded.

Key concepts:

ConversationSummaryMemory · Summarization trigger · Summary quality · Summary call overhead · Progressive degradation

What is it:

LLMs have difficulty using information from the middle of a very long context ("lost in the middle"). Information at the beginning and end is used better than information in the center.

Why learn:

Long context does not guarantee a better result. In some cases, less relevant context overcomes huge noisy context. This directly impacts the agent design.

Key concepts:

Lost in the middle · Primacy/recency effect · Relevance vs. size · Strategic ordering of the context · Chunk filtering

What is it:

Advanced compression techniques beyond the sliding window: extraction of key facts in a structured list, summarization by topic, and semantic compression that preserves the essentials.

Why learn:

Different use cases require different strategies. A support agent needs a different strategy than a search agent. Knowing the arsenal allows you to choose the right tool.

Key concepts:

Entity extraction · Fact list compression · Entity memory · Hybrid memory · Importance compression

What is it:

Symptoms and causes of short memory failures: agent forgets critical information, contradicts previous answers, or repeats questions already answered. How to identify and diagnose each case.

Why learn:

Memory bugs are difficult to diagnose because the agent "seems" to work but produces inconsistent results. Knowing failure patterns saves hours of debugging.

Key concepts:

Silent context truncation · Shift-to-shift inconsistency · Instruction forgetting · History debugging · Memory regression testing

======== MODULE 2.3 ========
2.3 ~45 min

🗄️ Long Term Memory and RAG

Embeddings, FAISS and Chroma in practice, complete RAG pipeline and how to debug bad retrieval.

What is it:

Agents without long-term memory start from scratch in each session. For a support agent, this means forgetting the customer's history. For a research agent, forgetting what has already been discovered.

Why learn:

Long-term memory is what separates a useful agent from a toy agent. Without it, you don't build systems that get smarter over time.

Key concepts:

Session vs. persistence · Memory types (episodic, semantic, procedural) · When RAG vs. fine-tuning · Persistence cost

What is it:

FAISS (Meta) and Chroma are databases specialized in storing and searching embedding vectors. They allow you to find documents semantically similar to a query in milliseconds.

Why learn:

They are the basis of any RAG system. FAISS is fast and local; Chroma is easy to use and has metadata filtering. Choosing and using the right bank is a central skill in RAG.

Key concepts:

Cosine similarity · HNSW index · Metadata filtering · Disk persistence · Scalability: FAISS vs. Chroma vs. Pinecone

What is it:

Embeddings transform text into high-dimensional numeric vectors where texts with similar meaning are close together in space. "Dog" and "dog" are close; "dog" and "plane" are far apart.

Why learn:

Understanding embeddings allows you to choose the correct embedding model, determine the ideal chunk size, and debug when the search returns irrelevant results.

Key concepts:

Vector dimensionality text-embedding-3-small vs. large · Chunk size · Overlap · Embedding quality per domain

What is it:

The complete RAG pipeline in 4 phases: (1) Indexing — chunking and embedding of documents; (2) Retrieval — vector search for similarity; (3) Injection — adding chunks to the context; (4) Generation — LLM responds with context.

Why learn:

RAG is the most used technique to provide specialized knowledge to agents without fine-tuning. Understanding the complete pipeline allows you to optimize each phase separately.

Key concepts:

Document loaders · Text splitters · Retriever · top-k · Score threshold · Prompt template with context

What is it:

When RAG returns irrelevant chunks, the problem can be: chunk too small (loses context), chunk too large (dilutes the signal), low top-k (loses the relevant chunk) or wrong threshold (includes garbage).

Why learn:

"RAG doesn't work" is the most common and least well diagnosed bug. Knowing how to identify which phase of the pipeline is failing saves days of work.

Key concepts:

Chunk size 256 vs. 512 vs. 1024 · MMR retrieval · Hybrid search · Reranking · Evaluation with RAGAS

What is it:

The cost of a RAG system has two parts: indexing (embedding all documents — paid once) and search (embedding the query for each call — always paid). At scale, these costs become significant.

Why learn:

A corpus of 10,000 documents may cost pennies to index but dollars per day to search. RAG cost planning is different from LLM cost planning.

Key concepts:

Embedding cost per token · Embedding cache · Hosted DB (Pinecone) vector costs vs. self-hosted (FAISS) Incremental index update

======== MODULE 2.4 ========
2.4 ~40 min

🔧 Tools and Function Calling

JSON Schema for tools, structured outputs with Pydantic, custom tools, chaining and security.

What is it:

Tools are external functions that LLM can call to perform actions in addition to generating text: searching the web, reading files, querying databases, calling APIs, executing code.

Why learn:

Without tools, the agent is just a sophisticated text generator. With tools, he can act in the real world. Tools are what transform an LLM into an executable agent.

Key concepts:

Function calling · Tool use · Tool schema · Tool result injection · Cycle: LLM decides → tool executes → result returns to LLM

What is it:

Each tool is defined by a JSON Schema with: name (identifier), description (LLM uses this to decide when to call), parameters (what needs to be passed) and required (mandatory fields).

Why learn:

The quality of the description determines whether LLM will call the right tool at the right time. A poorly described schema causes the agent to ignore available tools or call the wrong one.

Key concepts:

OpenAI function calling format · Anthropic tool use format · Description as instruction · Type annotations · Enum parameters

What is it:

Pydantic allows you to define data models that LLM must follow when responding. The LLM response is automatically validated and parsed into typed Python objects, eliminating manual parsing and format errors.

Why learn:

Agents that rely on manual text parsing break easily. Structured outputs ensure that the LLM output is always valid and usable by the following code.

Key concepts:

BaseModel · Field with description · response_format · Nested models · Validation errors and retry · instructor library

What is it:

Any Python function can become a tool. Practical examples: web search with Tavily, calculator with secure eval, file reader with encoding treatment. Each with correct JSON schema.

Why learn:

Customized tools are the heart of the value you deliver. The difference between a generic agent and an agent useful for your business lies in the specific tools you create.

Key concepts:

@tool decorator · StructuredTool · Stateful tool · Async tools · Error handling in the tool · Standardized return

What is it:

The agent uses the result of one tool as input to call another. Example: web search → extracts relevant data → calls API with this data → summarizes result. Each step depends on the previous one.

Why learn:

Tool chaining is what enables complex multi-step tasks. This is also where most bugs occur — the output from tool A is not the format expected by tool B.

Key concepts:

Tool result parsing · State passing between tools · Dependency between tools · Chaining error · Retry on partial failures

What is it:

Unprotected tools are attack vectors. An agent can be induced (prompt injection) to call tools with malicious parameters. Input validation, action allowlists and mandatory logging are essential defenses.

Why learn:

In production, tools have access to real systems. A compromised agent who can delete files, send emails or make transactions is a critical business risk.

Key concepts:

Prompt injection · Input sanitization · Operations allowlist · Human-in-the-loop for critical actions · Audit log of tool calls

🎯 Reasoning Patterns

Chain-of-Thought, ReAct, Plan-and-Execute, Reflection and when to use each pattern by problem type.

What is it:

Chain-of-Thought (CoT) is the technique of making the LLM explain its reasoning step by step before giving the final answer. "Think step by step" significantly improves quality in complex problems.

Why learn:

CoT improves accuracy in logic and math problems by up to 40%. It is the simplest pattern to implement and the starting point before any more sophisticated pattern.

Key concepts:

Zero-shot CoT ("think step by step") · Few-shot CoT (reasoning examples) · Internal Scratchpad · CoT in tool selection · Extra cost of tokens

What is it:

ReAct intersperses Thought, Action and Observation in a loop. LLM thinks out loud, calls a tool, observes the result and repeats until it has the final answer.

Why learn:

It is the default pattern for LangChain AgentExecutor and most frameworks. Understanding ReAct is understanding how 80% of agents in production work today.

Key concepts:

Thought/Action/Observation loop · Reasoning trace · Tool selection by thought · Stop condition · Maximum number of iterations

What is it:

It separates the agent into two: a Planner (large LLM) that creates the complete task plan, and an Executor (small LLM) that executes each step of the plan. Dramatically reduces costs in the execution stages.

Why learn:

For long tasks with many predictable steps, Plan-and-Execute reduces cost by up to 90% compared to pure ReAct (which uses an expensive model in each iteration). It is the pattern of costly production agents.

Key concepts:

Planner vs. Executor · Plan as list of steps · Replanning when step fails · Dependency DAG · Hierarchy of models by cost

What is it:

After generating a response, the agent uses a second LLM (or the same one with a different prompt) to critique the response, identify errors, and suggest improvements. The improved response goes through a new round of criticism.

Why learn:

Significantly improves quality in creative tasks, code and critical analysis. The agent who self-corrects often outperforms the agent who tries to get it right the first time.

Key concepts:

Actor vs. Critic · Self-critique prompt · Number of refinement iterations · Quality criterion · Cost vs. quality gain

What is it:

Practical guide: CoT for specific analytical problems, ReAct for exploration with tools, Plan-and-Execute for long and predictable pipelines, Reflexion for high-quality generation where error is costly.

Why learn:

Using the wrong standard is spending money without gaining quality. The correct choice depends on: number of steps, task predictability, cost tolerance and quality requirement.

Key concepts:

Dimensions of choice: latency, cost, quality, predictability · Pattern matching · Pattern by domain (code, writing, analysis, research)

What is it:

Comparative cost analysis on tokens per pattern: CoT spends ~2x more tokens than direct prompt. ReAct spends N×(reasoning + context) per iteration. Reflexion costs 3-5x more but may be necessary.

Why learn:

Architectural decision with direct financial impact. A production agent with thousands of daily executions can have a 10x difference in cost depending on the chosen pattern.

Key concepts:

Token budget by default · Cost per completed task · Quality-cost tradeoff · Quality benchmark · Break-even analysis

🔌 MCP in Practice

Model Context Protocol: server/client architecture, connecting database and external APIs, and creating your first MCP server.

What is it:

Model Context Protocol (MCP) is Anthropic's open standard for connecting agents to external tools in a standardized way. Just as USB standardized peripherals, MCP standardizes the connection of tools to agents.

Why learn:

MCP is quickly becoming the industry standard. Tools that support MCP work automatically with any MCP client — you write it once and use it on any agent.

Key concepts:

Open protocol · MCP server · MCP client · Transport (stdio, HTTP) · Automatic tool discovery · Industry adoption

What is it:

The MCP server exposes tools (functions) via a standardized protocol. The MCP client (the agent) discovers and calls these tools. Communication can be via stdio (local process) or HTTP (remote service).

Why learn:

Understanding the architecture allows you to debug connection issues, choose the correct transport (stdio for local, HTTP for production), and scale MCP servers independently of agents.

Key concepts:

JSON-RPC 2.0 · Tool registration · Tool invocation · Stdio transport · HTTP + SSE · Connection lifecycle

What is it:

Using the official Postgres MCP server, you can connect any agent to your database in minutes. The agent can execute queries, list tables and analyze data without any extra code.

Why learn:

Connecting the agent to the database transforms natural language queries into real results. It’s one of the highest immediate ROI use cases for agents in companies with structured data.

Key concepts:

mcp-server-postgres · Connection string · Read-only vs. read-write · SQL sanitization · Minimum permissions · Schema discovery

What is it:

Any REST API can be exposed as an MCP server. You define the tools (endpoints) with your schemas and the agent can call the API naturally, as if it were a Python function.

Why learn:

Allows you to integrate CRMs, ERPs, payment APIs and any external service to the agent without writing custom integration code. A well-made MCP server is reusable by multiple agents.

Key concepts:

API wrapper as MCP server · Auth handling (Bearer token) · Rate limiting · Error propagation · OpenAPI → MCP conversion

What is it:

Claude Code uses MCP natively to connect to external tools. Example: adding a Jira MCP server allows Claude Code to read issues and create PRs linked directly to the task.

Why learn:

Seeing MCP in action in a real product (Claude Code) makes what was abstract concrete. You see exactly how the configuration translates into agent behavior.

Key concepts:

claude_desktop_config.json · mcpServers section · Local vs. local server remote · Tool approval on Claude · MCP connection debug

What is it:

With the official MCP SDK in Python, you can create a working server in less than 30 lines. Defines tools with decorator @server.tool(), implements the logic and registers it on the client.

Why learn:

Creating your own MCP servers allows you to expose any business logic as a reusable tool. It is the basis for building an ecosystem of tools shared between multiple agents.

Key concepts:

mcp Python SDK · @server.tool decorator · Automatically generated tool schema · stdio_server() · Testing with MCP Inspector · Publishing to other agents

======== MODULE 2.7 ========
2.7 ~50 min

⚙️ First Real Agent with LangChain

AgentExecutor vs. LCEL, memory, LangChain tools and complete project: email assistant with debugging.

What is it:

LangChain abstracts the agentic loop (call LLM → process tool call → return result → call LLM again) into reusable components: chains, tools, memory, agents — without you rewriting the boilerplate.

Why learn:

It is the most adopted framework in the Python ecosystem for agents. Knowing LangChain means access to hundreds of ready-made integrations (tools, vectorstores, LLMs) without implementing from scratch.

Key concepts:

Chain abstraction · LLM unified interface · Integrations ecosystem · When LangChain helps vs. when it gets complicated · Alternatives: LlamaIndex, smolagents

What is it:

AgentExecutor is the legacy API that still dominates tutorials. LCEL (LangChain Expression Language) is the new paradigm based on Runnables composition with operator |. LCEL is more flexible and observable.

Why learn:

You will encounter both in practice. Understanding the difference avoids using deprecated APIs in new code and allows you to migrate legacy projects to LCEL consciously.

Key concepts:

Runnable interface · Pipe operator (|) · invoke vs. stream create_react_agent + AgentExecutor LCEL with RunnableWithMessageHistory

What is it:

Adding ConversationBufferMemory or RunnableWithMessageHistory to the agent so that it remembers the conversation history between calls. Includes SQLite persistence for cross-session memory.

Why learn:

Without memory, each interaction with the agent starts from scratch. With well-implemented memory, the agent accumulates context and becomes progressively more useful throughout a conversation.

Key concepts:

memory_key · return_messages · session_id · ChatMessageHistory · SQLChatMessageHistory · Passing memory to prompt

What is it:

LangChain has two ways to create tools: @tool decorator (simple, automatically generates schema from the docstring) and StructuredTool.from_function() (for tools with multiple parameters typed with Pydantic).

Why learn:

Creating correct tools in LangChain is what allows the AgentExecutor or LCEL to call external functions. The tool's docstring directly influences when and how LLM uses it.

Key concepts:

@tool decorator · args_schema with Pydantic · handle_tool_error · Tool with async · Passing state to tools · DuckDuckGoSearchRun, TavilySearchResults

What is it:

Complete project: agent that reads emails from a simulated inbox, sorts them by priority (urgent/normal/spam), and drafts personalized responses to urgent ones. Functional and commented code.

Why learn:

Real project that integrates all concepts: customized tools, memory, LLM as decision maker. In the end you have a functional agent that can be adapted for your own use case.

Key concepts:

Tool: read_inbox, classify_email, draft_reply · Specialized prompt system · Structured output for classification · Processing loop · Decision logging

What is it:

Debugging techniques without external tools: verbose=True in the AgentExecutor, custom callbacks that print each step, structured logging of tool calls and manual tracking of the agentic loop.

Why learn:

LangSmith is not always available and is costly. Knowing how to debug with basic tools ensures that you can diagnose problems in any environment, including production with limited access.

Key concepts:

verbose=True · BaseCallbackHandler · on_tool_start/end · on_llm_start/end · Logging input/output of each step · Identifying the failed step

🔍 Debugging and Tracing

LangSmith in practice, tracking LLM calls, identifying failures, quality metrics and good logging practices.

What is it:

Agents are non-deterministic (the LLM can make different decisions with the same input), have multiple interdependent steps, and an error in one step can manifest itself as an incorrect result much later.

Why learn:

Understanding why agents are difficult to debug is the first step to debugging them well. Applying sequential scripting techniques to a non-deterministic agent is a common pitfall.

Key concepts:

LLM non-determinism · Error propagation between steps · Dependency on external state · Bug reproducibility · Why logs are mandatory

What is it:

LangSmith is the official observability platform for LangChain. With 3 environment variables (LANGCHAIN_TRACING_V2, LANGCHAIN_API_KEY, LANGCHAIN_PROJECT), every execution of the agent is visible as a trace.

Why learn:

LangSmith transforms agent debugging from “printing random screenshots” to “viewing the complete execution graph”. It's the biggest productivity gain in agent debugging available today.

Key concepts:

LANGCHAIN_TRACING_V2=true · LANGCHAIN_API_KEY · LANGCHAIN_PROJECT · Free tier (generous) · Automatic traces without extra code

What is it:

In LangSmith, each agent execution appears as a tree of nodes: each LLM call, each tool call, each chain is a node with input, output, latency and cost. You see exactly what happened.

Why learn:

Viewing the complete trace reduces debugging time from hours to minutes. You see which LLM call generated the wrong reasoning, which tool returned the unexpected result, and where the loop went into crisis.

Key concepts:

Trace tree · Run hierarchy · LLM span · Tool span · Tokens per node · Latency per node · Export trace to share

What is it:

Methodology for identifying the root cause of an agent failure: start with the incorrect output, trace backwards through the trace, identify which step introduced the error and determine whether it is a bug in the prompt, the tool or the code.

Why learn:

Without methodology, agent debugging is a ghostbuster. With reverse tracking methodology, you can find the root cause in minutes even in agents with dozens of steps.

Key concepts:

Reverse trace · Trace comparison (run ok vs. run with error) · Prompt error vs. error. tool error vs. logic error · LangSmith annotations · Trace replay

What is it:

Objective metrics to evaluate agents in production: p50/p95 latency per task, average cost per execution, success rate (task completed without error), fallback rate and user satisfaction.

Why learn:

Without metrics, you don't know if the agent is getting better or worse with changes. Metrics allow you to detect regressions before the user complains and justify investments with data.

Key concepts:

Datasets in LangSmith · Custom evaluators · LLM-as-judge · Success rate tracking · Cost per run dashboard · Threshold alerts

What is it:

Logging standards for agents in production: log each tool call with input/output/cost/latency, log LLM decisions, use correlation IDs to track executions, and define log levels per environment.

Why learn:

An agent without logs is a black box in production. When something fails (and it will), logs are the only way to understand what happened without reproducing the problem.

Key concepts:

Structured logging (JSON) · Correlation ID · Tool call log with latency · LLM decision log · Retention policy · Alerts in production
