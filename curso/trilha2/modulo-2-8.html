Why agents are difficult to debug

Non-determinism, LLM dependency and loops: the three enemies of the developer.

🎲

Non-determinism

The same input can generate different outputs. Temperature > 0 guarantees variation. Reproducing a bug requires capturing the exact state.

🔗

LLM Dependency

Behavior changes with model versions, prompt changes, and even server load. What works today may fail tomorrow.

🔄

Infinite loops

The agent calls the same tool repeatedly, interprets the result wrong, or never decides it is finished. Without a limit, the cost explodes.

Why print() is not enough

Naive approach

agent.run("Pesquise sobre IA")
# Saída: uma string final
# Você não sabe:
# - Quantas chamadas ao LLM?
# - Qual tool foi chamada?
# - Qual foi o custo?
# - Por que parou?

With tracking

with langsmith.trace("agent-run"):
    agent.run("Pesquise sobre IA")
# Você vê:
# - 3 chamadas ao GPT-4o
# - 2 tool calls (search + calc)
# - $0.042 de custo
# - 8.3s de latência total

The Principle of Observability in Agents

A system is observable when you can understand its internal state just by analyzing its external outputs. For agents, this means capturing:

📥

Exact input for each call

📤

Output and tokens used

⏱️

Latency per step

❌

Errors and exceptions

Viewing a complete trace

Anatomy of a trace in LangSmith: what each node represents and how to read it.

Structure of a hierarchical trace

▼ AgentExecutor 8.3s | $0.042
▼ LLM: gpt-4o-mini 1.2s | 423 tokens
• Input: [SystemMessage, HumanMessage, AIMessage...]
• Output: tool_call → search("São Paulo climate")
▼ Tool: search 3.1s | $0.001
• Input: "São Paulo climate"
• Output: "28°C, partly cloudy..."
▼ LLM: gpt-4o-mini 0.9s | 312 tokens
• Input: [history + tool result]
• Output: "The weather in São Paulo is 28°C..."

What to inspect first

  • →Status:Success, Error or Timeout?
  • →Total latency:where is the bottleneck?
  • →Number of iterations:Excessive loops?
  • →Cost:within the expected budget?

Adding metadata to the trace

from langsmith import traceable

@traceable(
    run_type="chain",
    name="email-classifier",
    tags=["prod", "v2.1"],
    metadata={
        "user_id": "usr_123",
        "feature_flag": "gpt4o"
    }
)
def classify_email(text: str) -> str:
    return agent.run(text)

Using the @traceable decorator for custom functions

from langsmith import traceable
from langchain_openai import ChatOpenAI

@traceable  # Aparece como nó separado na árvore do trace
def my_preprocessing(text: str) -> str:
    """Limpeza e normalização do input."""
    text = text.strip().lower()
    return text[:2000]  # Trunca textos muito longos

@traceable
def my_postprocessing(response: str) -> dict:
    """Estrutura a resposta final."""
    return {"answer": response, "confidence": 0.9}

# Estas funções agora aparecem na hierarquia do trace LangSmith

Identifying the problematic LLM call

Techniques to isolate which step caused failure, incorrect response or high cost.

Symptom 1: Incorrect response

Inspect the prompt sent to LLM. Was the context correct? Was the instruction clear?

→ Click on the LLM node
→ Tab "Input"
→ See the System Message
→ See the injected context

Symptom 2: High cost

Which call consumed the most tokens? Unnecessarily long context?

→ Sort nodes by tokens
→ Identify the largest
→ Check that all
messages are necessary

Symptom 3: Timeout

Which node took the longest? Slow external tool? Overloaded LLM?

→ See latency per node
→ Identify bottleneck
→ Tool or LLM?
→ Add local timeout

Trace replay and comparison

One of LangSmith's superpowers is comparing two traces side by side to see exactly what changed between prompt versions:

from langsmith import Client

client = Client()

# Listar traces de um projeto
runs = client.list_runs(
    project_name="meu-agente",
    filter='and(eq(status, "error"), gt(total_tokens, 1000))',
    limit=10
)

for run in runs:
    print(f"ID: {run.id}")
    print(f"Status: {run.status}")
    print(f"Tokens: {run.total_tokens}")
    print(f"Latência: {run.end_time - run.start_time}")
    print("---")

# Acessar input/output de uma run específica
run = client.read_run("run-id-aqui")
print(run.inputs)   # Input exato
print(run.outputs)  # Output retornado

Quality metrics for agents

Latency, cost and success rate: the three metrics that every agent in production needs to monitor.

Latency

⏱️
P50 (median) < 5s
P95 < 15s
P99 < 30s

Monitor percentiles, not just averages. An average of 5s can hide P99 of 60s.

Cost

💰
By task < $0.05
Per user/day < $0.50
Anomaly alert 3× average

Set alerts for tasks that cost 3× above average — indicates infinite loop or prompt.

Success Rate

✅
Excellent > 95%
Acceptable 85–95%
Review agent < 85%

Include silent failures: Responses outside the expected format are also failures.

Metrics Dashboard with LangSmith SDK

from langsmith import Client
from datetime import datetime, timedelta
import statistics

client = Client()

# Coleta runs das últimas 24h
runs = list(client.list_runs(
    project_name="meu-agente",
    start_time=datetime.now() - timedelta(hours=24)
))

# Calcula métricas
latencias = [
    (r.end_time - r.start_time).seconds
    for r in runs if r.end_time
]
custos = [r.total_cost for r in runs if r.total_cost]
sucessos = sum(1 for r in runs if r.status == "success")

print(f"Total de runs: {len(runs)}")
print(f"Success rate: {sucessos/len(runs)*100:.1f}%")
print(f"Latência mediana: {statistics.median(latencias):.1f}s")
print(f"Latência P95: {sorted(latencias)[int(len(latencias)*0.95)]:.1f}s")
print(f"Custo médio: ${statistics.mean(custos):.4f}")
print(f"Custo total: ${sum(custos):.2f}")

Structured logging for production

Record each tool call with input, output and cost — and combine with business metrics.

Unstructured logging (avoid)

print("Iniciando agente...")
print("Chamando search tool")
print(f"Resultado: {result}")
print("Agente terminou")

# - Impossível filtrar por user_id
# - Sem timestamp preciso
# - Sem custo ou tokens

Structured logging (do)

import structlog, time

logger = structlog.get_logger()

logger.info("tool_call",

Complete callback handler for production

import time, structlog
from langchain_core.callbacks import BaseCallbackHandler
from langchain_core.outputs import LLMResult

logger = structlog.get_logger()

class ProductionCallbackHandler(BaseCallbackHandler):
    def __init__(self, user_id: str, session_id: str):
        self.user_id = user_id
        self.session_id = session_id
        self._tool_start_times = {}
        self._llm_start_times = {}

    def on_llm_start(self, serialized, prompts, run_id, **kwargs):
        self._llm_start_times[str(run_id)] = time.time()
        logger.info("llm_start",
            model=serialized.get("name"),
            prompt_chars=sum(len(p) for p in prompts),
            user_id=self.user_id
        )

    def on_llm_end(self, response: LLMResult, run_id, **kwargs):
        duration = time.time() - self._llm_start_times.get(str(run_id), time.time())
        usage = response.llm_output.get("token_usage", {})
        logger.info("llm_end",
            duration_ms=int(duration * 1000),
            prompt_tokens=usage.get("prompt_tokens", 0),
            completion_tokens=usage.get("completion_tokens", 0),
            total_tokens=usage.get("total_tokens", 0),
            user_id=self.user_id,
            session_id=self.session_id
        )

    def on_tool_start(self, serialized, input_str, run_id, **kwargs):
        self._tool_start_times[str(run_id)] = time.time()
        logger.info("tool_start",
            tool_name=serialized.get("name"),
            input_preview=str(input_str)[:300],
            user_id=self.user_id
        )

    def on_tool_end(self, output, run_id, **kwargs):
        duration = time.time() - self._tool_start_times.get(str(run_id), time.time())
        logger.info("tool_end",
            duration_ms=int(duration * 1000),
            output_preview=str(output)[:300],
            user_id=self.user_id,
            session_id=self.session_id
        )

    def on_tool_error(self, error, run_id, **kwargs):
        logger.error("tool_error",
            error=str(error),
            user_id=self.user_id,
            session_id=self.session_id
        )

# Uso
handler = ProductionCallbackHandler(user_id="usr_123", session_id="sess_456")
result = executor.invoke(
    {"input": "Analise este contrato"},
    config={"callbacks": [handler]}
)
Do vs Avoid

✓Good logging practices

  • ✓Log input and output truncated (max. 300 chars)
  • ✓Always include user_id, session_id and trace_id
  • ✓Record duration in milliseconds
  • ✓Use structured JSON format (structlog, loguru)
  • ✓Capture errors with full stack trace

✗Logging anti-patterns

  • ✗Log the full content of each message
  • ✗Ignore tool errors (only log in successfully)
  • ✗Log sensitive personal data (PII)
  • ✗Use print() without session context
  • ✗Log in synchronously to hot path
