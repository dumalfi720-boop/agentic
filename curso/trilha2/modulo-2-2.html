💾 Short Term Memory

📝 Conversation Buffer

The conversation buffer is the simplest form of memory in agents: amessage list(user + assistant) which is passed to LLM with each new call. Without it, each interaction starts from scratch — the agent doesn't know what happened before.

📋 Buffer Structure

# Message structure (OpenAI/Anthropic standard)

messages = [

{"role": "system", "content": "You are an assistant..."},

{"role": "user", "content": "What is the capital of Brazil?"},

{"role": "assistant", "content": "The capital is Brasília."},

{"role": "user", "content": "And what is the biggest city?"},

]

# LLM sees ALL messages and responds with context

Advantages of Simple Buffer

  • ✓Trivial implementation — a Python list
  • ✓No loss of information
  • ✓LLM has access to the entire conversation
  • ✓Easy to debug — you see exactly what LLM sees

Disadvantages

  • ✗Grows indefinitely — with no natural limit
  • ✗Cost grows linearly with conversation
  • ✗Eventually the context window pops
  • ✗Does not persist between sessions by default

💡When to use simple buffering

Ideal for short conversations (less than 20 turns) or prototypes. In production with long conversations, you need a management strategy. The simple buffer is the starting point — not the destination.

🪟Sliding Window

The sliding window only keeps theN most recent messagesin the buffer. When the conversation reaches the limit, the oldest messages are discarded (FIFO). The context is stable and the cost is predictable.

🖼️ Viewing the sliding window

Shift 1-5:
M1 M2 M3 M4 M5
← full window (5 msg)
Turn 6:
M1 M2 M3 M4 M5 M6
← M1 dropped, M6 added

📊 Window sizes by use case

5-10 messages

Simple Chatbots, FAQ. Conversations that do not accumulate complex context.

20-50 messages

Work assistants. Long conversations where recent history matters.

100+ messages

Research agents or long projects. Combine with compression.

⚠️ The pure window problem

Sliding window discards potentially critical information. If the user mentioned your name at the beginning of the conversation and it was discarded, the agent "forgets" it and may treat you as unknown. For critical context, use compression or long-term memory.

TOPIC 3
3

📋 Automatic summary

Instead of discarding old messages, automatic summary uses aLLM to compress old historyin dense text. The summary replaces the original messages, preserving the essence without losing critical information.

1

Trigger: context almost full

When the buffer exceeds N tokens (e.g. 80% of the context window), the system detects and activates summarization before making the next call to the agent.

2

Summarization call

A small LLM (Claude Haiku, GPT-4o-mini) receives the old messages and the prompt: "Summarize this conversation while preserving important facts, decisions made, and relevant context."

3

Buffer substitution

Old messages are replaced with a single "system" or "assistant" message with the summary. Recent messages are kept intact.

ConversationSummaryMemory on LangChain

from langchain.memory import ConversationSummaryMemory

from langchain_openai import ChatOpenAI

# Small LLM to summarize (save cost)

summary_llm = ChatOpenAI(model="gpt-4o-mini")

memory = ConversationSummaryMemory(

llm=summary_llm,

max_token_limit=1000# Summarize when 1k tokens pass

)

💥 The Long Context Problem: Lost in the Middle

Research shows that LLMs haveDifficulty using information from the context. Information at the beginning and end is processed better than information at the center of a very long context — the "lost in the middle" phenomenon.

📊 Evidence of the problem

  • •LLMs with 100k token context perform worse in the center vs. start/end of context (Stanford, 2024)
  • •For information retrieval, models with smaller and more relevant context outperform models with huge and noisy context
  • •The effect is more pronounced in smaller models and in tasks that require integration of multiple pieces of information.

Implications for design

  • →Place critical information at the beginning or end of the context
  • →Detailed system prompt at the beginning is always well processed
  • →Final statements ("respond in JSON") have high weight
  • →Don't trust that the LLM will "find" information in the middle of a long document

Practical solutions

  • ✓RAG with focused retrieval: fewer chunks, more relevant
  • ✓Reranking to put more relevant chunks first
  • ✓History summarization before injecting new documents
  • ✓Test with contexts of varying size before production

🗜️ Advanced compression strategies

In addition to general summarization, there are more precise compression strategies:extraction of key facts, entity memory and compression by importance. Each preserves a different type of information.

Entity Memory

Maintains a dictionary of entities mentioned in the conversation with their attributes. "John is a senior Python developer at company X."

entities = {
"John": "Senior Python Dev, Company X",
"Project": "Payments API, deadline Q2"
}

Fact List Compression

Extracts factual statements from conversation in a structured list. Denser than narrative summary — fewer tokens for the same information.

facts = [
"User prefers Python over JavaScript",
"Project budget: R$50k",
"Deploy on AWS, not Azure"
]

💡Hybrid Memory: the best of both worlds

The most robust strategy combines:Entity Memory(facts about people/entities),Fact List(decisions and preferences) +Sliding Window(N full recent messages). LLM receives the entities dict + list of facts + recent messages. Efficient context without loss of critical information.

TOPIC 6
6

🔴 When short memory fails

Short memory bugs are difficult to diagnose because the agentseems to workbut produces inconsistent results over time. Knowing failure patterns speeds up diagnosis.

🔴

Symptom: "Agent contradicts previous response"

Cause:Sliding window discarded the message where the agent stated something. Now, without seeing what he said before, he claims the opposite.

Solution:Increase window or use Entity Memory for facts that cannot be discarded.

🟡

Symptom: "Agent repeats question already asked"

Cause:Buffer silently truncated by framework (no error!). LLM does not see the answer that the user has already given.

Solution:Log buffer size on each call. Detect truncation and handle explicitly.

🔵

Symptom: "Agent ignores system prompt instruction after many switches"

Cause:Lost in the middle — the system prompt got buried in the middle of the huge context and LLM is not following it.

Solution:Reinject critical instructions periodically. Move formatting instructions to the end of the context.

🧪 How to test your agent's memory

  1. Mention specific information in message 1 (e.g. "my name is Carlos")
  2. Make 15+ exchanges on another topic
  3. Ask: "What do you call me?"
  4. If the agent doesn't know, the memory was lost mid-conversation
  5. Repeat with different window sizes to find the threshold
