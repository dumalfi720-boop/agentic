🧩 Why LangChain Exists

Building an agent from scratch means implementing the tool calling loop, managing message history, parsing LLM responses and handling errors — all from scratch. LangChainabstract this boilerplateinto reusable components so you can focus on the agent logic.

What LangChain abstracts for you

Without LangChain (from scratch)

  • • Implement the while tool_calls loop
  • • Parse API JSON tool calls
  • • Manage message list manually
  • • Implement retry on failures
  • • Manage memory and context
  • • Handle errors from each provider differently

With LangChain

  • ✓ AgentExecutor/LCEL takes care of the loop
  • ✓ Automatic tool calling
  • ✓ Memory objects for history
  • ✓ Configurable retry
  • ✓ Memory integrated into the prompt
  • ✓ Unified interface for OpenAI/Anthropic/etc.

When LangChain helps

Rapid prototyping, integration with many LLMs, use of ready-made ecosystem integrations (vectorstores, loaders, tools)

When LangChain gets complicated

Very simple agents (1-2 tools), when you need full control of the loop, when abstraction becomes accidental complexity

Alternatives

LlamaIndex (RAG-first), smolagents (Hugging Face, simplest), Pydantic AI (type-safe), LangGraph (agent graphs)

🆚 AgentExecutor vs. LCEL

LangChain has two agent building paradigms:AgentExecutor(legacy API but still dominant in tutorials) and theLCEL(LangChain Expression Language — the new, more flexible way). You will encounter both in practice.

AgentExecutor (old way)

from langchain.agents import create_react_agent, AgentExecutor

from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")

agent = create_react_agent(llm, tools, prompt)

executor = AgentExecutor(agent=agent, tools=tools)

result = executor.invoke({"input": "Question"})

  • + Easy to understand for beginners
  • + Lots of material on the internet
  • - API considered legacy
  • - Less observability by default

LCEL (new way)

from langchain import hub

from langchain.agents import create_react_agent

from langchain_core.runnables import RunnableWithMessageHistory

prompt = hub.pull("hwchase17/react")

agent = create_react_agent(llm, tools, prompt)

# Composition with operator |

chain = prompt | llm.bind_tools(tools) | parser

  • + Declarative composition with operator |
  • + Native streaming
  • + Better integration with LangSmith
  • - Greater learning curve

💡 Which one to use?

To learn: start with AgentExecutor (simplest). For new production: prefer LCEL (more observable and flexible). For legacy projects: don't migrate for the sake of migrating — only migrate when there is a clear win.

💾 Adding memory to the agent

Without memory, each agent invocation starts from scratch. For conversational agents, you needpersist history between calls. LangChain has in-memory (current session) and persistent (between sessions with SQLite or Redis) memory.

Complete implementation with RunnableWithMessageHistory

from langchain_community.chat_message_histories import SQLChatMessageHistory

from langchain_core.runnables.history import RunnableWithMessageHistory

# Persistence in SQLite — memory between sessions!

def get_session_history(session_id: str):

return SQLChatMessageHistory(session_id, "sqlite:///memoria.db")

agent_with_memory = RunnableWithMessageHistory(

agent_executor,

get_session_history,

input_messages_key="input",

history_messages_key="chat_history"

)

# session_id allows multiple users with separate memories

response = agent_with_memory.invoke(

{"input": "Hello! My name is Ana."},

config={"configurable": {"session_id": "usuario-123"}}

)

# On the next call, the agent already knows that the user's name is Ana!

Memory Persistence Options

In-Memory

ChatMessageHistory — in the current Python session only. You lose everything when the process restarts.

SQLite

SQLChatMessageHistory — persists between sessions. Ideal for development and small scale.

Redis

RedisChatMessageHistory — scales horizontally. Ideal for production with multiple workers.

🛠️ Tools with LangChain

LangChain has two ways to create tools: the decorator@tool(simple, generates schema from the docstring) andStructuredTool(for tools with multiple typed parameters). The docstring becomes the tool description in the JSON schema.

@tool decorator — simple

from langchain.tools import tool

@tool

def fetch_time(city: str) -> str:

"""Returns the current weather forecast for

a city. Use when the user asks

about climate or temperature."""

return f"25°C in {city}, sunny"

# Automatically generated schema!

StructuredTool — multiple parameters

from langchain.tools import StructuredTool

from pydantic import BaseModel

class BuscaInput(BaseModel):

city: str

days: int = 1

tool = StructuredTool.from_function(

func=search_time_fn,

📧 Project: Email Assistant

The complete project: an agent thatreads emails, sorts by priority and drafts responsesfor urgent ones. Integrates memory, structured output and customized tools into a functional and extensible system.

Complete commented code

from langchain.agents import AgentExecutor, create_react_agent

from langchain_openai import ChatOpenAI

from langchain.tools import tool

from pydantic import BaseModel

from langchain import hub

INBOX = [

{"id": 1, "from": "joao@empresa.com", "subject": "URGENT: server crashed",

"body": "Production has been off since 2pm. Customers reporting error 500."},

{"id": 2, "from": "newsletter@startup.com", "subject": "10 productivity tips",

"body": "Check out best practices for..."},

]

class EmailClassification(BaseModel):

category: str# "urgent", "normal", "spam"

priority: int#1-5

requires_response: bool

@tool

def ler_inbox() -> str:

"""Lists all emails in the inbox. Use to see available emails."""

return str([{"id": e["id"], "subject": e["subject"], "from": e["de"]} for e in INBOX])

@tool

def ler_email(email_id: int) -> str:

"""Reads the full content of an email by ID."""

email = next((e for e in INBOX if e["id"] == email_id), None)

return str(email) if email else "Email not found"

@tool

def draft_response(email_id: int, tone: str = "professional") -> str:

"""Draft a response to the email. Tone can be: professional, friendly, urgent."""

email = next((e for e in INBOX if e["id"] == email_id), None)

if not email: return "Email not found"

return f"Draft for {email['from']}: [Custom text based on '{email['subject']}' in tone {tone}]"

tools = [read_inbox, read_email, draft_response]

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.3)

prompt = hub.pull("hwchase17/react")

agent = create_react_agent(llm, tools, prompt)

executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

result = executor.invoke({

"input": "Check my inbox, sort emails and draft responses to urgent ones."

})

🐛 Debugging without LangSmith

LangSmith is great, but not always available or configured. Withverbose=True and custom callbacks, you have complete visibility of what the agent is doing right at the terminal — without external dependencies.

Callback handler for full debugging

from langchain.callbacks.base import BaseCallbackHandler

from datetime import datetime

class DebugCallbackHandler(BaseCallbackHandler):

def on_llm_start(self, serialized, prompts, **kwargs):

print(f"\n[LLM START] {datetime.now().strftime('%H:%M:%S')}")

print(f"Input estimated tokens: ~{len(prompts[0])//4}")

def on_llm_end(self, response, **kwargs):

output = response.generations[0][0].text

print(f"[LLM END] Output: {output[:200]}...")

def on_tool_start(self, serialized, input_str, **kwargs):

print(f"\n[TOOL CALL] {serialized['name']}")

print(f"Input: {input_str}")

def on_tool_end(self, output, **kwargs):

print(f"[TOOL RESULT] {str(output)[:300]}")

executor = AgentExecutor(

agent=agent, tools=tools,

callbacks=[DebugCallbackHandler()]# ← add here

)

verbose=True — the simplest

Addverbose=Trueto the AgentExecutor to see the complete Thought/Action/Observation trace in the console:

executor = AgentExecutor(

agent=agent, tools=tools,

verbose=True# ← that's enough

)

Identifying the failed step

  1. Read verbose output from end to beginning
  2. Find the last "Thought" before the error
  3. See which "Action" was called and with which inputs
  4. See "Observation" (tool result)
  5. Decide: bug in the prompt, in the tool or in the logic?
