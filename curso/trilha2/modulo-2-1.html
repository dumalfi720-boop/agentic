🔍 Model selection by capacity and cost

The choice of model isfirst architectural decisionof any agent. There is no universally better model — there is the right model for each use case, considering reasoning capacity, speed, cost and support for function calling.

🗺️Model map in 2026

Frontier Models (planning)

  • •Claude Opus 4.6— better reasoning, ideal for planner
  • •GPT-4o— excellent function calling, mature ecosystem
  • •Gemini Pro— native long context (1M tokens)

Small models (execution)

  • •Claude Haiku— fastest and cheapest in the Anthropic family
  • •GPT-4o-mini— 15x lower cost than GPT-4o
  • •Mistral/Llama— open-source, self-hosted, zero API cost

💡Selection golden rule

Use thecheapest model that still gets it rightfor each step. Start testing with the smallest model; Only escalate to frontier when the child demonstrates specific and documented failure.

📐 Context window in practice

The context window is theLLM working memory. Everything that LLM can consider when generating a response fits there: the system prompt, the conversation history, the tool schemas, the tool results and the documents injected via RAG. Exhausting the context breaks the agent.

What occupies the context

  • •System prompt: 500–2000 tokens typically
  • •Conversation history: grows with each turn
  • •Tool schemas: ~200–500 tokens per tool
  • •Tool results: variable, can be huge
  • •RAG documents: 500–3,000 tokens per chunk

Estimation of tokens per content

  • ≈1 token ~ 4 English characters
  • ≈1 token ~ 3 characters in Portuguese
  • ≈1 A4 page of text ~ 500 tokens
  • ≈1 medium Python file ~ 1,000 tokens
  • ≈1 image (GPT-4V) ~200–1,500 tokens

📊Context windows by model

  • Claude 3.5 Sonnet:200,000 tokens — fits ~300 pages of text
  • GPT-4o:128,000 tokens — fits ~190 pages
  • Gemini Pro 1.5:1,000,000 tokens — fits ~1,500 pages
  • Llama 3.1 (70B):128,000 tokens — but self-hosted with no API cost

⚠️Context overflow: the silent bug

When the context overflows, the API returns error 400 and the agent crashes. The problem: many frameworks silently truncate the context instead of throwing an error. LLM responds but without seeing part of the history — producing inconsistent responses without warning.

🌡️ Temperature and creativity

Temperature is the parameter that controls thedegree of randomnessin the LLM answers. At 0.0, the model always chooses the most likely token — producing deterministic answers. At 1.0 or more, less likely choices gain weight, generating more varied and creative responses.

0.0

Deterministic

Ideal for: code, SQL, JSON, structured analysis, data extraction. The same input always generates the same output.

0.3–0.5

Balanced

Best for: support responses, rating, summaries, most agent use cases in production.

0.7–1.0

Creative

Ideal for: creative text generation, brainstorming, marketing copy variations, natural dialogue.

✓ What TO DO

  • ✓Use temp=0 for tools that generate code or SQL
  • ✓Document the temperature used in each agent
  • ✓Test with different temps before going to production
  • ✓Use seeds together with low temperature for reproducibility

✗ What NOT to do

  • ✗High temperature for agent processing financial data
  • ✗Forgetting to set temperature and using the provider default
  • ✗Assuming that temperature 0 is always best
  • ✗Change temperature in production without regression testing

💡top_p: the other randomness parameter

top_p (nucleus sampling) is an alternative to temperature. Instead of scaling all probabilities, it limits choices to tokens that sum to probability p.Rule of thumb: only change one at a time. Changing temperature AND top_p at the same time makes it unpredictable which effect dominates.

💰 Cost per token in practice

Each call to LLM has a measurable cost:input tokens + output tokens. In a looping agent, these costs multiply with each iteration. An agent that appears cheap in testing may turn out to be cost-prohibitive in real-volume production.

📊 Example of actual cost per task

Email Parsing Agent with GPT-4o, 5 iterations per email:

System prompt (1,000 tokens × 5 iterations) = 5,000 input tokens
Conversation history (grows by 500 tokens/turn) = ~7,500 tokens input
LLM Answers (300 tokens × 5 iterations) = 1,500 tokens output
Total by email with GPT-4o ~$0.047 per email
Same agent with GPT-4o-mini ~$0.004 per email (12× cheaper)

Strategies to reduce cost

  • 1.Prompt cache: repeats the same system prompt → 50-90% discount
  • 2.Small template for simple steps (sort, format)
  • 3.Reduce iterations with Plan-and-Execute
  • 4.History compression before context grows too much

Mandatory budget limits

  • •Set max_tokens per call to limit output
  • •Set max_iterations in the agent loop
  • •Use accumulated token counter and stop before popping
  • •Monitor cost per task and alert 2× deviations

⚖️ Small vs large models

The most powerful agent cost optimization strategy isuse different templates for different steps. The big model makes reasoning difficult; the small model performs the repetitive steps. The cost difference can be 50-100×.

🧠

Large Template — For Planner

Claude Opus, GPT-4o, Gemini Pro

Responsible for understanding the objective, breaking it down into steps, choosing which tools to use and evaluating whether the result achieved the objective. Called a few times per task (1-3 times).

⚡

Small Model — For the Executor

Claude Haiku, GPT-4o-mini, Mistral 7B

Performs each step of the plan: formatting data, calling tools, parsing results, classifying entities. Called many times (N times per plan step). That's where you save.

💡When small model fails

Small models may fail in: complex multi-step reasoning, following long system prompt instructions, JSON with deep nested structure, and tool selection when descriptions are ambiguous. In these cases, scale up to the larger model occasionally — not all the time.

🧭 How the LLM decides the next action

The LLM decision process is fundamentally anext-token prediction tasksophisticated. But from the agent's perspective, the LLM looks at the current state of the context and decides: should I respond to the user, call a specific tool, or ask for more information?

The 4-step decision process

1

Read the full context

System prompt (paper and instructions), history (what happened), tool schemas (what you can do) and result of the last tool (what you know now).

2

Assess the state of the objective

Do you have enough information to respond? Was the objective achieved? Is there ambiguity that needs to be resolved?

3

Choose between 3 options

A) Respond to the user (objective achieved) · B) Call a tool (need more information or action) · C) Request clarification (objective ambiguous).

4

Generates the structured response

If tool call: generates JSON with tool name and parameters. If answer: generates text. The runtime reads this output and takes the corresponding action.

🔑What makes the decision better

The quality of the LLM decision directly depends on 3 elements that you control:

  • 1.Clear system prompt:explicitly defines the role, objective and when to use each tool
  • 2.Accurate tool descriptions:LLM uses the description field to decide when to call each tool
  • 3.Stop sequences:specific tokens that force LLM to stop and let the runtime act
