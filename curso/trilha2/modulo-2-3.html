🧠 The problem of amnesia

Short memory (buffer) exists within a session. When the session ends, the agentforget everything. For a customer support agent, this means asking the same issue every time. For a research agent, it means redoing work already done. Long-term memory solves this.

📚

Episodic Memory

What happened in past interactions. "Last week the user reported bug X and it was resolved like this..."

🗂️

Semantic Memory

General and specialized knowledge. "The API documentation says that the /orders endpoint accepts these parameters..."

⚙️

Procedural Memory

How to do things. "The onboarding process follows these 7 steps in the correct order..."

RAG vs. Fine-tuning: when to use each one

Use RAG when:

  • •Data changes frequently (documents, prices, cases)
  • •You need to cite the source of the information
  • •Data volume is large (GB of documents)

Use Fine-tuning when:

  • •Do you want to change the style/tone of the LLM
  • •Knowledge is stable and does not change
  • •You have thousands of curated Q&A examples

🗃️ Vector banks: FAISS and Chroma

Vector banks specialize in storing and fetchinghigh dimensional vectors(embeddings). Instead of searching by exact keyword, they find documents semantically similar to your query in milliseconds.

FAISS (Goal)

  • +Extremely fast (search millions of vectors in ms)
  • +Runs locally, without a server, without API costs
  • +Supports GPU for massive scale
  • -No native metadata filtering
  • -Low-level interface — more code to use

Best for: local prototyping, large datasets, no server dependency

Chroma

  • +Simple, Pythonic API — lower learning curve
  • +Native metadata filtering — search by date, category, etc.
  • +Automatic persistence in SQLite
  • -Slower than FAISS on very large datasets
  • -Limited scalability without Chroma Cloud

Best for: smaller/medium projects, need for metadata filtering, starting a project

💻 Creating a vectorstore with Chroma (5 lines)

from langchain_chroma import Chroma

from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

vectorstore = Chroma(

persist_directory="./my_bank",

embedding_function=embeddings

)

# Add documents

vectorstore.add_texts(texts=["doc1...", "doc2..."])

# Search similarity

results = vectorstore.similarity_search("my query", k=3)

🔢 Embeddings in practice

Embeddings transform text into numeric vectors wheretexts with similar meaning are close together in space. "Dog" and "dog" are close; "dog" and "plane" are far apart. It's not magic — it's geometry applied to meaning.

🗺️ Intuition without math

Imagine a map where each word/sentence has a position (coordinates). The embedding model learned to place similar texts close to each other:

Region "Finance"

share, dividend, stock market, investment, income

Region "Animals"

dog, cat, dog, feline, canine

Region "Code"

function, loop, variable, debug, compile

Embedding models in 2026

  • text-embedding-3-small— 1536 dims, $0.02/1M tokens, great value for money
  • text-embedding-3-large— 3072 dims, $0.13/1M tokens, more accurate
  • voyage-3(Anthropic) — excellent for code and technical documentation
  • nomic-embed— open-source, self-hosted, free

Critical chunking parameters

  • chunk_size:size of each piece of text (256-1024 tokens)
  • chunk_overlap:overlap between chunks (10-20% of chunk_size)
  • separators:where to cut — paragraph, sentence or word
  • Rule:smaller chunk = more accurate, larger chunk = more context

🔄 Complete RAG Pipeline

RAG (Retrieval-Augmented Generation) is the technique of giving the LLM access to an external knowledge base at the time of generation. The pipeline has4 distinct phases: index, retrieve, inject and generate.

1

Indexing (done once)

Load documents → divide into chunks → generate embedding for each chunk → store in the vector database. Paid once, permanent benefit until documents change.

2

Recovery (for each query)

Receives the user's question → generates question embedding → searches for the K most similar chunks in the vector database by cosine similarity.

3

Context injection

The retrieved chunks are added to the prompt as context: "Based on the following documents: [chunks]. Answer: [user question]."

4

Generation

LLM responds using the injected context. He can cite sources, be more precise and not "hallucinate" about information that is explicitly in the documents.

💻 Full RAG on LangChain (working code)

from langchain.chains import RetrievalQA

from langchain_chroma import Chroma

from langchain_openai import OpenAIEmbeddings, ChatOpenAI

from langchain.text_splitter import RecursiveCharacterTextSplitter

from langchain_community.document_loaders import TextLoader

# 1. Indexing

loader = TextLoader("documentacao.txt")

docs = loader.load()

splitter = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=50)

chunks = splitter.split_documents(docs)

vectorstore = Chroma.from_documents(chunks, OpenAIEmbeddings())

#2-4. Recovery + Injection + Generation

qa = RetrievalQA.from_chain_type(

llm=ChatOpenAI(model="gpt-4o-mini"),

retriever=vectorstore.as_retriever(search_kwargs={"k": 3})

)

response = qa.invoke("What is the return policy?")

🐛 Bad retrieval debug

"RAG doesn't work" is the most common bug in LLM projects. Butwhich part failed?It could be a very small chunk, insufficient top-k, wrong threshold, or low quality embedding for the domain. Each cause has a specific solution.

🟡 Problem: chunks too small (256 tokens)

Symptom:The chunk contains the keyword but lacks the context to be useful to LLM.

Solution:Increase to 512-1024 tokens. Use chunk_overlap of 10-20% to avoid losing phrases when cutting.

🔴 Problem: top-k too low (k=1 or k=2)

Symptom:The chunk with the response exists but is not among the first returned.

Solution:Increase to k=5 or k=10. Use MMR (Maximal Marginal Relevance) for diversity.

🔵 Problem: Bad embedding for the domain

Symptom:Semantically related documents do not appear as similar.

Solution:Use specialized embedding templates (voyage-3 for code, multilingual-e5 for PT-BR).

💡 Evaluating RAG with RAGAS

RAGAS is an open-source framework that evaluates your RAG pipeline on 4 metrics:Faithfulness(answer is in the chunks?),Answer Relevance(Is the answer relevant to the question?),Context Recall(were relevant chunks recovered?) andContext Precision(are recovered chunks all relevant?). Use before going to production.

💸 Vector memory cost

A RAG system has two distinct costs:indexing(embedding of all documents — paid once) andsearch(query embedding for each interaction — always paid). At scale, difference matters.

📊 Example of real cost

Knowledge base: 1,000 documents (500 average tokens each) = 500,000 tokens

Full indexing (text-embedding-3-small) $0.01 (single)
1,000 searches/day (query ~20 tokens) $0.0004/day
Total monthly cost (embeddings) ~$0.01 indexing + $0.012 search = $0.022

The actual cost of RAG is dominated by the cost of LLM to generate the response, not the embeddings.

Self-hosted (FAISS/Chroma local)

  • ✓Zero vector storage cost
  • ✓Data stays in your infrastructure
  • ✗You manage backup and availability
  • ✗Scalability limited by RAM memory

Hosted (Pinecone, Weaviate Cloud)

  • ✓Unlimited automatic scalability
  • ✓Guaranteed availability SLA
  • ✗Cost: Pinecone Serverless ~$0.033/1M reads
  • ✗Data in the provider's cloud
