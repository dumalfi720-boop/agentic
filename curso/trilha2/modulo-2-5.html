💭 Chain-of-Thought (CoT)

Chain-of-Thought is the technique for doing the LLMexplain your reasoning step by stepbefore giving the final answer. Instead of going straight to the result, the model “thinks out loud” — and this significantly improves quality in complex problems.

Comparison: no CoT vs. with CoT

NO CoT

Prompt: "If John has 3 times as many apples as Mary, and Mary has 8, how many apples does John have?"

→ Answer: "John has 11 apples."

✗ Wrong (3×8 = 24, not 11)

WITH CoT ("think step by step")

Same prompt + "Think step by step before responding."

→ "Mary has 8 apples. John has 3 times as many, so John has 3 × 8 = 24 apples."

✓ Correct

Zero-shot CoT

Add "Think step by step" to the prompt. Works on most Frontier models without examples.

Few-shot CoT

Include 2-3 examples of step-by-step reasoning in the prompt. More accurate, more input tokens.

CoT in Tool Selection

The LLM explains why it chose a specific tool before calling it. Improves choice by 30%.

💡 Where CoT impacts the most

Mathematics, logic, code, multi-step reasoning and critical analysis benefit most from CoT. For simple fact retrieval or straightforward classification tasks, CoT adds cost without real benefit.

⚡ ReAct: Reason + Act

ReAct is the default pattern for most agents in production. He interspersesThought → Action → Observationin a loop until the goal is reached. LLM "thinks out loud", chooses an action (tool), sees the result and repeats.

Trace of a real ReAct agent

Thought:

The user wants to know the current price of PETR4. I need to get this information in real time. I'm going to use the stock quote tool.

Action: fetch_quotation

Input: {"ticker": "PETR4", "exchange": "B3"}

Observation:

PETR4: R$38.42 (+1.2%) at 3:32 pm. Volume: 12.5M shares.

Thought:

I have the necessary information. I can respond to the user directly with the current quote.

Final Answer:

PETR4 is trading at R$38.42 today, up 1.2%.

Advantages of ReAct

  • ✓Flexible — adapts the next step based on the current result
  • ✓Auditable reasoning — you see exactly what the agent thought
  • ✓Simple to implement — it's the AgentExecutor pattern

Limitations

  • ✗Cost grows with each iteration (expensive model with each thought)
  • ✗Can go into infinite loop if misconfigured
  • ✗Not ideal for tasks with a predictable plan (unnecessarily expensive)

📋 Plan-and-Execute

Plan-and-Execute separates the agent into two: oneplanner(Large LLM, called once) that breaks down the task into steps, and aExecutor(small LLM, called step by step) that executes each item. Reduces costs by up to 90% for long tasks.

Plan-and-Execute Architecture

1. Planner (Claude Opus — 1 call)

Input: "Search the 3 biggest players in

cloud computing and compare prices"

Output: [

"1. Search for information about AWS",

"2. Search for information about Azure",

"3. Search for information about GCP",

"4. Compare Standard VM Prices",

"5. Create comparative table"

]

2. Executor (GPT-4o-mini — 5 calls)

Step 1: fetch_web("AWS pricing ec2")
Step 2: fetch_web("Azure VM pricing")
Step 3: fetch_web("GCP compute pricing")
Step 4-5: format_table(data)

🔁 Reflection pattern

In the Reflection pattern, the agent generates a response and then uses a second LLM (theCritic) to criticize, identify errors and suggest improvements. The improved response goes through a new round of criticism until it reaches sufficient quality.

A

Actor — generates the response

Receives the objective and produces a first complete response. Doesn't try to be perfect — produces quickly and delegates criticism.

C

Critic — evaluates and scores

Receives the objective + response from the Actor. Identifies factual, logical, completeness or stylistic errors. Produces specific, actionable feedback.

R

Reviewer — improves based on feedback

Actor receives feedback from the Critic and reviews the response. Cycle repeats until Critic gives sufficient score or maximum number of iterations.

When Reflection pays off

  • ✓Code generation where bugs are costly
  • ✓Creative writing where quality is different
  • ✓Critical analysis where imprecision has consequences
  • ✓Generation of executive reports

Control parameters

  • max_iterations:usually 2-3 (cost increases linearly)
  • quality_threshold:Minimum Critic rating to stop
  • Critical model:may be smaller than the Actor (save)
  • Critical prompt:be specific about quality criteria

🗺️ When to use each pattern

There is no universally better standard. The choice depends on4 dimensions: number of task steps, predictability of the plan, cost tolerance and quality requirement. The table below summarizes the decision.

Standard Steps Predictability Relative Cost Best for
CoT 1 High 1-2× Analysis, mathematics, point logic
ReAct 2-10 Low N×model_cost Exploration with tools, dynamic search
Plan+Execute 5-20 High 10-90× smaller Long, predictable pipelines
reflection 1 (w/ revisions) High 3-5× Critical quality, code, professional writing

💡 Matching patterns

Standards are not mutually exclusive. Powerful example:Plan-and-Execute + Reflection— Planner creates the plan (internal CoT), Executor executes each step (local ReAct by step), and Reflexion evaluates the final result before delivering. Intermediate cost, maximum quality.

TOPIC 6
6

💰 Cost vs. quality by default

The pattern choice hasdirect financial impactin production. An agent with thousands of daily executions can have a difference of 10× in the monthly cost depending on the chosen pattern. Understanding the trade-off is part of the architecture.

📊 Monthly cost analysis (10,000 tasks/day)

CoT (GPT-4o, 1 call) ~$750/month
ReAct (GPT-4o, 5 medium iterations) ~$3,750/month
Plan+Execute (Opus plan + mini execute) ~$900/month
Reflection (GPT-4o, 3 iterations) ~$11,250/month

Practical architectural decision rules

1. If the plan is always the same (same task type), use Plan-and-Execute. The planner can be called once per template, not per task.
2. If the quality has direct financial value (executive report, commercial proposal), Reflexion with 2 iterations pays the extra cost.
3. Start with ReAct to validate the issue. Then optimize for Plan-and-Execute when you have clear execution patterns.
