🔩 The minimum viable agent

Before any code, define theminimum required to call something agent. If any of these 5 components are missing, you have a function, a chatbot, or a script — but not an agent.

🔩 The 5 mandatory elements

🧠

LLM

The model that reasons and decides actions

🔧

Tools

At least one real action in the outside world

🔁

Loop

Repeated execution until stop condition

👁️

Observation

Tool result injected back into context

🛑

Stop Condition

Clear criteria to know when you stopped

📋 What is NOT mandatory in MVP

You can add later:

  • ○Long memory (RAG) — great, but not required
  • ○Sophisticated planning — start linear
  • ○Framework (LangChain, etc.) — unnecessary at first
  • ○Graphical interface — terminal is sufficient to validate

Already include in the MVP:

  • ✓max_iterations — avoid infinite loop
  • ✓Print of each action — loop visibility
  • ✓Try/except in tool calls — fails gracefully
  • ✓Token counting — cost awareness
======== TOPIC 2 ========

🛠️ Defining tools in Python

A tool is a Python function with aWell described JSON schema. The schema is the contract between the LLM and the code — it tells the model what the function does, what parameters it accepts, and how to call it.

📋 Example: 3 tools for a research agent

# Tools like JSON schema (what the model sees)

tools = [

{

"name": "search_web",

"description": "Search for information on the web. Use when you need current or specific data that is not in your knowledge base. Returns title, URL and snippet of results.",

"input_schema": {

"type": "object",

"properties": {

"query": {"type": "string", "description": "Search term"},

"num_results": {"type": "integer", "default": 5}

},

"required": ["query"]

}

},

{

"name": "save_result",

"description": "Saves a search result to a markdown file. Use ONLY when you have enough information to generate the final report.",

"input_schema": {

"type": "object",

"properties": {

"filename": {"type": "string"},

"content": {"type": "string"}

},

"required": ["filename", "content"]

}

}

]

💡The 3 golden rules for descriptions

1

Describe thewhen to use— "Use when you need current data" provides better guidance than "Web Search"

2

Describe theWhen NOT to use— prevents the model from abusing high-cost tools

3

Describe thethat returns— "Returns title, URL, and snippet" helps the model know what to expect

======== TOPIC 3 ========

🔁 The code execution loop

The loop is the heart of the agent. In Python with the Anthropic SDK, the pattern is straightforward: as long as the model returns tool_use, run the tool and add the result to the context. When end_turn returns, you are finished.

🔁 The complete commented loop

import anthropopic

client = anthropic.Anthropic()

# Initial state

messages = [{"role": "user", "content": objective}]

iteration = 0

while iteration < MAX_ITERATIONS:

iteration += 1

print(f"--- Iteration {iteration} ---")

# 1. THINK: call the model with the current context

response = client.messages.create(

model="claude-sonnet-4-6",

max_tokens=4096,

tools=tools,

messages=messages

)

# 2. EVALUATE: check stop condition

if response.stop_reason == "end_turn":

print("Agent completed the task!")

break

# 3. ACT: execute returned tools

tool_results = []

for block in response.content:

if block.type == "tool_use":

print(f"Calling tool: {block.name}")

result = dispatch_tool(block.name, block.input)

tool_results.append({

"type": "tool_result",

"tool_use_id": block.id,

"content": str(result)

})

#4. OBSERVE: add results to context

messages.append({"role": "assistant", "content": response.content})

messages.append({"role": "user", "content": tool_results})

else:

print(f"Limit of {MAX_ITERATIONS} iterations reached")

📊The dispatch_tool function: action router

def dispatch_tool(name: str, inputs: dict) -> str:

"""Runs the correct tool based on the name"""

try:

if name == "search_web":

return search_web(**inputs)

elif name == "save_result":

return save_result(**inputs)

else:

return f"Error: tool '{name}' not found"

except Exception as e:

return f"Error executing {name}: {str(e)}"

Dispatch always returns string — never propagates exception. The agent receives the error and can decide what to do with it.

total_tokens = 0

for iteration in range(1, MAX_ITERATIONS + 1):

======== TOPIC 5 ========
5

💥 Break the agent on purpose

The quickest way to deeply understand a system isbreak it with intention. Each experiment below will reveal a specific behavior you need to know about before going into production.

E1

Experiment 1: No max_iterations

Remove the iteration limit. Give an impossible-to-achieve goal (“Search ALL 2024 AI Articles”). Watch what happens.

What you will learn:

The agent goes into a loop, spends tokens indefinitely, costs rise without stopping. This makes max_iterations non-negotiable, not optional.

E2

Experiment 2: Wrong schema in the tool

Change the "query" type from "string" to "integer" in the schema. Observe what the model does when trying to call the tool.

What you will learn:

The model may try to call with the wrong type. The tool fails. With a good error handler, the agent tries again or adapts. With a bad handler, the agent enters an inconsistent state.

E3

Experiment 3: Tool that always fails

Make search_web throw an exception every time. See how many times the agent tries before giving up.

What you will learn:

Without a circuit breaker, the agent tries repeatedly. With a structured error in the return, the agent is able to replan. The difference between "throw exception" and "return error string" is critical.

E4

Experiment 4: Ambiguous objective

Change the goal to "Do the best you can." Observe: does the agent stop, ask for clarification, or invent a goal?

What you will learn:

Ambiguous objectives make the agent invent success criteria. Sometimes this is OK (proactive agent), sometimes it is dangerous (agent decides what "best" means without your input).

🎓 Why breaking on purpose is the best training

Each break gives you a visceral intuition of the risk — not theoretical, but lived. After these 4 experiments, you will never again deploy an agent without max_iterations, error handling and a clear objective. Informed fear is the best architectural guide.
