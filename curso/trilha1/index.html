======== MODULE 1.1 ========
1.1 ~35 min

🔄 What changed

From AI that responds to AI that executes missions. The qualitative leap that defines the agentic era.

What is it:

The transition from AIs that answer questions (ChatGPT basic mode) to AIs that receive objectives and execute them autonomously in multiple steps.

Why learn:

Understanding this shift is the mental model that separates those who use AI as a tool from those who use it as a system. Without it, you build fragile automations.

Key concepts:

Punctual response vs. mission execution · Single vs. single interaction continuous loop · No autonomy vs. controlled autonomy

What is it:

Direct comparison between the traditional AI model (question-answer) and the agentic model (goal-execution) in 5 dimensions: autonomy, memory, tools, planning and cycle.

Why learn:

Visualizing the difference avoids confusion. Many people call it an "agent" which is simply an API call with an elaborate prompt.

Key concepts:

Controlled autonomy · Persistent memory · External tools · Stepwise planning · Evaluation loop

What is it:

The market context: market growing from $7.8B to $52B by 2030, 94% of engineering leaders with skills gaps, 40% of enterprise apps with embedded agents by the end of 2026.

Why learn:

The window of opportunity exists now. Whoever masters agentic engineering in the next 12-18 months will have a lasting advantage in the market.

Key concepts:

MCP adopted by OpenAI in 2025 · Accelerated standardization · 1,445% increase in multi-agent queries · Engineer = agent orchestrator

What is it:

Clear criteria to identify a real agent: it has an execution loop, uses external tools, evaluates its results and can replan. A chatbot with a personality is not an agent.

Why learn:

The market uses "agent" to sell anything with AI. You need technical criteria to separate the real from the marketing.

Key concepts:

Mandatory loop · Real tool use · Output evaluation · Ability to replan · Difference between chatbot, automation and agent

What is it:

The analogy that makes it easier to understand: an agent is like a competent employee you hired for a mission — not a Google to answer questions. You define the objective, he plans and executes it.

Why learn:

Analogies are more durable than technical definitions. When you think "agent = mission employee", you will naturally know what he can and cannot do.

Key concepts:

Objective → Plan → Execution → Report · Autonomy within limits · Scale to human when necessary · Governance as an employment contract

What is it:

Concrete examples of agentic systems in use in 2026: Claude Code (developer agent), Devin (autonomous software engineer), GitHub Copilot Workspace (PR agent), SDR agents in sales, legal contract review agents.

Why learn:

Seeing real applications anchors learning. You stop learning an abstract concept and start seeing opportunities all around you.

Key concepts:

Developer Agent · Sales Agent · Legal Agent · HR Agent · Workflow Automation · The Cost of NOT Having Agents

======== MODULE 1.2 ========
1.2 ~40 min

🔁 The Agentic Loop

Think → Act → Observe → Evaluate → Adjust. The cycle that defines any real agent. You will see this in everything.

What is it:

The step in which the LLM receives the objective, analyzes the available context (memory + current state) and decides which action to take next. It's the "reasoning step" — chain-of-thought in practice.

Why learn:

Understanding the thinking phase allows you to diagnose when an agent makes the wrong decision — this is usually where the problem begins.

Key concepts:

Chain-of-Thought · ReAct (Reason + Act) · Reasoning Scratchpad · Available Context · Next Action Decision

What is it:

Executing the chosen action — calling an API, executing code, searching the web, writing a file, sending a message. This is where the agent leaves the "box" and interacts with external systems.

Why learn:

Actions have real consequences — deleting a file, sending an email, spending tokens. Understanding this phase is understanding risk and governance.

Key concepts:

Tool calling · Function calling · External API · Irreversible consequences · Permission control · Sandboxing

What is it:

The return of the action — the API output, the code result, the search response — is injected back into the agent context. Without observation, there is no loop.

Why learn:

Observation problems (poorly formatted result, silent error, timeout) are frequent causes of agents that "appear" to work but produce garbage.

Key concepts:

Tool output · Context injection · Observation error · Result parsing · Context size after observation

What is it:

Based on what it observed, the agent assesses whether it is progressing towards the goal. If yes, continue. If not, adjust the plan. A script executes blindly — the agent evaluates and corrects itself.

Why learn:

This is the ability that justifies the name “agent”. Without evaluation, you have a glorified script, not an agent.

Key concepts:

Success criteria · Replanning · Error detection · Iteration limit · Stopping condition · Escalation to human

What is it:

Complete walkthrough of a research agent: objective received → research → quality assessment → new research → synthesis → delivery. We see each iteration of the loop with what the agent "thinks" at each step.

Why learn:

Seeing the loop in action is much more powerful than reading the definition. You will internalize the pattern and recognize it in any framework.

Key concepts:

Iteration 1, 2, 3 · Incremental progress · Token count per loop · Total execution cost · Total time

What is it:

The 3 classic problems of the agentic loop: infinite loop (agent does not find stopping criteria), cascading hallucination (error in step 2 contaminates steps 3 and 4), and explosive cost (100 iterations = 100x the expected cost).

Why learn:

Every agentic system in production has had at least one of these problems. Knowing them before building is what separates the developer who prevents from the developer who puts out fires.

Key concepts:

max_iterations · Timeout · Budget limit · Error propagation · Circuit breaker · Human escalation

======== MODULE 1.3 ========
1.3 ~40 min

🧩 Agent Components

LLM, Memory, Tools, Planning and Governance. The 5 pieces every real agent needs to have.

What is it:

The language model is the agent's decision center. Receives the current state (goal + memory + result of actions), reasons and decides the next step.

Why learn:

The choice of LLM (Claude, GPT-4, Gemini, local models) directly affects the quality of reasoning, the cost per iteration and the available context limits.

Key concepts:

Context window · Temperature · Native function calling · Cost per token · Reasoning vs. execution · Small model for action, big for planning

What is it:

Short-term memory = context of the current conversation (token window). Long-term memory = vector database or database that persists between sessions. An agent without long memory restarts from scratch every time.

Why learn:

Without persistent memory, agents cannot learn from previous executions or maintain context between sessions. It's the difference between an amnesiac assistant and a collaborator who knows his job.

Key concepts:

Conversation Context · RAG · Vector Bank (FAISS, Chroma) · Episodic vs. semantic · Retrieval cost · Memory relevance

What is it:

Tools are functions that the agent can call: searching the web, executing Python code, reading files, querying a database, sending emails. Each tool is an extra capability beyond the pure LLM.

Why learn:

Without tools, the agent is stuck in the text context. With tools, he can act in the real world. The quality and safety of the tools defines the agent’s power and risk.

Key concepts:

Function schema · Typed input/output · Tool safety · MCP (Model Context Protocol) · Tool chaining · Permissions per tool

What is it:

For complex objectives, the agent needs to decompose the problem into subtasks, order by dependency and execute sequentially or in parallel. It's the difference between "make a website" and an executable 7-step plan.

Why learn:

Planning is where the agent shows his intelligence. A good plan results in efficient execution. A bad plan leads to loops, cascading errors, and exploding cost.

Key concepts:

Task decomposition · Dependency graph · Parallel vs. sequential · BMad (Breakdown-Manage-Delegate) · Plan-and-Execute pattern

What is it:

Governance is the system of rules that controls the agent: which actions are allowed, which ones need human approval, how much they can spend, what should be logged in and how to audit history.

Why learn:

Without governance, an agent with access to the system can delete files, spend a fortune on tokens, leak data or perform irreversible actions. Governance is not optional in production.

Key concepts:

Tool permissions · Budget cap · Audit log · Human-in-the-loop · Guardrails · Reversible actions vs. irreversible

What is it:

Diagram showing how LLM + Memory + Tools + Planning + Governance connect in a minimal functional architecture. Each missing component makes the agent more fragile.

Why learn:

Having the complete map before starting to build avoids the most common mistake: creating a brilliant agent in demos that fails in production due to a lack of an essential component.

Key concepts:

Minimum viable architecture · Simplicity trade-offs · What to add first · LLM Diagram → Memory → Tools → Loop → Governance

⚡ Vibe Coding vs. Agentic Engineering

Evolution, not replacement. Understand where Vibe Coding ends and Agentic Engineering begins — and why you need them both.

What is it:

Vibe Coding is the paradigm popularized by Andrej Karpathy in 2025: you describe what you want in natural language and AI generates the code. Fast iteration, focus on results, low friction for non-programmers.

Why learn:

Vibe Coding is what opened the door for millions of people to create software. Understanding it is understanding the starting point of the journey that leads to Agentic Engineering.

Key concepts:

Prompt → generated code · Iteration per conversation · Claude/Cursor/Bolt/Lovable · Focus on speed · No automatic loop

What is it:

Vibe Coding fails when the system needs multiple automated steps, state maintenance between sessions, autonomous execution without constant human intervention, or when the error in a step must be fixed automatically without going back to the human.

Why learn:

Recognizing the limits of Vibe Coding is what motivates you to learn Agentic Engineering. It's not abandonment — it's a natural upgrade when the project grows.

Key concepts:

Context limit · Dependency on human in the loop · No persistence · Fragility in production · "Works on my computer"

What is it:

Agentic Engineering comes in when you need the system to run itself, perform multiple steps, evaluate its own results, and recover from failures — without you in the conversation all the time.

Why learn:

Knowing the right time to upgrade avoids the opposite mistake: using Agentic Engineering for things that would be resolved with a good prompt.

Key concepts:

Signal 1: repetitive task with variations · Signal 2: multiple sequential steps · Signal 3: you want to sleep and let it run · Signal 4: high error cost

What is it:

Direct comparison across 8 dimensions: autonomy, memory, loop, tools, governance, cost, scalability and complexity. Each dimension with practical example of when one approach surpasses the other.

Why learn:

Having the table in your head allows you to quickly decide which approach to use for each problem — without defending one as “always better”.

Key concepts:

Code generation vs. orchestration · Focus on speed vs. system focus · Manual iteration vs. automated loop · Low governance vs. structured governance

What is it:

Advanced professionals use Vibe Coding for rapid prototyping and Agentic Engineering for production. The flow: Vibe Coding to validate the idea in hours → Agentic to scale and automate.

Why learn:

Abandoning Vibe Coding when learning Agentic is a common mistake. The two coexist in the workflow of those who produce faster.

Key concepts:

Prototype in Vibe Coding · Refactor to Agentic · Claude Code as a bridge · When to rewrite vs. when to evolve

What is it:

10 real-world scenarios to rate: Vibe Coding, Agentic Engineering, or both? Ex: "create a portfolio website" vs. "process 500 contract PDFs per day" vs. "generate weekly sales report".

Why learn:

Deciding when to use each approach is a practical skill. The exercise trains judgment before entering the technical part.

Key concepts:

Criteria: frequency · Criterion: necessary autonomy · Criterion: error tolerance · Criterion: implementation cost vs. operating cost

🌐 The Ecosystem in 2026

Claude Code, Cursor, Devin, n8n, CrewAI, LangGraph. The complete map of tools and where each one fits into the agentic architecture.

What is it:

The agentic ecosystem is organized into 4 layers: Models (Claude, GPT-4, Gemini), Frameworks (LangGraph, CrewAI, AutoGen), Agentic IDEs (Claude Code, Cursor, Devin) and Automation Platforms (n8n, Make, Zapier AI).

Why learn:

Without the map, you don't know what you are comparing. A person who says "I'm using LangGraph" and another who says "I'm using Claude Code" may be building completely different — or complementary — things.

Key concepts:

Model layer · Framework layer · Agentic IDE layer · Automation layer · How the layers connect

What is it:

Tools where an agent operates within the development environment: reads code, writes, runs terminal, tests and corrects. Claude Code is agent-first at the terminal. Cursor is agent-first in the editor. Devin is the most autonomous of the three.

Why learn:

They are the most immediate tools for a developer. You can use them today — and they're the best entry point to experiencing what it's like to work with a real agent.

Key concepts:

Claude Code (CLI, skills, MCP) · Cursor (Composer, agent mode) · Devin (standalone complete task) · GitHub Copilot Workspace · When to use each

What is it:

Python frameworks for building custom agents. LangGraph: state graph for precise control. CrewAI: teams of agents with roles. AutoGen: multi-agent conversation. Trail 3 goes in depth with everyone — here’s just the map.

Why learn:

These frameworks are the choice when you need custom logic that ready-made IDEs don't cover. They are the construction layer, not the use layer.

Key concepts:

LangGraph (graph, state, fine-grained control) · CrewAI (roles, delegation) · AutoGen/AG2 (conversation, reflection) · OpenAI Agents SDK · When to build vs. when to use ready

What is it:

Visual workflow platforms that have added agentic capabilities. n8n is the most powerful and self-hostable. Make (formerly Integromat) and Zapier AI focus on ready-made integrations. They are the link between agents and existing corporate systems.

Why learn:

For business automation involving dozens of systems (CRM, ERP, email, Slack), low-code platforms are more efficient than custom Python frameworks.

Key concepts:

n8n (self-hosted, code + visual) · Make (visual, rich integrations) · Zapier AI (simpler, faster) · When low-code beats pure code

What is it:

Model Context Protocol (MCP) is the open standard created by Anthropic and adopted by OpenAI in March 2025. It defines how agents discover and use tools in a standardized way — databases, APIs, files, external systems.

Why learn:

MCP became the "HTTP of agents". Any tool that implements MCP works with any agent that supports it. It's the integration pattern you need to understand now.

Key concepts:

MCP Server · MCP Client · Tool Discovery · OpenAI Adoption · Claude Code + MCP · Creating your MCP Server (Track 3)

What is it:

Decision guide for which tool to start based on your profile: dev who wants to increase productivity (Claude Code), builder who wants to create a product (LangChain + LangGraph), process automation (n8n), executive who wants to see it in action (Cursor).

Why learn:

Choice paralysis is the biggest enemy in the beginning. Having clear criteria for choosing the first tool accelerates practical learning.

Key concepts:

Dev profile · Builder profile · Automator profile · First tool vs. full stack · "Use one tool until it hurts, then learn the next one"

🤖 First Agent

Hands-on without framework. You'll build a minimally functional agent in pure Python — and understand every line you write.

What is it:

A minimal agent needs: 1 LLM, 1+ tools defined as Python functions, 1 loop that continues until the goal is reached, and 1 mechanism for observing the results of the tools. Nothing else is mandatory to get started.

Why learn:

Starting with the minimum is what ensures you understand each piece before adding complexity. Agents built without understanding the minimum accumulate invisible technical debt.

Key concepts:

LLM call · Tool definition · While loop · Tool execution · Result injection · Stop condition

What is it:

Tools are Python functions with a JSON schema that describes: name, description, parameters and types. LLM reads the schema and decides when and how to call each tool. The schema is the "documentation" that the agent consults.

Why learn:

A poorly written schema results in an agent not using the tool correctly, using it at the wrong time, or ignoring it completely. The quality of the schema is directly proportional to the quality of the behavior.

Key concepts:

JSON schema · Tool description · Typed parameters · Docstring as instruction · Pydantic for validation · Good naming practices

What is it:

The agent's main loop: (1) call LLM with current state, (2) check if it requested a tool call, (3) execute the tool, (4) inject result back, (5) repeat. Terminates when LLM no longer requests tools (goal reached) or when it reaches max_iterations.

Why learn:

Writing the loop once from scratch is the most important exercise in Track 1. Once you do this, any framework will make sense — because you will recognize the same abstracted pattern.

Key concepts:

while not done · tool_calls check · dispatch_tool() · messages.append() · max_iterations · finish_reason

What is it:

Step-by-step construction of an agent that: receives a topic, decides to search the web, reads the results, decides if it needs more information, and delivers a structured summary. ~50 lines of Python without any framework.

Why learn:

This is the module's portfolio project. A functional agent in 50 lines is the best technical business card for those just starting out in the field.

Key concepts:

anthropic.Anthropic() · tools=[] · tool_use block · tool_result block · Total execution cost · Time per iteration

What is it:

4 failure experiments: (1) remove max_iterations and see infinite loop, (2) give tool wrong schema and see behavior, (3) make tool return error and see what the agent does, (4) give ambiguous objective and see how the agent interprets it.

Why learn:

Seeing the agent fail in a controlled way is the most valuable learning experience. You will find all these failures in production — better to find them now, with experiments, than in deployment.

Key concepts:

Infinite loop in practice · Poorly defined schema · Tool error handling · Vague objective vs. precise objective · Cost of a buggy run

What is it:

Preview of what Track 2 adds to the simple agent you just built: long memory with vector database, RAG, tools with access to real databases, advanced reasoning patterns (ReAct, Plan-and-Execute) and LangChain as an abstraction.

Why learn:

Seeing where the path goes motivates you to continue. You will enter Track 2 knowing exactly what you will learn and why each component matters.

Key concepts:

Track 2 preview · From simple agent to complete assistant · Why LangChain exists · When to stop reinventing the wheel · Concrete next steps
