🔁 The Agentic Loop: Think → Act → Observe → Evaluate → Adjust

Every real agent executes a continuous cycle. Understanding each phase of this loop — and where it can fail — is what separates those who build agents from those who just use AI tools.

======== TOPIC 1 ========
1

🧠 Think: the reasoning phase

Before taking any action, the agentthink. This phase is where the LLM analyzes the objective, examines the available context and decides what the next action will be. This is where the quality of the template and prompt make the biggest difference.

🧠What happens during reasoning

LLM receives the complete context — objective, history of previous actions, results of tools already executed — and generates reasoning before deciding the next action. This internal reasoning is the heart of the loop.

Chain-of-Thought (CoT)

  • • The “think aloud” model before responding
  • • Step-by-step reasoning improves accuracy by up to 40%
  • • Visible in scratchpad / thinking tokens
  • • Extra cost in tokens, but valid for complex tasks

ReAct (Reason + Act)

  • • Default: Thought → Action → Observation
  • • Reasoning and action interspersed in the same loop
  • • Each observation feeds the next thought
  • • Base of most modern frameworks

📋 What is in context during Thinking

1.
System prompt: Persona, constraints, available tools and your goal as an engineer
2.
Message history: Everything that happened in previous loop iterations
3.
Tool results: Results of previously called tools
4.
Current objective: The mission defined by the human or parent planner

💡Practical Tip

Use templates withextended thinking(Claude claude-sonnet-4-6 thinking, o1) for tasks that require multiple logical steps. For simple tasks, avoid the extra cost. The rule: if a human would need more than 30 seconds to think, use thinking enabled.

======== TOPIC 2 ========
2

⚡ Act: execution with tools

After reasoning, the agentacts. This means issuing a tool call — a structured instruction to perform an action in the real world. This is where AI stops being text and becomes a consequence.

📊Anatomy of a Tool Call

A tool call is a JSON structure that the model returns in place of text. Your code intercepts this and takes the corresponding action.

// The model returns this:

{

"type": "tool_use",

"name": "search_web",

"input": {

"query": "Bitcoin price today",

"max_results": 5

}

}

✓ Good practices in Action

  • ✓ Validate tool inputs before executing
  • ✓ Set timeout by tool call
  • ✓ Log each action with timestamp and cost
  • ✓ Use sandboxing for destructive actions
  • ✓ Require human confirmation for irreversible actions

✗ Common mistakes in the Action phase

  • ✗ Give write permissions without needing to
  • ✗ Execute SQL without query validation
  • ✗ Tools without retry logic for transient faults
  • ✗ Ignore permission errors from external systems
  • ✗ Call APIs with rate limits without backoff

⚠️Actions have real consequences

Unlike a chatbot that only generates text, an agent in action can: delete files, send emails, spend money via API, modify databases in production. Treat each tool as if it were a junior developer with admin access — set minimum permissions and full auditing.

======== TOPIC 3 ========
3

👁️ Observe: reading the result

After acting, the agentobservethe result of the action. The tool's output is injected back into the context as a special message of the typetool_result, and the next cycle of reasoning begins with this information.

👁️What enters the context after observation

OK

Successful result

The tool's output is serialized and added astool_result. The agent knows what happened and can continue.

ERR

Error Result

The error is also injected into the context. Well-designed agents can read the error and try an alternative approach instead of crashing.

BIG

Very large output

Did a search return 50 pages? Does a file have 10,000 lines? You need a truncation strategy — context has a limit and overflow causes silent failures.

💡Observation Parsing Tip

Never inject raw output from tools without processing. Before adding to context:

  • • Limit the maximum size (e.g. first 2000 characters of a file)
  • • Serialize complex objects as formatted JSON, not Python repr
  • • Add useful metadata: timestamp, tool used, HTTP status
======== TOPIC 4 ========
4

⚖️ Evaluate: What separates agent from script

The phase ofassessmentThis is where a true agent differs from linear automation. After observing the result, the agent decides: was the objective achieved? Is the result enough? Do you need to try another way?

🔍 The 4 questions of the Assessment phase

1

Was the objective achieved?

Success criteria need to be defined in the reasoning system. Without clear criteria, the agent never knows when to stop.

2

Was the result a recoverable error?

Network error is different from permission error. The agent must classify errors and decide whether it can try again or needs to escalate.

3

Is the current plan still valid?

If research revealed that the original approach was wrong, the agent can (and should) replan before continuing.

4

Is it time to stop and ask for help?

Human-in-the-loop is not weakness — it's architecture. Defining when to escalate to human is as important as defining when to go it alone.

📊Replanning: when the agent changes strategy

Replanning happens when the assessment detects that the current plan will not work. A good agent does not insist on a failed strategy:

  • •External API returned 429? Attempt with cached data.
  • •Data format different than expected? Adaptive parse.
  • •Impossible sub-task? Remove from plan, report limitation.
  • •Ambiguous objective? Stop and ask the human for clarification.

🔄 The loop in practice: complete walkthrough

Let's see the complete loop in action. Objective:"Search Bitcoin price over the last 7 days and generate a trend summary."— 4 real iterations with cost per iteration.

I.1

Iteration 1 — Plan and fetch data

~800 tokens · ~$0.002

Think:"I need price data. I will use the web search tool to find the current price and 7-day history."

Act: search_web("Bitcoin price last 7 days 2026")

Note:Returned 5 results with prices from financial websites

I.2

Iteration 2 — Extract structured data

~1200 tokens · ~$0.003

To assess:"The search results have text mixed with prices. I need structured data. I will extract the prices from a direct API."

Act (replanning): call_api("coinbase.com/v2/prices/BTC-USD/historic")

Note:JSON with prices for the last 7 days returned successfully

I.3

Iteration 3 — Calculate trend

~900 tokens · ~$0.002

Think:"I have the data. I will calculate percentage change, maximum and minimum price for the period."

Act: run_python("calcular_tendencia(precos)")

Note:+12.3% in the period, maximum $98,400, minimum $87,200

I.4

Iteration 4 — Generate summary (DONE)

~600 tokens · ~$0.001 · TOTAL: ~$0.008

To assess:"Goal achieved. I have data and analysis. I can generate the final summary."

Result:Formatted summary with +12.3% uptrend, market context and monitoring recommendation

======== TOPIC 6 ========
6

⚠️ When the loop becomes a problem

The agentic loop is powerful — and that's why it's dangerous when it gets out of control. Without the correct guardrails, an agent can get into trouble.infinite spiral, generate cascading hallucinations and accumulate costs explosively.

🔁 Infinite Loop

The agent assesses that the objective was not achieved, tries again, fails the same way, tries again... indefinitely.

Solution:max_iterations with explicit fallback

🌀 Waterfall Hallucination

The agent hallucinates a tool result, uses this false result to generate another result, and the error amplifies with each iteration.

Solution:Tool output validation before injecting into context

💸 Explosive Cost

With no limit on iterations or budget cap, a poorly configured agent can spend hundreds of dollars on a single lost execution.

Solution:Budget cap per execution + alert as a percentage of the limit

🛑 Saturated Context

After many iterations, the context becomes full of history. The model starts to lose track of the original objective and generates inconsistent responses.

Solution:Sliding window + automatic history summary

🛡️ The 3 mandatory guardrails

1️⃣

max_iterations = 20

Never leave a loop open. Set a realistic maximum for your task and add an explanatory fallback when reached.

2️⃣

budget_cap = $5.00

Set a cost ceiling per run. Monitor spent tokens in real time and stop with report when near limit.

3️⃣

circuit_breaker for repeated errors

If the same error occurs 3 times in a row, stop and escalate to the human. Don't let the agent insist on a strategy that has clearly failed.
