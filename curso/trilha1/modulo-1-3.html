🧠 LLM: the agent’s brain

The LLM is the central component — it is what reasons, decides and generates actions. Butnot all LLMs are equally goodfor all types of agents. Choosing the wrong model is like hiring the wrong professional for a project.

🧠Model selection criteria

Context Window

Determines how much loop history can be maintained

  • • Claude: 200k tokens
  • • GPT-4o: 128k tokens
  • • Gemini 1.5: 1M tokens

Reasoning vs. Speed

Central trade-off in agentic architecture

  • • Opus: maximum reasoning, slow
  • • Sonnet: ideal balance in most
  • • Haiku: speed, simple tasks

Cost per Token

In a loop with 20 iterations, it multiplies quickly

  • • Haiku: ~$0.25/M tokens
  • • Sonnet: ~$3/M tokens
  • • Opus: ~$15/M tokens

💡Hybrid model strategy

Don't use the same template for everything. The most efficient architecture uses different models per task type:

Planning

Sonnet / Opus

High complexity

Execution

Sonnet

Cost/quality balance

Verification

Haiku

Fast and cheap

💾 Short Term Memory

An agent's short-term memory is theconversation buffer— the message history that remains in context during execution. It is temporary, limited by the model's token window, and is reset when execution ends.

🗂️ How short memory is managed

1

Full buffer (start)

All messages are in context. Simple, but grows linearly with each loop iteration.

2

Sliding window (medium term)

Keeps only the last N messages. Efficient, but may lose important context from the beginning of execution.

3

Automatic summary (long term)

When the buffer fills, LLM itself condenses the history into a summary that replaces the old messages. Best value for money for long loops.

📊Cost impact

Short memory is charged on each iteration because all context tokens are reprocessed by the model with each call.

  • •Iteration 1: 500 tokens in context → cost
  • •Iteration 5: 2,500 tokens → 5X cost
  • •Iteration 20: 10,000 tokens → 20X cost
  • •Conclusion:sliding window or summarization are mandatory for long loops

🗄️ Long Term Memory

Long-term memory persists between runs. This is where the agent stores accumulated knowledge, interaction history, user preferences and reference documents. Without it, the agent starts from scratch every time.

🗄️Types of long memory

Vector Bank (RAG)

Stores documents as embeddings. Semantic search retrieves the most relevant ones for the current context.

• FAISS — local, open-source, fast

• Chroma — local, easy to use

• Pinecone — cloud, infinite scale

• Weaviate — hybrid + filters

Relational Bank / KV

For structured data: preferences, action history, past results, settings per user.

• SQLite — local, serverless

• PostgreSQL — production, relations

• Redis — cache, fast sessions

• DynamoDB — serverless, scale

🔍 How RAG (Retrieval-Augmented Generation) works

1 Documents are transformed into embeddings (numeric vectors) and stored
2 During execution, the agent's query also becomes embedding
3 The bank searches for documents with the greatest semantic similarity (cosine similarity)
4 The top-K documents are injected into the context as relevant knowledge

Result: agent with access to an unlimited knowledge base without exploding the context

💡 When to use which type of memory

Use vector banking (RAG) to:

  • • Technical documentation and manuals
  • • Company knowledge base
  • • Conversation history by similarity

Use relational banking to:

  • • Preferences and settings per user
  • • History of actions performed
  • • Structured metrics and results

🔧 Tools: the agent’s hands

Tools are the agent's action capabilities — they are what allow you to leave the text world for the real world. A well-defined tool is the difference between an agent that works and one that hallucinates trying to do what it cannot.

📋 Anatomy of a well-defined Tool

# Tool schema in Anthropic format

{

"name": "send_email",

"description": "Sends an email to a recipient. Only use when the user has explicitly asked to send an email. NEVER use for internal notifications.",

"input_schema": {

"type": "object",

"properties": {

"to": {"type": "string", "description": "Recipient email"},

"subject": {"type": "string"},

"body": {"type": "string"}

},

"required": ["to", "subject", "body"]

}

}

A descriptionis the instruction for using the tool. It guides the model on when to use it and when NOT to use it — that's safety.

✓ Well-designed tools

  • ✓ Clear and descriptive name (verb + object)
  • ✓ Description includes when NOT to use
  • ✓ Explicitly defined input types
  • ✓ Returns structured error, never raw exception
  • ✓ A responsibility per tool (SRP principle)

✗ Common errors in Tools

  • ✗ Ambiguous names like "process_data"
  • ✗ Empty description or just the name repeated
  • ✗ Tool that does 5 different things
  • ✗ No error handling — exception leaks into context
  • ✗ Optional parameters with no documented default value

📊MCP: Model Context Protocol

MCP is Anthropic's open standard for integrating tools into any agent. In 2025, OpenAI also adopted the protocol, making it the de facto industry standard. With MCP, you define tools once and any compatible agent can use them.

📋 Planning: divide and conquer

An agent without planning is a reactive agent — it performs one action at a time without seeing the whole. The component ofplanningtransforms a complex objective into a structured sequence of executable sub-tasks.

🗺️ Task Decomposition in practice

Objective:"Create a market report on generative AI to present to the board on Friday"

1 Search market data (parallel with 2)
1a Search analyst reports (Gartner, IDC, McKinsey)
1b Collect funding and M&A data from 2025-2026
2 Competitive analysis (parallel with 1)
3 Synthesize insights (depends on 1 and 2)
4 Generate slides with views (depends on 3)

Parallel Execution

Independent sub-tasks run at the same time. Dramatically reduces total time on research-intensive tasks.

Example: Searching 5 sources simultaneously instead of sequentially

Sequential Execution

Sub-tasks with dependencies run in order. Ensures that each step uses the correct results from the previous ones.

Example: generate slides only after having analyzed the data

🛡️ Governance: the agent contract

Governance is not bureaucracy — it is the difference between a trusted agent in production and a system that can cause irreparable harm. THEagent contractdefines what it can do, how it reports and how it is audited.

🛡️ The 6 pillars of agentic governance

🔐

Minimum permissions

The agent only accesses what it needs for the current task. Extra permission = extra attack surface.

💰

budget cap

Spend limit on tokens and API calls per run. Set before starting, monitor in real time.

📋

Audit log

Every tool call, result and decision recorded with timestamp. Non-negotiable in production.

👤

Human-in-the-loop

Human approval points for high-impact actions. Set the threshold before building.

↩️

Preferred reversible shares

Always prefer an action that can be undone. Deleting a file vs. move to trash.

🧪

Sandbox before production

Test with data and staging systems. Never validate a new agent directly in production.

⚠️Irreversible actions require human approval

Create an explicit list of actions that can never be performed without human confirmation. Examples:

Permanently delete production data
Send bulk communications to customers
Make financial transfers above the limit
