🏗️ The real challenge of integrating agents into enterprises

Most companies operate between50 and 200 internal systems— ERPs, CRMs, HR systems, legacy databases, internal multi-team APIs, BI systems and SaaS tools. An enterprise agent needs to communicate with all of them in a reliable, secure and traceable way.

🗺️The reality of the enterprise environment

In proof-of-concept projects, you integrate the agent with 2 or 3 well-documented APIs. In enterprise production, the reality is different: undocumented APIs, heterogeneous authentications, systems that frequently crash and teams that don't want you to access their data.

POC environment

  • →2-3 well-documented APIs
  • →Simple authentication (API key)
  • →Test data available
  • →Eventual fault tolerance

Enterprise Production Environment

  • →50-200 systems with varying quality
  • →OAuth, SAML, JWT, mTLS certificates
  • →Sensitive data with compliance
  • →99.9% SLA required

📊Enterprise market data

  • •73%of enterprise AI projects fail in the integration phase, not the model phase (Gartner, 2025)
  • •Average Fortune 500 company has164 SaaS systemsin simultaneous use (Okta, 2025)
  • •45%of the cost of an enterprise agent project is in the integration layer

💡Enterprise integration principle

Build the integration layer as aseparate agent infrastructure. The agent must not know how to authenticate to SAP — it calls a tool that abstracts this. This allows you to switch systems without rewriting the agent.

🌐 REST and GraphQL: authentication, retry and pagination

REST and GraphQL are the dominant standards in modern APIs, but implementing them correctly for agents requires special attention torobust authentication, handling rate limits and efficient pagination — each token consumed by the agent has a cost.

🔐 OAuth 2.0 flow for agents

1

Client Credentials Flow

For server-to-server agents: agent uses client_id + client_secret to obtain access_token. There is no user involved. Token expires — agent must automatically renew before expiration.

2

Automatic Token Refresh

Implement proactive refresh: if token expires in less than 5 minutes, renew before making the call. Never wait to receive a 401 to renew — this causes the agent's tool call to fail.

3

Rate Limit with Exponential Backoff

When receiving 429 Too Many Requests, wait the time indicated in the Retry-After header. If there is no header, use exponential backoff: 1s, 2s, 4s, 8s... with jitter to avoid thundering herd.

GraphQL: advantages for agents

  • ✓Fetches exactly the fields needed — fewer tokens in the agent context
  • ✓A query can fetch data from multiple related entities
  • ✓Typed schema — agent can discover available fields via introspection
  • ✓Mutations for writing with updated data return

Common REST/GraphQL mistakes

  • ✗Hardcoding credentials in the tool code
  • ✗Do not handle pagination — only get the first page
  • ✗Retry without backoff — causes more rate limiting
  • ✗Search all fields when you only need 3

📄 Pagination for agents: cursor-based

Enterprise APIs with large volumes of data use cursor-based pagination. The agent must iterate pages until it reaches the stopping criterion — not fetch everything at once.

# Padrão correto de paginação para agentes
async def fetch_all_records(api_client, endpoint, filter_params):
    results = []
    cursor = None
    while True:
        page = await api_client.get(endpoint,
            params={**filter_params, "cursor": cursor, "limit": 100}
        )
        results.extend(page["data"])
        cursor = page.get("next_cursor")
        if not cursor or len(results) >= MAX_RECORDS:
            break
    return results

📨 Webhooks and events: externally triggered agents

Webhooks reverse the flow: instead of the agent calling external systems, external systems call the agent when something happens. This allowsevent-based asynchronous processing— the agent only executes when there is real work to do.

🔄 Webhook architecture for agents

1

Signature validation

Always validate the webhook's HMAC signature before processing. Without this, any malicious agent can send fake requests to the endpoint and trigger the agent with forged data.

2

Mandatory idempotence

Webhooks can be delivered more than once (automatic retry). Use idempotency key — record the ID of the processed event and ignore duplicates. Without this, the same agent runs twice for the same event.

3

Immediate response, asynchronous processing

Reply 200 OK immediately upon receiving the webhook (in less than 5s, otherwise the system resends). Place the event in a queue for asynchronous processing. The agent processes from the queue, not the HTTP request.

⚠️The idempotence problem in webhooks

Real scenario: a CRM sends "signed contract" webhook. The agent processes the event and triggers an onboarding email. 30 seconds later, CRM resends the webhook due to timeout. Without idempotence, the customer receives two onboarding emails. With financial data, this can mean double billing.

Solution:IF NOT EXISTS (SELECT 1 FROM processed_events WHERE event_id = ?) THEN process AND INSERT INTO processed_events

📬 Legacy systems and queues: RabbitMQ and Kafka

Every large company has legacy systems that don't speak REST — IBM mainframes, ERPs from the 90s, core COBOL banking systems.Message queuesthey are the middleware that allows modern agents to communicate with these systems without modifying the legacy.

RabbitMQ — for business messaging

  • •Ideal for low volume tasks with complex routing
  • •Exchanges and bindings for message routing by type
  • •Dead Letter Queue for messages that failed N times
  • •Message acknowledgment: agent confirms after processing
  • •Queue priority: urgent agents process first

Apache Kafka — for event streams

  • •Ideal for high volume — millions of events/second
  • •Partitioned topics: parallel agents process at scale
  • •Configurable retention: replay historical events
  • •Consumer groups: multiple agents share the load
  • •Exactly-once semantics with Kafka transactions

🔄 Pattern: agent as queue consumer

The most robust standard for integration with legacy systems: the legacy system publishes events in a known format (e.g. new order, new customer, new contract). The agent consumes from the queue and processes. The legacy does not know that the agent exists.

1 Legacy system publishes message to the queue:{"event": "new_order", "order_id": "ORD-123"}
2 Agent consumes the message, fetches order details via API
3 Agent processes (risk analysis, notification, CRM update)
4 Agent acknowledges (ack) the message — removed from the queue

💡Delivery guarantee

At-least-once delivery is the secure standard: the message can be delivered more than once, but never lost. Combine with idempotence in agent processing. Exactly-once is possible with Kafka transactions, but it has overhead — only use it when idempotence is not enough.

TOPIC 5

🔀 API Gateway for agents: centralization and control

Instead of each agent implementing authentication, rate limiting and logging in their own way, theAPI Gateway centralizesall these concerns. Every request to/from agents passes through the gateway — single point of control, auditing and optimization.

🏛️ What API Gateway does for you

Centralized authentication

Validates JWT/API key before reaching the agent. The agent never sees invalid credentials — the gateway already filters.

Rate limiting per agent

Limits calls per second per agent or per user. Prevents a buggy agent from consuming all available resources.

Logging and traceability

Every request/response logged centrally with correlation ID. Crucial for debugging agent behavior in production.

Intelligent Routing

Directs requests to the correct agent based on path, header or content. Allows A/B testing between agent versions.

⚙️ AWS API Gateway vs. Kong

AWS Serverless, native integration with Lambda/ECS where agents run. Managed (no infrastructure to maintain). Best for 100% AWS stacks. Lambda authorizer for custom authentication.
Kong Open-source, multi-cloud, highly extensible via plugins. Best for hybrid environments (on-premise + cloud). Rich plugin ecosystem: advanced rate limiting, JWT, OAuth, caching, transformations.

⚠️Without API Gateway: problems in production

No centralized gateway: each agent implements different auth (inconsistency), no visibility of who calls what (impossible debugging), no rate limiting (one agent takes down all the others), no auditing (impossible compliance). The gateway is not optional in enterprise.

🔑 Secrets management: Vault, Secrets Manager and zero-trust

The rule is absolute:NEVER hardcode credentials on agents. Never in hardcoded environment variables, never in committed config files, never in source code. An agent with exposed credentials is an attack vector for all systems it accesses.

🏛️ HashiCorp Vault for Agents

Vault providesdynamic secrets— instead of a fixed database password, the agent asks Vault for a temporary credential with a TTL of 1 hour. When the agent finishes, the credential automatically expires. Credential leak expires on its own.

1 Agent boots and authenticates to Vault with its identity (AppRole, Kubernetes ServiceAccount)
2 Vault checks that the agent has permission for the requested secret (least privilege policy)
3 Vault returns dynamic credential with TTL — agent uses only during task
4 TTL expires: Invalid credential automatically. Vault rotates credential on target system

AWS Secrets Manager

  • •Native integration with IAM — agent assumes role with permission
  • •Automatic rotation of RDS passwords without downtime
  • •Multi-region replication for distributed agents
  • •CloudTrail audit of all access to secrets
  • •$0.40/secret/month — negligible cost vs. risk

What NEVER to do

  • ✗Hardcodar API keys in the agent code
  • ✗Commit .env with credentials in git
  • ✗Pass credentials in CI/CD environment variables without encryption
  • ✗Log API responses that contain tokens
  • ✗Use same credential for prod and dev

💡Zero-trust for agents

Zero-trust means: each agent has a verifiable identity, access only to the minimum necessary (least privilege), each request is authenticated (without implicit trust due to being on the same network) and all access is audited. Deploy from the first agent in production — retrofitting is painful.
