🏗️ Multi-tenant architecture: true data isolation

Multi-tenancy is the ability of a single deployment to serve multiple clients withguaranteed insulationbetween them. A customer can never see or affect another's data — neither via bug nor via misconfiguration.

🗄️ Database Isolation Strategies

Row-Level Security (RLS) — Recommended

Best value for money

Postgres RLS: every query automatically filtered by tenant_id via bank policy. Developer cannot forget the filter — the bank guarantees it.

CREATE POLICY tenant_isolation ON agent_runs USING (tenant_id = current_setting('app.tenant_id')::uuid);

Schema per Tenant — Greater isolation

High operating cost

Postgres schema separated by tenant. Strong isolation, but migrations need to run N times (once per tenant). Makes it difficult to scale over hundreds of tenants.

Database per Tenant — Maximum isolation

Enterprise only

Separate database per tenant. Full isolation, compliance easier to demonstrate. High cost. It only makes sense for enterprise customers with specific regulatory requirements.

🔑 Tenant context propagation

The tenant_id must flow throughout the stack — from the HTTP request to the database. Any point where the tenant_id is lost is a potential data leakage.

# Middleware que injeta tenant_id em cada request
class TenantMiddleware:
    async def __call__(self, request, call_next):
        tenant = await get_tenant_from_jwt(request)
        # Propaga para banco via connection var
        await set_db_tenant_context(tenant.id)
        # Propaga para cache (Redis namespace)
        request.state.tenant_id = tenant.id
        request.state.redis_prefix = f"tenant:{tenant.id}:"
        response = await call_next(request)
        return response

⚠️The risk of the noisy neighbor

A tenant that consumes excessive resources degrades performance for all others. Implement: CPU limits per tenant, queue priority per plan (enterprise goes first), storage quotas and alerts when tenant exceeds 80% of the limit. Isolation isn’t just about data — it’s about resources, too.

💳 Billing by using tokens: metering and Stripe

Usage-based billing is the natural model for agentic SaaS — customers pay for what they consume. Implementing correctly requiresaccurate meteringper tenant and reliable integration with the billing platform.

📊 Metering pipeline

1 Every LLM call goes through the metering middleware: captures input + output tokens per call
2 Usage events sent to queue (Kafka): {tenant_id, agent_id, tokens, model, cost, timestamp}
3 Consumer aggregates real-time usage: Redis counters per tenant per billing period
4 Stripe Meter API: reports usage to Stripe every hour (or at the end of the billing cycle)
5 Stripe automatically generates invoice at the end of the cycle with usage breakdown

🔄 Token-to-credit conversion

Exposing tokens directly to the client creates confusion (what is a token?). Convert to credits or another understandable unit.

💡Metering guarantees

Metering must be: idempotent (duplicates do not charge twice), persistent (does not lose events if the service restarts), real-time (client sees current usage, not just at the end of the month) and auditable (client can question and you can prove). Incorrect metering is a legal liability.

⚡ Rate limiting and quotas: soft and hard limits

Rate limiting protects the SaaS from customers consuming excessive resources (accidentally or not) and ensures that all tenants have fair access to the system.Soft limitsalert;hard limitsblock.

⚡ Token Bucket Algorithm for agents

Token bucket is the most suitable algorithm for rate limiting agent APIs: it allows short bursts while limiting the average rate smoothly.

How it works

  • • Bucket starts full (ex: 1000 tokens)
  • • Each request consumes N tokens
  • • Tokens are replenished at a fixed rate (e.g. 100/min)
  • • Burst: can make quick requests until the bucket is empty
  • • After burst: returns to replenishment rate

Implementation with Redis

Soft limits (alerts)

  • ✓70% of limit: email for tenant admin user
  • ✓90% of limit: Tenant Slack + dashboard shows warning
  • ✓95% of the limit: plan upgrade suggestion
  • ✓No service interruption — just alerts

Hard limits (lock)

  • •100% limit: new executions blocked
  • •Returns 429 Too Many Requests with Retry-After
  • •Runs in progress are not interrupted
  • •Overage option: charge extra and unlock

📊 Admin panel: visibility for tenant and SaaS

There are two types of panel: thetenant dashboard(the client sees their own data) and theSaaS admin dashboard(you see all tenants). Both are critical — one for customer retention, the other for business operation.

📱 Tenant Dashboard

What the customer needs to see:

  • •Current vs. current period runs previous month
  • •Real-time token consumption + projection
  • •Execution history with status and duration
  • •Top agents by use and cost
  • •Quota alerts and optimization suggestions
  • •Estimated cost for the end of the cycle

🎛️ SaaS Admin Dashboard

What you (SaaS owner) need to see:

  • •MRR, ARR, churn, new tenants of the month
  • •Tenants close to the limit (upgrade opportunity)
  • •Inactive tenants (churn risk)
  • •Infra cost vs. revenue per tenant (LTV/CAC)
  • •Real-time operational errors and alerts
  • •Gross margin by plan and trend

💡Transparency as retention

Customers who see their data in real time have 40% lower churn than customers without visibility (internal B2B SaaS data). Cost and usage visibility builds trust and reduces invoice surprises — the biggest churn trigger in variable-usage SaaS.

🎯 Automated onboarding: time-to-value in minutes

Time-to-value is the main predictor of churn in B2B SaaS. If the customer doesn't experience value in the first few hours, the probability of converting from trial to paid drops drastically.Automated onboardingcompresses this time from days to minutes.

🔄 Automatic provisioning flow

1. Customer registers → automatically created tenant (UUID, subdomain, default plan settings)
2. Redis namespace created, database schema initialized, first-party integrations configured (Slack bot provisioned)
3. 3 pre-configured agents created (customer industry templates) ready to customize and run
4. In-app tutorial starts automatically: "Run your first agent in 5 minutes"
5. Activation Milestone: First Agent Run Successfully → Celebration Email Trigger + Upsell

💰 Pricing strategy: models and margin guarantee

Agentic SaaS pricing is complex because the cost is variable (tokens) and the perceived value is difficult to measure. The pricing model mustcapture value proportional to the benefitwhile ensuring positive operating margin across all plans.

By execution

Ex: R$0.50/execution

Advantages

  • • Predictable for the customer
  • • Easy to calculate ROI
  • • Clear scalability

Disadvantages

  • • Executions with very different costs
  • • Can be expensive for simple workflows
  • • Need to define what an “execution” is

By tokens (usage-based)

Ex: R$0.05/1k tokens

Advantages

  • • Reflects real cost directly
  • • Fair for all use cases
  • • Familiar to devs

Disadvantages

  • • Unpredictable for customers
  • • Difficult to budget
  • • "Anxiety pricing" — customers are hesitant to use

By outcome (outcome-based)

Ex: R$50/revised contract

Advantages

  • • Maximum alignment of incentives
  • • Customer pays proportional to the amount received
  • • Greater willingness to pay

Disadvantages

  • • Difficult to define and measure "result"
  • • Margin risk if cost varies a lot
  • • High contractual complexity

📐 Calculating minimum margin per plan

For each plan, calculate the maximum usage scenario and check for positive margin:

# Starter Plan: R$99/mês
# Quota: 500 execuções, 2M tokens
# Custo máximo se cliente usa 100% do plano:
custo_tokens = 2_000_000 * 0.000015  # Claude Haiku: $0.000015/token
custo_infra = 15  # storage, redis, compute
custo_stripe = 99 * 0.029 + 0.30     # Stripe fee
custo_total = custo_tokens + custo_infra + custo_stripe

# R$30 tokens + R$15 infra + R$3.17 Stripe = R$48.17
# Receita: R$99
# Margem bruta: R$50.83 (51%) — SAUDÁVEL ✓

# Se margem < 30%, revisar pricing ou quota

💡Recommendation: hybrid flat + usage

The most robust model for agentic SaaS: flat monthly fee (predictability for the customer, revenue baseline for you) + usage fee above the included quota (captures value from power user customers). This model combines predictability with revenue scalability.
