Case 1 — Azure AI Studio + Prompt Flow

☁️

Context: Retail Banking — Document Analysis Pipeline

Implementation with Azure AI Studio and Prompt Flow for contract processing

A midsize retail bank used Azure AI Studio with Prompt Flow to build an agentic mortgage contract analysis pipeline. Before the agent, 8 analysts processed 120 contracts/day with an error rate of 3.2%. The objective was to scale to 1,000 contracts/day while maintaining accuracy above 99%.

Prompt Flow architecture

# Prompt Flow flow — 5 sequential nodes
Node 1: document_intake— OCR + normalization of the contract PDF
Node 2: entity_extraction— GPT-4o extracts parts, values, deadlines, guarantees
Node 3: compliance_check— Claude validates against BACEN rules/resolution 4,966
Node 4: risk_scoring— proprietary risk model (Azure ML endpoint)
Node 5: human_review_gate— score < 0.85 → human queue; above → self-approval

Throughput increase

vs 96.8% pure human

Annual savings
Headcount reduction + errors

What worked well

  • ✓Prompt Flow with visual DAG accelerated iteration
  • ✓Azure AI Content Safety automatically blocked jailbreaks
  • ✓Human-in-the-loop on node 5 maintained BACEN compliance
  • ✓Native integration with Azure Active Directory (RBAC)

Challenges encountered

  • ✗High latency in contracts with many pages (>50s)
  • ✗Cost per contract 3x above forecast in the first month
  • ✗Hallucination in contracts with non-standard language (12%)
  • ✗Lock-in in Azure made it difficult to evaluate alternatives

Case 1 Lesson

The biggest learning was about the cost of hallucination in legal domain. A contract with a clause incorrectly extracted and approved by the agent generated a dispute of R$450k. The solution was to add a cross-validation node between two models (GPT-4o and Claude) — full agreement required for self-approval. The ~18% additional cost was justified by eliminating critical errors.

Case 2 — OpenAI Enterprise Controls

🤖

Context: Strategy Consulting — Research Agent with Assistants API

Multi-agent with Assistants API and File Search for market analysis

A large strategy consultancy implemented a multi-agent system using the OpenAI Assistants API with File Search to accelerate market research. Before: 2 analysts, 2 weeks per report. Objective: 1 analyst, 2 days per report.

Multi-Agent Architecture with Assistants API

# System of 3 specialized agents with handoff
Agent 1: ResearchAgent
tools: [file_search, web_search]
vector_store: 50k internal consultancy documents
output: research_brief.json with sources cited
Agent 2: AnalysisAgent
tools: [code_interpreter, file_search]
input: research_brief.json from Agent 1
output: financial_model.xlsx + analysis_narrative.md
Agent 3: ReportAgent
tools: [file_search]
input: outputs of agents 1 and 2
output: final report in PDF-ready format
# Orchestrator: checks quality at each handoff, rejects and re-tries if score < 0.8
5x
Time reduction
2 weeks → 2 days
40%
More sources per report
By vector store reading scale
$18k
Monthly API cost
High, but vs $120k in salaries

OpenAI Enterprise Controls Used

✓
Zero Data Retention— logs not used to train models
✓
SSO + SCIM— Okta integration, granular access control
✓
Audit Logs— who accessed which agent, when, with what data
✓
Usage Policies— blocking specific content categories

Case 2 Lesson

The biggest risk was confidentiality: a research agent with access to documents from multiple clients could unduly cross-reference information. The solution was Vector Stores per client (total isolation), with Orchestrator ensuring that each agent run only accesses the correct client's vector store. Implementing this retroactively took 3 weeks — it would have been trivial if planned from the beginning.

Case 3 — Databricks AI Governance

🔶

Context: National Retail — Demand Forecasting Agent

MLflow + Unity Catalog + Mosaic AI for supply chain agent

A retail chain with 800 stores used the Databricks stack (MLflow, Unity Catalog, Mosaic AI) to build a demand forecasting agent that automatically adjusts purchase orders. The agent combines proprietary ML models with LLMs for reasoning about external events (weather, holidays, campaigns).

Databricks Technical Stack

MLflow
  • • Tracking experiments and model versions
  • • Registration of hyperparameters and metrics
  • • Model Registry with stage (Staging/Production)
  • • Audit of who promoted which model when
Unity Catalog
  • • Unified data and model catalog
  • • Automatic lineage: data → feature → model → decision
  • • Fine access control (column-level security)
  • • Audit of who accessed which data
Mosaic AI Agent
  • • Agent with access to the ML + LLM reasoning model
  • • Tool calling for supplier APIs
  • • Human review before issuing orders over R$50k
  • • Feedback loop: buyer can correct forecast

Given:historic_sales (catalog.bronze.sales)
→Feature:demand_features (catalog.gold.features)
→ Model:demand_forecast_v3 (models:/demand_forecast/3)
→ Agent:supply_chain_agent (agents:/supply-chain/prod)
→ Decision:purchase_order_2025_03_15.json

Results

Governance in Action

•Automatic drift detection — alerts when accuracy drops >5%
•Explainability via SHAP — buyer sees top forecast features

Case 3 Lesson

Automatic lineage was the difference in an internal audit. The auditor questioned an atypical purchase order decision — in 30 seconds, the team presented the complete path: raw data → feature → model → agent → decision, including the exact version of the model and the data that fed it. This would be impossible without Unity Catalog. Companies that build lineageless AI systems are creating audit liabilities that will explode eventually.

Case 4 — Brazilian Credit Fintech (LGPD + BACEN)

🇧🇷

Context: Personal Credit Fintech — Risk Analysis Agent

Agentic system for granting credit under LGPD and BACEN Resolution 4,966

A Brazilian personal credit fintech (series B, 2M customers) needed to scale credit analysis while maintaining compliance with the LGPD (personal data protection), BACEN Resolution 4,966/2021 (which requires explainability of credit models) and the ANPD guidelines on automated data processing.

Specific Regulatory Challenges in Brazil

1
Art. 20 LGPD — Right to Human Review

The data subject has the right to human review of decisions made exclusively by automated means. The loan officer can NEVER be the final word — there must always be a human available for recourse.

2
BACEN 4,966 — Explainability of Models

Credit models must be explainable. For LLMs used in credit decisions: the score is not enough, you need the reasons. Fintech solved this with SHAP values ​​in ML models + LLM summary of reasons in natural language for the customer.

3
Data Retention and Consent

Data sent to external LLM APIs needs a clear legal basis. The fintech opted for an on-premise fine-tuned language model for sensitive data, using only external APIs for reasoning with synthetic/anonymized data.

Compliance Architecture

# Credit decision pipeline with BACEN/LGPD compliance
1. Collection:cpf_data, score_bureau, payment_history
2. Anonymization:remove PII before any external API
3. ML Model (on-premise):credit_score = predict(anonymized_features)
4. LLM on-premise (Llama 3.1):reasons = explain(credit_score, SHAP_values)
5. Decision:if score >= threshold: APPROVE else: REFUSE
6. Notification:email with score + reasons in pt-BR + link_recurso_humano
7. Registration:immutable audit_log for BACEN (10 years)

Case 4 Lesson

The biggest insight was that compliance and performance are not opposites — they are complementary when well designed. The explainability required by BACEN forced fintech to better understand its own model, leading to improvements that reduced defaults by 8%. Regulatory compliance has become a competitive advantage: regulated fintechs have access to products that non-regulated fintechs cannot offer.

Section 5: Critical Analysis
5

Critical Analysis — What the Cases Don’t Show

Success stories tend to omit failures, failed attempts and hidden costs. See what rarely appears in presentations:

The Promissory Note Problem

Agentic AI projects are often sold internally with ROI projections of 500–1000%. Real ROI takes 6–18 months to materialize. During this period, projects are canceled due to a lack of immediate results. Solution: define process metrics (not just results) from day 1.

The Cost of Retraining and Maintenance

Models degrade over time (data drift). The cost of maintaining an agent in production is typically 30–50% of the original development cost per year. This rarely appears in initial business plans, generating surprises in the budget.

Underrated Human Resistance

In all 4 cases above, the biggest blockage was not technical — it was human. Analysts who rejected agent recommendations even when correct, managers who reversed automated decisions out of "instinct", and teams who subtly sabotaged adoption. Change management consumes 30–40% of the total effort and is rarely budgeted adequately.

The Risk of Vendor Consolidation

Azure, OpenAI and Databricks create lock-in. Migrating from OpenAI to Anthropic or vice versa is technically simple (API switch), but migrating an entire platform of agents with Vector Stores, Fine-tuning and integrations is a 3–6 month project. Consider this when negotiating contracts and architecting systems.

Patterns of Failure

  • ✗Start with complex, high-risk use case
  • ✗Ignore change management until the project fails
  • ✗Do not measure baseline before the agent goes into production
  • ✗Promise full automation without HITL for edge cases
  • ✗Underestimating maintenance and monitoring costs

Patterns of Success

  • ✓Start with high-volume, low-risk use case
  • ✓Continuously measure and share wins
  • ✓Have a human "Agent Owner" responsible for each agent
  • ✓Treating compliance as a design constraint, not afterthought
  • ✓Plan for 3–5 iteration releases before scaling
Section 6: 10 Universal Lessons
6

10 Universal Lessons from Cases

1
Baseline Before Everything

Measure the current process (time, cost, errors) BEFORE implementing the agent. Without baseline, you can’t prove ROI — nor can you tune the agent.

2
Compliance is Design, Not Retrofitting

Resolving LGPD, BACEN or EU AI Act after the system is in production costs 10x more and may require complete redesign of the architecture.

3
HITL Is Not Weakness

Human-in-the-Loop at critical points is not a sign that the agent is not working — it is intelligent architecture. Eliminate HITL gradually with data, not in a hurry.

4
Tenant Isolation is Non-Negotiable

In multi-client systems, data separation between clients must be guaranteed by architecture (separate vector stores, RLS, schemas), not just by application logic.

5
Explainability Sells

Users adopt agents more when they understand the reasons for decisions. Explainability isn't just regulatory — it's product design that increases trust and adoption.

6
Real Cost = API + Infra + People + Maintenance

API cost is the visible part of the iceberg. Observability infrastructure, maintenance team, retraining and incidents account for 60–70% of total TCO.

7
Automatic Lineage Saves Audits

The ability to answer "why did this agent make that decision 6 months ago" doesn't come from good memory — it comes from automatic lineage implemented from the beginning.

8
Drift is Inevitable, Surprise is Avoidable

Every model degrades. Set up performance alerts from day 1. When the alert goes off, have a runbook ready — don't improvise in a crisis.

9
Small Wins, Big Trust

Quick wins in 2–4 weeks build political capital for more ambitious projects. Try to automate the loan officer on the first project and you will lose executive support.

10
The Human is the Feature, Not the Bug

The best agentic systems amplify humans, not replace them. The credit analyst who uses the agent to process 10x more contracts is more valuable than the unsupervised agent. Position it like this internally.
