🤔 Why and when to include humans in the process

Human-in-the-loop is not an admission that the agent doesn't work — it issmart design. Well-designed agentic systems use HITL strategically: at points where the cost of an error is greater than the cost of a human reviewing it.

🎯 When HITL is mandatory

1.
Irreversible actions

Data deletion, financial transfer, mass email sending, contract termination. If you can't undo it, human approves.

2.
Low model confidence

The agent returns a confidence score below the threshold defined for that type of decision. Uncertainty is a reason for escalation.

3.
Financial impact above the limit

Set clear limits: agent can approve payments up to R$500, above that requires human approval.

4.
Highly sensitive data

Any action involving RESTRICTED data (health, financial, legal) needs human review before execution.

HITL as a long-term strategy

Every decision that humans approve is training data. Over time:

  • ✓Approval rate without review rises (agent improves)
  • ✓Confidence threshold may rise gradually
  • ✓Financial limits of autonomy can expand
  • ✓HITL overhead decreases as agent matures

When NOT to use HITL

  • ✗Low-impact reversible actions (analysis, draft)
  • ✗High-frequency processes where latency matters
  • ✗Tasks where the agent has a history of 99%+ accuracy
  • ✗Informational notifications (without consequential action)

💡HITL as a trust differentiator

Paradoxically, agents with well-designed HITL have more internal adoption than fully autonomous agents. Stakeholders trust more when they know that there is human review at critical points. Use HITL not just as a technical defense, but as a business case for adoption.

🚦 Checkpoint types: approval, review and notification

Using the same type of checkpoint for all situations is inefficient.Calibrate the correct checkpoint typefor each point in the flow, it defines whether HITL adds value or just bureaucracy.

🔴

Approval Gate — blocking

The agent stops completely and awaits human approval. Without approval, do not perform the next action. No response timeout: automatic escalation.

Use when:irreversible action, high financial impact, access to restricted data, legal or regulatory decision

🟡

Review Gate — with review option

Human can approve (agent continues), request review (agent redoes with new context) or reject (agent aborts). Allows iteration without completely blocking.

Use when:quality matters but process can iterate, output has ambiguity that humans can clarify

🟢

Notification Only — informative

Agent executes and informs the human of what was done. No blocking. Human can reverse if necessary, but flow does not stop.

Use when:low-impact reversible action, information that the stakeholder needs to have but does not need to approve

🗺️ Mapping checkpoints in the agent flow

For each agent step, explicitly define: which type of checkpoint? What is the trigger? Who receives? What is the timeout? What's the climb?

🖥️ Approval UX: Clear context, not JSON dump

If the approval process is difficult or confusing, humans approve without reading — completely eliminating the value of HITL. THEApproval UXdefines whether the checkpoint is a real defense or just security theater.

📋 Anatomy of a good approval notification

APPROVAL REQUIRED

SDR Agent · Lead: TechCorp LTDA

⏱ 3h left

What the agent wants to do:

Send personalized commercial proposal toJoão Silva (CTO)of TechCorp LTDA with a value ofR$85,000/year.

Qualification

Score: 87/100

Size

240 employees

Interaction

3 emails, 1 call

Effective approval UX

  • ✓3-5 line executive summary of what the agent wants to do
  • ✓Pre-calculated relevant data (not raw data)
  • ✓Visible timer with deadline for approval
  • ✓One-click approve for mobile (most approve on mobile)
  • ✓Optional comment field when approving/rejecting

UX that causes approval without reading

  • ✗JSON dump of agent internal state
  • ✗Too frequent notifications (approval fatigue)
  • ✗Click-through approval process
  • ✗Insufficient context to make informed decision
  • ✗No indication of urgency or consequence for not responding

💡30 second rule

The human must be able to understand what the agent wants to do and make an informed decision in less than 30 seconds. If your approval process takes longer, the UX is wrong. Test with stopwatch: simulate approval and measure the time until an informed decision.

⏸️ Asynchronous approval: pause, notify, resume

Synchronous approval (agent is in a loop waiting) wastes resources and creates timeouts. The correct pattern ispause-notify-resume: agent persists state, sends notification and resumes from the exact point when approved.

🔄 Asynchronous approval flow

1

Persist full state

Agent serializes current state (context, tools already executed, next action pending) and persists in database. Can be restarted without losing progress.

2

Send notification with approval token

Sends Slack/email with unique approval link (signed JWT token). The link contains the approval_request_id that identifies which decision is being made.

3

Human approves via link

Human clicks on the link, sees formatted context, makes decision (approve/reject/request_revision). Action registered with identity, timestamp and reasoning.

4

Webhook resumes agent

System sends webhook to the agent with the decision. Agent loads persisted state, applies the decision and continues execution from the exact point where it left off.

⏰

Timeout: automatic escalation

If there is no response in N hours: re-notify, escalation to supervisor. If still no response: self-approval (for low-risk actions) or self-rejection with warning.

📋 Audit of human decisions: who approved what

Recording human decisions in the context of HITL is not just compliance — it isthe dataset that will train the system to need approval less and less. Each approval is an example of "what the agent proposed was correct". Every rejection is an example of “here the agent made a mistake”.

📊 Audit trail schema of human decisions

Mandatory fields:

  • •approval_id: unique ID of the decision
  • •approver_id: who approved (verified by SSO)
  • •decision: approve/reject/request_revision
  • •timestamp: with UTC timezone
  • •context_snapshot: what the agent proposed

Fields for continuous improvement:

  • •reasoning: why approved/rejected
  • •time_to_decide: how long it took
  • •modifications: what was changed (if revision)
  • •confidence_reported: was the human safe?
  • •outcome: subsequent result of the decision

📊 Using audit to improve the agent

  • •High rejection rate in a certain type of action = agent is systematically making mistakes in this case
  • •High approval time = insufficient context or bad UX for this type of decision
  • •Reasoning for recurring rejection = pattern that can become a new rule for the agent
  • •Approval rate rising = agent may have expanded autonomy in this category

⚠️Immutable audit trail is a compliance requirement

To comply with SOX, BACEN and similar regulations, the audit trail must be immutable — it cannot be retroactively modified or deleted. Use append-only storage: AWS CloudTrail, Kafka with retention configured, or database with triggers that block UPDATE/DELETE.

📊 HITL metrics: measure to improve

HITL metrics are thesign of maturity of the agentic system. Growing approval rate means agent learning. High approval time means bad UX. Frequent escalation means very low trust threshold.

Approval Rate

87%

Checkpoints approved without review request

Goal: grow monthly. Indicates which agent is getting more hits.

Time-to-Approve

18 min

Average human approval time

Target: below the defined SLA. High indicates poor UX or insufficient context.

Escalation Rate

4%

Checkpoints escalated by timeout

Target: below 5%. High indicates problems with the notification process.

💡HITL as a path to autonomy

The long-term goal isn't to have more HITL — it's to have HITL only where it really matters. Measure approval rate by action category. When a category reaches 95%+ approval for 3 consecutive months, consider removing the checkpoint and moving to notification-only. Well-managed HITL becomes less and less necessary.
