========== TOPIC 1 ==========
What is it

What is it

O Claude Codeis an official Anthropic CLI that turns Claude into a development agent that operates directly on your terminal. Unlike editor plugins, it has full access to the file system, can execute commands, create branches, run tests, and navigate code like a senior developer — all from natural language instructions.

Why use

Why use

  • +Perform complex multi-file tasks without constant supervision
  • +Natively integrates with git, GitHub CLI, pytest and any terminal tool
  • +Persistent context viaCLAUDE.md— the agent "knows" the project rules
  • +Configurable permissions per project, with granular allow/deny lists
How to configure

How to configure

Install globally via npm and authenticate with your Anthropic key:

# 1. Install Claude Code globally
npm install -g @anthropic-ai/claude-code

# 2. Verify installation
claude --version

# 3. Authenticate (opens browser for OAuth)
claude auth login

#4. Or use API key directly
export ANTHROPIC_API_KEY="sk-ant-..."

# 5. Enter the project directory and start
cd meu-projeto
claude

# 6. Non-interactive (headless) mode
claude -p "Adicione testes para o módulo de autenticação"

Recommended basic structure for any project:

meu-projeto/
├── CLAUDE.md                  # Agent memory and rules
├── .claude/
│   ├── settings.json          # Permissions and hooks
│   └── skills/
│       ├── fix-issue/
│       │   └── SKILL.md       # Customized skill
│       └── deploy/
│           └── SKILL.md
└── src/                       # Project code
========== TOPIC 2 ==========

What is it

O CLAUDE.mdis a Markdown file in the project root that Claude Code automatically reads at the start of each session. It works like theagent persistent memory— contains the project context, code conventions, useful commands, rules of behavior, and any information that Claude needs to know to work correctly in the repository.

The difference from a regular prompt is that theCLAUDE.mdit is specific to the project, it is in the repository (and can be versioned), and it is read automatically — the developer does not need to repeat the context in each conversation.

Why use

  • +Eliminates the need to explain the project from scratch each session
  • +Ensures that Claude follows team conventions (snake_case, docstrings, etc.)
  • +Prevent unwanted actions with explicit rules ("never commit to main")
  • +Can be shared across the team via version control

How to configure

Recommended sections for a complete CLAUDE.md:

#CLAUDE.md

## Project Context
API REST com FastAPI + PostgreSQL para gestão de tarefas.
Stack: Python 3.11, SQLAlchemy 2.0, Pydantic v2, pytest.

## Main Commands
- Rodar: `uvicorn app.main:app --reload`
- Testes: `pytest --cov=app`
- Lint: `ruff check app/`

## Conventions
- snake_case para funções e variáveis
- Type hints obrigatórios em todas as funções
- Docstrings no formato Google Style

## Claude's Rules
- NUNCA commitar em main ou develop
- Sempre criar branch: feat/, fix/, refactor/
- Rodar pytest antes de qualquer commit
- Nunca expor SECRET_KEY ou DATABASE_URL

## Architecture
app/routers/   → endpoints por recurso
app/models/    → SQLAlchemy models
app/schemas/   → Pydantic schemas
app/services/  → lógica de negócio
========== TOPIC 3 ==========

What is it

Claude Code's permissions system controls exactly which commands the agent can execute autonomously, which require user confirmation, and which are completely blocked. The settings are in.claude/settings.jsonand can be defined per project.

Why use

  • +Prevents accidental destructive actions (rm -rf, git push --force)
  • +Frees up flow for safe, repetitive actions without manual confirmation
  • +Configurable by project — a production project can have stricter rules
  • +Three modes:default(asks everything),self-edit(editions without confirmation),bypass-permissions(CI/CD)

How to configure

// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(git add:*)",        // git add allowed
      "Bash(git commit:*)",
      "Bash(pytest:*)",          // run tests without confirming
      "Bash(pip install:*)",
      "Bash(python:*)"
    ],
    "deny": [
      "Bash(git push --force:*)", // blocked permanently
      "Bash(rm -rf:*)",
      "Bash(sudo:*)"
    ]
  }
}

Execution modes via flag:

# Standard mode (question before unconfigured actions)
claude

# Auto-edit mode (edits files without confirmation)
claude --auto-edit

# Headless mode for CI/CD (full bypass, use with caution)
claude --dangerously-skip-permissions -p "rodar suíte de testes"
========== TOPIC 4 ==========

What is it

Hooks are shell commands that Claude Code automatically executes in response to session lifecycle events. They allow you to integrate the agent with external systems, perform security checks, notify teams and automate recurring tasks without manual intervention.

PreToolUse

Runs before any tool (Bash, Edit, etc.)

PostToolUse

Executes after a tool is used successfully

SessionStart

Runs once when starting a new Claude session

Notification

Executes when Claude generates a notification for the user

Why use

  • +Automatically run lint after every Python file edit
  • +Send Slack notifications when agent completes a long task
  • +Record audit logs of all executed commands
  • +Block dangerous commands with custom check before execution

How to configure

// .claude/settings.json — hooks section
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",        // applies to all Bash
        "hooks": [{
          "type": "command",
          "command": "echo '[Audit] Bash: $CLAUDE_TOOL_INPUT'"
        }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit",         // after editing file
        "hooks": [{
          "type": "command",
          "command": "ruff check app/ --quiet"
        }]
      }
    ],
    "Notification": [
      {
        "matcher": "*",
        "hooks": [{
          "type": "command",
          "command": "notify-send 'Claude Code' '$CLAUDE_NOTIFICATION'"
        }]
      }
    ]
  }
}
========== TOPIC 5 ==========

What is it

Skills areagent macrosactivated by commands in the format/nome-da-skill. Each skill is defined by a fileSKILL.mdinside.claude/skills/nome/and describes in natural language the steps Claude must follow — like an execution recipe.

The difference to a simple prompt is that skills are reusable, versionable, shareable across the team and can encapsulate complex multi-step flows.

Why use

  • + /fix-issue 123— automatically fixes a GitHub issue
  • + /commit— creates commit with message generated from diff
  • + /review-pr 45— reviews a PR and suggests improvements
  • + /deploy staging— runs deployment pipeline for staging

How to configure

Structure of a SKILL.md for the skill/fix-issue:

# Skill: fix-issue

## Usage
/fix-issue <numero-da-issue>

## Steps
1. Ler issue: `gh issue view <numero> --json title,body`
2. Identificar arquivos relacionados com Grep/Glob
3. Criar branch: `git checkout -b fix/issue-<numero>`
4. Implementar correção com Edit/Write
5. Rodar testes: `pytest`
6. Commitar: `git commit -m "Fix: resolve #<numero>"`

## Rules
- Nunca modificar main diretamente
- Se testes falharem, corrigir antes de commitar
- Mencionar o número da issue na mensagem de commit

## Tools
Bash, Read, Edit, Write, Grep, Glob

Location in the project:

.claude/skills/
├── fix-issue/
│   └── SKILL.md      → activated with /fix-issue
├── commit/
│   └── SKILL.md      → activated with /commit
└── review-pr/
    └── SKILL.md      → activated with /review-pr
========== TOPIC 6 ==========

What is it

MCP (Model Context Protocol)is an open protocol that allows Claude to connect to external data sources and tools in a standardized way. Already thesubagentsare child instances of Claude invoked by the toolTask— allow you to parallelize work, such as having the main agent orchestrate multiple subagents working on different parts of the code simultaneously.

Why use

  • +MCP connects Claude to databases, APIs, Slack, Jira, without custom code
  • +Subagents allow parallel work — 4 modules being implemented at the same time
  • +Worktrees isolate each subagent in its own directory — no file conflicts
  • +Headless mode (claude -p "...") allows full automation via scripts and CI/CD

How to configure

Connect an MCP server to Claude Code:

# Add MCP server to Claude
claude mcp add postgres-server \
  --command "npx @anthropic/mcp-server-postgres" \
  --env DATABASE_URL="$DATABASE_URL"

# List configured MCPs
claude mcp list

# Use in an interactive session (Claude now has access to the bank)
claude
# "List the 10 most recent database tasks"

Use the Task tool to create parallel subagents:

# Prompt for the main agent to orchestrate subagents
claude -p "
Implemente as seguintes features em paralelo usando subagentes:

1. Subagente A: Criar endpoint POST /api/v1/tasks com validação
2. Subagente B: Adicionar testes para o endpoint de autenticação
3. Subagente C: Atualizar documentação do README

Use a ferramenta Task para cada subagente.
Aguarde todos completarem antes de fazer o commit final.
"

Working with worktrees for isolation:

# Create worktree to work on isolated feature
git worktree add .claude/worktrees/feat-auth feat/auth-refresh

# Claude Code automatically creates worktrees with /worktree
/worktree feat-notifications

# List active worktrees
git worktree list

Complete Example: CLAUDE.md

File contentsCLAUDE.mdcreated for this demo project.

CLAUDE.md markdown 
# CLAUDE.md — Task Management REST API

## Project Context
API REST com FastAPI + PostgreSQL para gestão de tarefas.
Stack: Python 3.11, SQLAlchemy 2.0, Pydantic v2, pytest.
Repositório: github.com/nmaldaner/task-api

## Development Commands
- Instalar: `pip install -r requirements.txt`
- Rodar: `uvicorn app.main:app --reload --port 8000`
- Testes: `pytest --cov=app --cov-report=html`
- Migrations: `alembic upgrade head`
- Lint: `ruff check app/`

## Code Conventions
- snake_case para variáveis e funções
- PascalCase para classes e schemas Pydantic
- Type hints obrigatórios em todas as funções públicas
- Docstrings no formato Google Style
- Imports: stdlib > third-party > local (separados por linha em branco)

## Claude Code Rules
- NUNCA commitar diretamente em main ou develop
- Sempre criar branch: feat/, fix/, refactor/, docs/
- Rodar pytest antes de qualquer commit
- Nunca expor SECRET_KEY ou DATABASE_URL em código
- Nunca usar git push --force sem confirmação explícita

## Architecture
app/routers/   → endpoints organizados por recurso
app/models/    → SQLAlchemy models (tabelas do banco)
app/schemas/   → Pydantic schemas (request/response)
app/services/  → lógica de negócio
tests/         → testes com pytest + fixtures

## Important Environment Variables
DATABASE_URL, SECRET_KEY, ALGORITHM, ENVIRONMENT, DEBUG

## Top URLs
- Swagger: http://localhost:8000/docs
- Health: http://localhost:8000/health
- API: http://localhost:8000/api/v1/
========== BLOCK SETTINGS.JSON EXAMPLE ==========

Full Example: settings.json

Configuration file in.claude/settings.jsonwith permissions and hooks configured.

.claude/settings.json json 
{
  "permissions": {
    // Commands allowed without confirmation
    "allow": [
      "Bash(git add:*)",
      "Bash(git commit:*)",
      "Bash(git checkout:*)",
      "Bash(pytest:*)",
      "Bash(pip install:*)",
      "Bash(python:*)"
    ],
    // Commands always blocked
    "deny": [
      "Bash(git push --force:*)",
      "Bash(rm -rf:*)",
      "Bash(sudo:*)"
    ]
  },
  "hooks": {
    // Execute before any Bash command
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "echo '[Hook] Checking security...'"
      }]
    }],
    // Execute after editing a file
    "PostToolUse": [{
      "matcher": "Edit",
      "hooks": [{
        "type": "command",
        "command": "echo '[Hook] File edited. Consider running tests.'"
      }]
    }]
  },
  "env": {
    "ANTHROPIC_API_KEY": "${ANTHROPIC_API_KEY}",
    "ENVIRONMENT": "development"
  }
}
