====================================== TOPIC 1: WHAT IS THE CODEX CLI ======================================
Subsection: What is it
A

What is it

O OpenAI Codex CLIis a terminal agent launched by OpenAI in April 2025. It is an open-source tool written in TypeScript (with parts in Rust) that runs directly in your terminal and allows you to interact with OpenAI models — such aso4-mini, gpt-4o e o3— to perform software development tasks autonomously.

Unlike tools like Copilot (which just completes code) or ChatGPT (which responds in a web interface), Codex CLI acts directly on your file system: reads files, writes code, runs shell commands, creates git branches, and navigates the project like a real developer sitting next to you.

Subsection: Why use
B

Why use

  • ▸ Native integration with OpenAI:direct access to reasoning models like o3 and o4-mini, especially good at programming and complex logic.
  • ▸ Open source:code available on GitHub, auditable, contributory and hostable on our own infrastructure.
  • ▸ Granular approval modes:you control exactly the agent's level of autonomy — from just suggestions to fully autonomous execution.
  • ▸ Ideal for CI/CD:mode--headlessallows you to integrate Codex into automated pipelines without human intervention.
  • ▸ Configurable sandbox:the agent can be restricted to only reading files, or freed to execute any shell commands as needed.
Subsection: How to configure (difference from Claude Code)
C

Codex CLI vs Claude Code — when to use each

Both are endpoint agent tools, but each has distinct strengths. The choice depends on your ecosystem, preferred model and desired level of autonomy.

◆Use Codex CLI when:

  • ·Already using OpenAI API and want consistency
  • ·Need o3/o4-mini for deep reasoning
  • ·Want to control the tool's source code
  • ·Automate CI/CD tasks with --headless
  • ·Prefer configuration via TOML

◆Use Claude Code when:

  • ·Want the best integrated UX experience
  • ·Need very long window context
  • ·Uses the Anthropic ecosystem (Claude Opus 4)
  • ·Prefer CLAUDE.md for persistent context
  • ·Works a lot with analyzing large files
====================================== TOPIC 2: INSTALLATION AND SETUP ======================================
What is it
A

What is the setup process

Installing the Codex CLI is done via npm and only requires an OpenAI API key. The complete process takes less than 5 minutes: install the package globally, export the environment variable with the key and start the first session within the project directory.

Why use
B

Why the setup is simple

Because it is distributed as an npm package, Codex CLI does not require additional dependencies beyond Node.js 18+. Authentication is done via OpenAI's standard environment variable (OPENAI_API_KEY), which means that any project that already uses the OpenAI API already has authentication configured.

How to configure
C

How to install and configure

1. Install globally via npm:

terminal 
$ npm install -g @openai/codex

# Check installation
$ codex --version
1.0.0

2. Configure authentication:

~/.bashrc or ~/.zshrc 
# Add to your shell profile file
export OPENAI_API_KEY="sk-proj-..."

# Reload profile
$ source ~/.bashrc

3. First run (interactive mode):

terminal 
# Enter project directory
$ cd meu-projeto

# Start interactive session (default mode: suggest)
$ codex

# Or pass a task directly
$ codex "add email validation on POST /users endpoint"

# With specific approval mode
$ codex --approval-mode auto-edit "fix broken tests"

4. Minimum structure of a project with Codex:

file structure

├── AGENTS.md          # Persistent context for Codex
├── .codex/
│   └── config.toml   # Codex CLI configuration

├── package.json

====================================== TOPIC 3: AGENTS.MD ======================================
What is it
A

What is AGENTS.md

O AGENTS.mdis the Codex CLI persistent context file — automatically injected at the start of every session. It is the direct equivalent ofCLAUDE.mdfrom Claude Code, but with some important differences in semantics and recommended structure.

The file serves as a permanent "briefing": it describes the project, code conventions, available commands, rules of behavior and any context that the agent needs to work effectively without having to re-read the entire project every session.

Why use
B

Why AGENTS.md is essential

Without a well-written AGENTS.md, the agent needs to infer the project context each session, which increases token costs, reduces response accuracy, and can lead to inconsistent behavior. With a complete AGENTS.md:

  • ▸The agent immediately knows the conventions without having to analyze existing code
  • ▸Test and build commands are immediately available — the agent can run tests correctly
  • ▸Security rules (never modify X, always create branch Y) are respected from the beginning
  • ▸The explained architecture prevents the agent from placing code in the wrong place
How to configure
C

Mandatory sections and how to write

Mandatory sections

  • Project context (what is it, stack)
  • Development commands
  • Test commands
  • Code conventions
  • Rules of behavior
  • Folder architecture/structure

Differences: AGENTS.md vs CLAUDE.md

  • · Name:AGENTS.md (Codex) vs CLAUDE.md (Claude Code)
  • · Config:AGENTS.md referenced in config.toml
  • · Scope:AGENTS.md allows multiple context files via extra_context
  • · Format:both use standard Markdown

====================================== TOPIC 4: CONFIG.TOML — MODES ======================================
What is it
A

What is config.toml

The file.codex/config.tomlis the core Codex CLI configuration for the project. Defines which template to use, how the agent should ask for approval before acting, which shell commands are allowed, and which files should never be automatically modified.

Why use
B

The three approval modes

suggest

The agent only suggests changes. You see the diff and decide if it applies. Ideal for exploration and learning. Slower, safer.

auto-edit

The agent edits files automatically, but asks for approval before executing shell commands. Balance between speed and control.Recommended for daily development.

full-auto

The agent acts without asking for approval for anything. Ideal for CI/CD and headless automation. Use only with restricted sandbox and blocked commands configured.

How to configure
C

How to configure — complete config.toml

Network sandbox modes:

read-only

Only reads files. Doesn't write, doesn't execute anything.

workspace-write

Writes only to the current workspace. Blocks access to the system.

danger-full-access

Full access to the system. Use only in insulated containers.

.codex/config.toml 
# Codex CLI — Project Configuration

[model]
provider = "openai"
name = "o4-mini"       # Alternatives: "gpt-4o", "o3"
temperature = 0.2

[approval]
mode = "auto-edit"     # suggest | self-edit | full-auto
require_approval = [
  "*.env", ".env.*",
  "package.json",
  "Dockerfile"
]

[sandbox]
network = "workspace-write"
allow_shell = ["npm test", "npm run lint", "git status"]
deny_shell = ["npm publish", "git push --force", "sudo"]

[context]
agents_file = "AGENTS.md"
extra_context = ["README.md"]
max_context_files = 10

[telemetry]
enabled = false
====================================== TOPIC 5: HEADLESS MODE AND AUTOMATION ======================================
What is it
A

What is headless mode

Codex CLI's headless mode allows you to run the agent without an interactive interface, receiving the task as an argument and returning the result via stdout. It is activated with the flag--headless(or-H) and combined with--approval-mode full-autofor fully autonomous execution.

This opens up a huge range of possibilities: Codex can be invoked in CI/CD pipelines to automatically review code, generate changelogs, correct code style, create documentation or even open pull requests based on GitHub issues.

Why use
B

Why use automation with Codex

  • ▸ Automatic code review:Codex analyzes each PR and comments on quality, security, and convention issues before a human reviews it.
  • ▸ Linting auto-fix:Instead of just reporting errors, the agent can automatically fix them in a separate commit.
  • ▸ Test generation:Every time a new feature file is created, a CI job can ask Codex to generate the corresponding tests.
  • ▸ Automatic changelog:When merging a branch, Codex analyzes the commits and generates a semantic changelog entry.
How to configure
C

How to integrate with GitHub Actions

.github/workflows/codex-review.yml 
name: Codex Code Review

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  codex-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Instalar Codex CLI
        run: npm install -g @openai/codex

      - name: Revisar PR com Codex
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
        run: |
          codex --headless \
            --approval-mode full-auto \
            "Please review the changes to this PR. Check:\
             1) AGENTS.md \ code conventions
             2) Possible bugs or security issues \
             3) Adequate test coverage \
             Create the .codex-review.md file with your feedback."

      - name: Comentar review no PR
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('.codex-review.md', 'utf8');
            await github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: review
            });
headless use via command line 
# Basic headless mode
$ codex --headless "list all endpoints without tests"

# With full-auto (without approval)
$ codex -H --approval-mode full-auto \
  "fix all ESLint errors and commit"

# With output to file
$ codex -H "generate changelog of last 10 commits" \
  > CHANGELOG_DRAFT.md

# In pre-commit script

====================================== TOPIC 6: COMPARATIVE ======================================
What is it
A

What are each tool

Both are tools foragentic codingthat run in the terminal, read and write files, execute commands and interact with your repository autonomously. The fundamental difference is in the model ecosystem: Claude Code uses Anthropic models (Claude Sonnet, Claude Opus), while Codex CLI uses OpenAI models (GPT-4o, o3, o4-mini). Each ecosystem has its strengths.

Why use
B

Why comparing matters

Most projects can use both tools in a complementary way. Knowing the strengths of each one allows you to choose the right tool for each type of task. For example: using Codex CLI with o4-mini for complex refactorings that require deep thinking, and Claude Code for quick code analysis with a long context window.

- Add tests for new code
====================================== CONFIG.TOML EXAMPLE BLOCK ======================================

Example: commented .codex/config.toml

complete configuration
.codex/config.toml 
# Codex CLI — Project Configuration
# Documentation: https://github.com/openai/codex

[model]
provider = "openai"
name = "o4-mini"             # Main model
# Alternatives: "gpt-4o", "o3", "o4-mini"
temperature = 0.2                # Low = more deterministic

[approval]
# Modes: "suggest" | "auto-edit" | "full-auto"
mode = "auto-edit"
# Files that always need manual approval
require_approval = [
  "*.env", ".env.*",
  "package.json", "package-lock.json",
  "Dockerfile", "docker-compose*.yml"
]

[sandbox]
# Modes: "read-only" | "workspace-write" | "danger-full-access"
network = "workspace-write"
allow_shell = [             # Commands allowed without approval
  "npm test", "npm run lint", "npm run build",
  "git status", "git diff", "git log --oneline -10"
]
deny_shell = [              # Commands always blocked
  "npm publish", "git push --force",
  "rm -rf", "sudo"
]

[context]
agents_file = "AGENTS.md"     # Context read in every session
extra_context = [
  "README.md", "ARCHITECTURE.md"
]
max_context_files = 10

[telemetry]
enabled = false              # Disable data sending
