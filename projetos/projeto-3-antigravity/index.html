TOPIC 1: What is Antigravity
Tabs
Tab: What is it

O Antigravityis an integrated development environment (IDE) created by Google with a radically different premise from traditional editors: the AI agent is themain operator, and the human developer is thesupervisor.

While tools like Cursor and GitHub Copilot add AI as aassistantto the traditional human flow, Antigravity reverses this logic. The agent plans, executes, tests and verifies — the human approves or redirects.

Cursor / VSCode + Copilot
  • • Human as operator
  • • AI as autocomplete
  • • Line-by-line suggestions
  • • Limited context
Claude Code/Codex CLI
  • • Agent at the terminal
  • • Repository context
  • • Command execution
  • • No integrated IDE
Antigravity
  • • Agent as operator
  • • IDE + terminal + browser
  • • Plan → Execute → Check
  • • Native Google Cloud

Antigravity unifies editor, terminal, browser and external tools in a single environment, with the Gemini agent orchestrating all parts via MCP (Model Context Protocol).

Tab: Why use

Antigravity represents a paradigm shift in software development. Instead of the developer writing code and using AI to autocomplete, the agent takes on the role of executor while the human sets goals.

1
10x development speed
The agent implements complete features — from backend to frontend — while you review at a high level.
2
Native integration with Google Cloud
Firebase, Cloud Run, Vertex AI and Google Workspace integrated directly into the environment, without extra configuration.
3
MCP as an extensibility standard
Any tool that supports MCP can be integrated with Antigravity, creating an extensible ecosystem.
4
Granular permissions control
You define exactly what the agent can do automatically and what requires explicit human approval.
Tab: How to configure

The entry point for any project in Antigravity is the fileworkspace.json. It defines the model, available tools, workflows and permissions.

Basic workspace.json structure 
{
  "name": "my-project",
  "agent": {
    "model": "gemini-2.0-flash-exp",
    "context_files": ["CONTEXT.md"],
    "auto_approve": false
  },
  "permissions": {
    "file_write": "ask",
    "shell_execute": "ask"
  }
}

See topic 2 for the complete and detailed installation configuration.

TOPIC 2: Installation and Configuration

Antigravity installs as a native application for macOS, Windows and Linux. Unlike editor plugins, it is a complete, self-contained environment that comes with integrated MCP server, runtime for tools and terminal sandbox.

System Requirements
  • •macOS 13+, Windows 11, Ubuntu 22.04+
  • •Node.js 20 LTS (managed by installer)
  • •8GB RAM minimum (16GB recommended)
  • •Google Account (required for authentication)
  • •Internet connection (cloud model)
What's included
  • •Code editor with syntax highlighting
  • •Terminal integrated with sandbox
  • •Built-in browser (Chromium)
  • •Local MCP Server
  • •Action approval panel

Native installation ensures that Antigravity has full access to the filesystem, system processes and network devices with granular permissions control — something editor extensions can't offer.

Key advantage: unified environment

By having an editor, terminal, browser and MCP in the same process, the agent can observe the results of their actions in real time — write code, compile it, open it in the browser and check the visual result, all without leaving the context.

Google Account as identity

Authentication via Google Account allows Antigravity to directly access your GCP resources, Firebase and Google Workspace without having to manually configure credentials.

TOPIC 3: CONTEXT.md

O CONTEXT.mdis the Gemini agent's persistent memory file in Antigravity. Equivalent toCLAUDE.mdby Claude Code, it is automatically loaded at the start of each session and provides the agent with the context needed to make correct decisions.

CLAUDE.md (Claude Code)
  • • Loaded by Claude automatically
  • • Focus on commands and conventions
  • • Simple Markdown
  • • No workflow section support
CONTEXT.md (Antigravity)
  • • Support for the Workflows section
  • • Referenced in workspace.json
  • • Multiple context files
  • • Domain and glossary section

You can have multiple context files (ex:CONTEXT.md e ARCHITECTURE.md), all listed incontext_filesin workspace.json.

Without CONTEXT.md, the agent starts each session without knowledge of the project — it needs to explore the code to understand the stack, conventions and business rules. With CONTEXT.md, it starts already informed and makes correct decisions from the first prompt.

Reduces convention errors

The agent knows that you use camelCase, that you prefer Fastify over Express, and that commits should follow Conventional Commits — without needing to learn this on the first failure.

Sets security limits

The restrictions section ensures that the agent never commits directly to main, never stores credentials and always asks for approval before deploys.

Provides domain context

The business glossary and domain rules prevent the agent from implementing incorrect logic due to not understanding the application context.

CONTEXT.md must have well-defined sections. The most important are:

1
Complete technology stack
List all technologies, versions and the rationale for each choice. The agent will use this to make implementation decisions.
2
Code conventions
Naming, commit style, typing rules, testing patterns — the more detailed, the fewer revisions you will need to make.
3
Mandatory restrictions
What the agent should NEVER do (commit to main, log credentials, deploy without approval). This is the most security critical section.
4
Domain Glossary
Define business-specific terms to avoid ambiguity. Ex: what is a "Sprint", what does "archived" mean in a Task.
Tip: update CONTEXT.md along with the code

Whenever you change the stack, add a rule or discover problematic agent behavior, update CONTEXT.md. Treat it as living documentation.

TOPIC 4: MCP in Antigravity

O Model Context Protocol (MCP)is the open standard that allows the agent communicate with external tools in a standardized way. In Antigravity, the MCP is the engine main extensibility — everything the agent can do other than edit text is done via MCP servers.

Available MCP Servers
📁
filesystem
Reading and writing files
⬡
github
PRs, issues, branches
🔵
google-workspace
Drive, Calendar, Gmail
🔥
firebase
Firestore, Auth, Hosting
⬛
postgres
Queries and migrations
🌐
browser
Web automation (Playwright)

MCP solves the fundamental problem of AI agents: how to give secure and controlled access to external resources. Instead of giving the agent unrestricted access to the system, each MCP server exposes only the necessary operations.

Principle of least privilege

The filesystem MCP server can be configured with--root .to limit access to the project directory only, never the entire filesystem.

Standardized interface

The agent uses the same API to interact with any tool. Adding a new MCP server does not require changes to the agent.

Full audit

Each MCP call is logged, allowing you to audit exactly what the agent did, when, and to which file or resource.

MCP servers are configured in the sectiontools.mcp_serversfrom workspace.json:

workspace.json — mcp_servers section 
"mcp_servers": [
  {
    "name": "filesystem",
    "command": "npx @modelcontextprotocol/server-filesystem",
    "args": ["--root", "."]
  },
  {
    "name": "github",
    "command": "npx @modelcontextprotocol/server-github",
    "env": {
      "GITHUB_TOKEN": "${GITHUB_TOKEN}"
    }
  },
  {
    "name": "google-workspace",
    "command": "npx @google/mcp-server-workspace",
    "auth": "google-oauth"
  }
]
Environment variables on MCP servers

Use the syntax"${NOME_DA_VAR}"to reference system environment variables in workspace.json. Never hardcode tokens or secrets directly into the file.

TOPIC 5: Automated Workflows

Os WorkflowsThese are sequences of pre-defined steps that the agent executes when receiving a certain type of task. Antigravity implements the standardPlan → Execute → Verifyas a fundamental development loop.

Loop visualization
Antigravity fundamental loop
PLAN
Analyze and describe
→
EXECUTE
Implements
→
VERIFY
Test and confirm
→
APPROVE
Human reviews
Workflow: feature
1plan — map changes
2implement — write code
3test — create and run tests
4review — wait for approval
Workflow: bugfix
1analyze — reproduce bug
2fix — implement fix
3verify — confirm solution

Structured workflows prevent a common problem in agents: impulsive execution. Without a planning phase, the agent may start implementing the wrong approach and only notice the problem after many changes.

The Plan phase as a checkpoint

Before writing code, the agent describes what it intends to do. You can stop here if the approach is wrong, saving time.

The Verify phase as a guarantee of quality

The agent does not consider a task complete until it runs the tests. This forces you to create tests along with the code, not after.

The Approve phase as human control

Even withauto_approve: false, you maintain visibility and control of each change before it is committed.

Workflows are defined in workspace.json and activated by commands in the agent chat:

workspace.json — workflows section 
"workflows": {
  "feature": {
    "steps": ["plan", "implement", "test", "review"]
  },
  "bugfix": {
    "steps": ["analyze", "fix", "verify"]
  }
}
Chat with agent — trigger examples 
# Activate feature workflow
@agent implement feature: authentication with Google OAuth

# Activate bugfix workflow
@agent fix bug: users unable to log out of Safari

# Activate deployment workflow
@agent deploy to staging
TOPIC 6: Integration with Google Cloud

As it is a Google product, Antigravity has native integration with the entire Google Cloud ecosystem. The agent can interact with GCP services using your Google Account credentials without additional configuration.

Firebase
  • • Firestore: queries and writes via MCP
  • • Authentication: manage users
  • • Hosting: automatic frontend deployment
  • • Functions: deploy cloud functions
  • • Storage: uploading and downloading files
Google Cloud Platform
  • • Cloud Run: container deployment
  • • Vertex AI: ML models and embeddings
  • • Cloud SQL: managed database
  • • Secret Manager: manage credentials
  • • Cloud Build: CI/CD pipelines
Vertex AI
  • • Access to Gemini 1.5 Pro and Flash
  • • Embeddings for semantic search
  • • Fine-tuning of customized models
  • • AI Pipelines for batch processing
Google Workspace
  • • Sheets: read and write spreadsheets
  • • Docs: generate documentation
  • • Drive: manage files
  • • Calendar: synchronize events

Native integration eliminates friction between development and deployment. Instead of exiting the IDE to access the Firebase console or Cloud Run, the agent does this directly from Antigravity while you watch.

WORKSPACE.JSON COMMENTED

workspace.json — Complete reference

Antigravity configuration file with comments explaining each field.

workspace.json
Antigravity Config 
{
  // Project name — appears in the Antigravity dashboard
  "name": "my-agentic-project",
  "version": "1.0.0",

  // Gemini agent settings
  "agent": {
    // Template to use — gemini-2.0-flash-exp is faster and cheaper
    "model": "gemini-2.0-flash-exp",

    // temperature 0.2 = more deterministic responses to code
    "temperature": 0.2,

    // Files loaded as context at the start of each session
    "context_files": ["CONTEXT.md", "ARCHITECTURE.md"],

    // false = agent always asks for approval before acting
    "auto_approve": false,

    // Maximum number of steps per task before stopping and asking for guidance
    "max_iterations": 20
  },

  // Tools available to the agent
  "tools": {
    "mcp_servers": [
      {
        "name": "filesystem",
        // npx downloads and runs the MCP server automatically
        "command": "npx @modelcontextprotocol/server-filesystem",
        // --root . Limits access to the project directory only
        "args": ["--root", "."]
      },
      {
        "name": "github",
        "command": "npx @modelcontextprotocol/server-github",
        // ${GITHUB_TOKEN} reads from system environment variable
        "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
      }
    ],
    // Built-in browser (Chromium) to check visual result
    "browser": { "enabled": true, "headless": false },
    // Sandboxed terminal = isolated from the host system
    "terminal": { "enabled": true, "sandbox": true }
  },

  // Pre-defined workflows with their steps
  "workflows": {
    "feature": {
      "steps": ["plan", "implement", "test", "review"]
    },
    "bugfix": {
      "steps": ["analyze", "fix", "verify"]
    }
  },

  // Permissions: "ask" = requests approval, "allow" = automatic, "deny" = blocked
  "permissions": {
    "file_write": "ask",
    "networking": "ask",
    "shell_execute": "ask"
  }
}

CONTEXT.md — Example Structure

Essential sections that every CONTEXT.md must have for an efficient agent.

CONTEXT.md
Markdown 
# CONTEXT.md — Antigravity / Gemini Agent Context

> This file is automatically loaded by Antigravity as persistent context.

---

## Technology Stack

- **Frontend:** React 18 + TypeScript + Tailwind CSS
- **Backend:** Node.js 20 + Fastify + Prisma
- **Cloud:** Google Cloud (Cloud Run + Firebase + Cloud SQL)

## Code Conventions

- Component files: PascalCase (UserProfile.tsx)
- Commits: Conventional Commits in English
- Never use \`any\` in TypeScript — use \`unknown\`

## Mandatory Restrictions

### The agent must NEVER:
- Commit directly to \`main\`
- Store or log credentials
- Deploy without explicit approval

## Domain Glossary

- **Task:** Unit of work within a Sprint
- **Sprint:** 2 week development period
- **Archived:** Task completed and removed from active listing
